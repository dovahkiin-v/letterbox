## ADR-066 — Letterbox is versioned with a once-a-day best-effort "update available" check, reversing the "no version check" anti-scope

**Date:** 2026-06-12
**Context:** Letterbox shipped at v1.0.0 framed as a *frozen reference artifact* that *never touches the network* — `CONTRIBUTING.md` even listed "no phone-home, no auto-update, **no version check** … first run is silent" in its **never-by-design** anti-scope. But the artifact is not actually static: it keeps gaining fixes and small features (ADR-063/064/065 all post-date the "frozen" framing), shipped from the git `main` HEAD with no PyPI release. A user on an older checkout has no way to learn a newer version exists — the only signal is the `__version__` on `main`, which nobody re-reads. Mitos solved the identical problem with a once-a-day, fail-silent update check (`mitos/_update.py`); the ask was to mirror that for letterbox. The tension: doing so reverses an explicit, named anti-scope promise, and letterbox's whole identity leans on "no network."
**Decision:** Add a best-effort update check (`letterbox/_update.py`), mirroring mitos's design but using the **stdlib `urllib`** rather than adding `requests` (P7 dependency skepticism; letterbox keeps its near-zero runtime footprint). On a **human-facing** CLI command the launcher fetches `__version__` from `letterbox/__init__.py` on `main` (raw.githubusercontent.com), at most once per 24 h (cached under `~/.cache/letterbox/`, honouring `XDG_CACHE_HOME`), with a 1.5 s timeout, and prints a one-line `📦 … (current → latest)` notice to **stderr** before dispatch — so the launcher surfaces it ahead of the harness PTY. The check is **fully fail-silent** (no network, a timeout, or a parse error yields no notice), **opt-out** via `LETTERBOX_NO_UPDATE_CHECK=1`, and is **never run for `letterbox mcp`** (the agent's stdio server — a stray line would corrupt the JSON-RPC stream). `letterbox.__version__` becomes the **single source of truth**: `pyproject.toml` declares `dynamic = ["version"]` and reads it via `[tool.setuptools.dynamic]`, so packaging and the dunder can never drift (which would silently skip the nudge). A bump-`__version__`-per-meaningful-change ritual is documented in `CONTRIBUTING.md`; this ADR bumps to **1.1.0**. The user-facing positioning is reframed from "frozen / never touches the network" to "**versioned, unsupported** — versioned and occasionally updated at the author's whim, with no roadmap, SLA, or maintenance commitment" (README + CONTRIBUTING), and the anti-scope bullet is rewritten to distinguish what is still forbidden from the one new exception.
**Rationale:** The nudge only works if it is on by default — an opt-in update check that nobody enables tells nobody anything — and the cost is a single **read-only GET of a public version string**: no telemetry, no identifiers, no data about the user leaves the machine (GitHub sees only the request), so the *substantive* privacy guarantees ("nothing about you is collected or sent") are preserved even as the literal "never contacts any server" claim is retired. Reversing the anti-scope is a deliberate **owner** decision, not a drift — `CONTRIBUTING.md` is updated in the same change so the promise and the behaviour stay honest, rather than the code quietly contradicting a documented "never." Stdlib `urllib` over `requests` keeps the dependency story clean and is the right call for a tool that prides itself on inertness. Surfacing before dispatch (not in a `finally` like mitos's short-lived verbs) fits letterbox's flagship command — the launcher takes over the terminal, so the notice belongs *before* the PTY, which is also the moment the user described ("someone runs letterbox, throw a note"). **This does not touch ADR-002:** the *message protocol* remains file-based with no daemon and no network; the update check is an out-of-band CLI convenience, orthogonal to message transport. **Tradeoff accepted:** the first human launch of a day now makes one ≤1.5 s network attempt unless opted out — bounded, cached, and fail-silent, but no longer a categorically silent first run.
**Supersedes:** Reverses the "no phone-home, no auto-update, **no version check** / first run is silent" anti-scope item in `CONTRIBUTING.md`, and the absolute "never touches the network" framing in `README.md`. Does **not** supersede ADR-002 (files are the source of truth; the protocol has no daemon and no network) — that guarantee is unchanged.

## ADR-065 — `send_message` returns a confirmation envelope that tells the agent to stand down, and reports who will actually be notified

**Date:** 2026-06-12
**Context:** Letterbox is interrupt-driven by design: the peer's watcher injects a 📬 when it replies, and an endpoint is never notified of its own writes (Filter 6 / ADR-022). But `send_message` returned a bare `{"id": …}`, which gave the sending agent no signal that the send was complete or what to do next. Two failure modes followed. (1) **Post-send polling** — an agent with no "you're done" cue would loop on `check_messages`/`check_latest_message` waiting for the reply, burning context against a bridge that was already going to wake it. (2) **The empty-room trap** — a broadcast into a channel with no other live participant lands on disk but pings no one, and because the watcher's start watermark suppresses pre-launch backlog (ADR-024), a peer that joins *later* is not auto-notified of it either. An agent told only "sent" would stand down and wait forever for a reply that cannot come. This empty-room state is not an error — a session can legitimately hold a channel alone and have a peer (e.g. Gemini) added later without a restart.
**Decision:** `send_message` returns a confirmation envelope: `id` (unchanged, the message-id stem), `delivered` (`True` once the message is on disk), `notified` (the labels the watcher will raise a 📬 for — the directed target, or every *other* live participant for a broadcast, sender always excluded), and `notice` (plain-English guidance). The `notice` tells the agent to **stop and end its turn rather than poll** — but only promises a wake-up when `notified` is non-empty. For a broadcast into an empty room (`notified == []`) the notice instead warns that no 📬 was raised and a later joiner won't be auto-notified, steering the agent to relay to the human rather than wait. `notified` is computed by `_notified_labels` from the same live pid-locks `_assert_recipient_live`/`channel_info` use (ADR-061); for a directed send it is exactly the validated recipient (Filter 7 suppresses the 📬 for everyone else, ADR-062). The change is **purely additive** — `id` stays a top-level key, so every existing caller and test that reads `result["id"]` is untouched. The agent-facing docs (the `send_message` docstring and the `letterbox` skill) gain a "stand down after sending" instruction.
**Rationale:** The synchronous tool return is the natural acknowledgment surface — the agent already gets a value back, so the confirmation belongs there, not in an injected notification (a 📬 for your own send would be noise and would fight the own-write filter the whole architecture rests on). Reporting `notified` rather than a bare "sent" keeps the "you'll be pinged, don't poll" promise *truthful*: it is made only when someone is actually live to reply, and withheld (with a warning) in the one case — the empty-room broadcast — where standing down would strand the agent. Warning rather than *failing* the empty-room broadcast is deliberate and mirrors ADR-064's broadcast carve-out: a broadcast is for everyone present and future, and the load-first-add-peers-later workflow is a real, supported pattern; failing it would break that. Reusing the existing pid-lock liveness substrate adds no new state.
**Supersedes:** None. Extends the `send_message` contract (ADR-062 directed addressing, ADR-064 send-time validation) with a richer return shape and codifies the interrupt-driven "don't poll after sending" etiquette that ADR-024's no-backlog watcher already implied.

## ADR-064 — `send_message(to=…)` validates the recipient against live participants at send time, failing loud instead of sending an unnotified message

**Date:** 2026-06-11
**Context:** ADR-062 activated directed addressing with *observable-but-not-notifiable* semantics: the watcher's Filter 7 suppresses the 📬 for any message whose `recipient` is non-empty and is not this endpoint. That filter compares labels with a case-sensitive `!=` (`watcher.py`), and labels flow verbatim from `--as` (`resolve_sender_label`, no normalization). The consequence surfaced live on channel `cartolina-language-pairings-vision`: Antigravity-hosted Gemini sent `send_message(to="Claude")` while the Claude endpoint was launched `--as claude`. `"Claude" != "claude"` → Filter 7 dropped every 📬, so Claude was never woken. The message was not lost (the read path applies no recipient filter — `check_messages` still returned it), but a directed message that never notifies its target has silently failed its one job: attention. Nothing at send time caught the mismatch; the sender had no signal anything was wrong.
**Decision:** `send_message` validates a directed `to` against the channel's **live participants** (the pid-locks `list_live_participants` reads, same source as `channel_info`'s `participants`) **before any disk write**. A `to` that names no live participant raises a new `UnknownRecipientError` (exported from `mcp_server`) whose message names who *is* live and, when the only difference is capitalization, suggests the correct casing (`Labels are case-sensitive — did you mean 'claude'?`). Directing at the sender's own label is also refused (own writes are never echoed back per ADR-022, so it would notify no one) with a nudge to broadcast. Broadcast (`to` unset / empty) is unvalidated: it is for everyone, present and future, and carries no liveness requirement. The watcher's Filter 7 is **unchanged** — it remains the case-sensitive notify gate; this ADR adds a loud send-time pre-check in front of it rather than loosening it. The accompanying agent-facing docs (the `send_message` docstring, the `letterbox`/`letterbox-setup` skills, `docs/AGENT_POINTER.md`) are reframed from "silently looks ignored" to "raises an error."
**Rationale:** This is the framework's loud-at-the-boundary principle: a request the system cannot honor as intended (notify this peer) fails visibly at the call site rather than degrading into a no-op. Validating against *live* participants rather than any-label-ever is deliberate — directed addressing exists to control *attention* (ADR-062), and a peer that is not running cannot receive attention (ADR-024: no backlog surfacing on launch, so a directed message to an offline label would never be seen either). The check reuses the existing pid-lock liveness substrate (ADR-061), so it introduces no new state. Case-sensitive matching is **kept**, not replaced with case-folding: making labels case-insensitive would also have to change the identity-collision and pid-lock comparisons (`watcher.py` Filter 5 / `locks.py`) to stay consistent, a larger "labels are case-folded everywhere" redesign; the narrow, reversible fix is to make the existing case-sensitivity *visible* at send time. **Tradeoff accepted:** a directed message to a real-but-stale pre-upgrade peer (one not in `participants` because its old launcher wrote no pid-lock — see ADR-062's version-skew note) is now refused; the documented cure (relaunch the stale session, or broadcast) is unchanged, and broadcast remains available as the escape hatch.
**Supersedes:** None. Extends ADR-062 (directed addressing) with a send-time guard; the watcher's notify-path filter and ADR-022's own-write read filter are untouched.

## ADR-063 — Claude Code now needs the delayed-terminator write too: a 2026-06-10 update added a fast-return submission gate

**Date:** 2026-06-10
**Context:** ADR-057 established the delayed-terminator write for TUIs that gate submission on keystroke timing (Gemini, Antigravity at `terminator_delay = 0.1`), and explicitly carved Claude Code out: it was "verified live" to submit on a combined `b"text\r"` write, so `ClaudeAdapter.terminator_delay` stayed at the base default `0.0`. A Claude Code update shipped 2026-06-10 changed this. With the old combined write, an injected 📬 notification now **lands in Claude Code's input box but does not submit** — the `\r` arriving in the same burst as the text is swallowed as a newline-in-box rather than a discrete Enter, exactly the failure mode ADR-057 documented for Gemini's `KeypressContext`. Confirmed live on channel `demo-2`: directed messages into a Claude Code participant appeared and hung; the same messages into Gemini (already delayed) submitted fine.
**Decision:** Set `ClaudeAdapter.terminator_delay = 0.1`, joining Gemini and Antigravity on the delayed-terminator path: write the text, wait past the harness's fast-return window, then write the `\r` as a separate, discrete Enter. No new mechanism — this is a config flip on the existing ADR-057 substrate. The contract test (`test_adapters_claude.py`) is updated to assert `0.1`.
**Rationale:** The combined-write path was only ever a per-harness empirical fact, never an invariant — ADR-057 already treated the terminator strategy as a tunable property of each harness's current TUI, not a fixed truth. When the harness changed its submission gate, the right response was to flip the same knob that already fixed the other two, not to invent a new path. `0.1` is the proven value across the other two adapters; if a future Claude Code widens its window past 100 ms it can be raised, but it held in live testing. Diagnosis note for future readers: the symptom is harness-version-coupled, so a recurrence after a Claude Code update should be checked against this ADR before suspecting the injection or watcher layers (which are unchanged).
**Supersedes:** The Claude-specific carve-out in ADR-057 (Claude keeps the combined-write path / `terminator_delay = 0.0`). The rest of ADR-057 — the delayed-terminator mechanism and the Gemini/Antigravity values — stands unchanged.

## ADR-062 — A channel is N-party: directed messages are *observable but not notified*, and `channel_info` reports live `participants`

**Date:** 2026-06-10
**Context:** The read filter has always been "every message that is not my own write is for me" (`_is_own_write`, ADR-022), and the message format carried a `recipient` field that was inert ("informational in v1"). That already made a channel an N-party broadcast bus mechanically — three endpoints (`claude-review`, `claude-commit`, `gemini`) on one channel all see each other's traffic — but two things were missing for it to be *usable* as a group: (1) no way to address one participant specifically, and (2) no way to discover who is in the room (`channel_info` reported a singular `peer` = whoever spoke last, which can't answer "who can I talk to?").
**Decision:** Activate the dormant `recipient` field as **directed addressing**, with *observable-but-not-notifiable* semantics. `send_message(body, to=None, in_reply_to=None)`: `to` unset → broadcast (recipient `None`, everyone notified); `to=<label>` → directed (recipient set; an empty `to` normalizes to broadcast). The **read path is unchanged** — `check_messages`/`check_latest_message` still return every non-own message regardless of recipient, so a directed message stays readable by everyone. Only the **watcher** changes: a new Filter 7 suppresses the 📬 notification when `recipient` is non-empty and is not this endpoint. So a directed message wakes only its target; others can still read it on demand. `channel_info` gains a `participants` list — the labels currently **running** on the channel, derived from the live pid-locks (ADR-061), so it surfaces even a participant who has launched but not yet spoken.
**Rationale:** Observable-not-notifiable is the honest semantic for a shared-directory protocol: the file physically lives in the channel, so claiming privacy would be a lie (§13.3 trust model). What directed addressing genuinely controls is *attention* — who gets pinged — not *visibility*. This keeps the human/observer transparency of a group channel while letting agents run focused sub-dialogues (`claude-review` → `claude-commit`) without spamming everyone. Sourcing `participants` from the pid-locks (who is running) rather than message history (who has spoken) answers the question an agent actually has before addressing a directed message — there is no reply to observe yet. The change set is small precisely because the substrate was already N-party; this is activation, not redesign.
**Supersedes:** None. Activates the v1-dormant `recipient` field (ADR-022's own-write filter is unchanged — recipient filtering lives only in the watcher's notify path) and extends ADR-056's `channel_info` oracle with `participants`.

## ADR-061 — A duplicate sender label on the same channel is refused via a per-channel pid-lock

**Date:** 2026-06-10
**Context:** With N-party channels (ADR-062), two endpoints launched under the same `--as` label on one channel are indistinguishable — their messages collide on identity, and the watcher already had to emit an identity-collision warning to flag the misconfiguration after the fact. Nothing prevented the bad launch in the first place. The natural scenario that motivated this — running `claude-review` and `claude-commit` as distinct participants — only works if the system pushes you toward distinct labels.
**Decision:** Before spawning, the launcher claims a **pid-lock** at `state_dir/locks/<channel>/<sender_label>.pid` containing its pid. If a lock already exists and its pid is alive (`os.kill(pid, 0)`), the launch is refused with a fail-loud vector that suggests a distinct name (`--as <label>-2`); a stale lock (dead pid or unreadable contents) is silently overwritten. The lock is released on every teardown path (clean exit, signal, crash-rollback) via `run_launcher`'s `finally`. The lock is **per-channel**, not global: `claude` on `demo` and `claude` on `review` are different conversations and must not false-collide. The lock primitives live in a new Tier-1 leaf `locks.py` (stdlib-only) so both the launcher (Tier 4) and the MCP server (Tier 4, for `participants`) can share the convention without violating the Tier-4 sibling bulkhead (§13.5).
**Rationale:** Refusing at launch is strictly better than warning after the fact — the collision never enters the channel. Per-channel scoping is the correct granularity because identity attribution is a per-channel property; a global lock would forbid legitimate cross-channel reuse of a label. Liveness via `os.kill(pid, 0)` is best-effort (a recycled pid could read as alive) — acceptable for a human-interactive guard under the same trust model as the rest of letterbox, not an enforcement boundary. The TOCTOU window between the existence check and the write is microseconds on an interactive CLI and is left unhardened deliberately.
**Supersedes:** None. Complements the watcher's identity-collision warning (ADR-022 / Vision §3.4.1) by preventing the collision upstream rather than only diagnosing it.

## ADR-060 — Antigravity's letterbox MCP server is wired via a local `agy` plugin whose server file MUST be named `mcp_config.json`

**Date:** 2026-06-05
**Context:** ADR-054 established that Gemini and Antigravity are settings-wired (they don't take Claude's `--mcp-config` flag), and the first live Antigravity bridge confirmed the PTY layer works — but the agent had **no letterbox tools**, because nothing registered the MCP server with `agy`. Unlike Gemini (a flat `mcpServers` key in `~/.gemini/settings.json`), `agy`'s own `settings.json` (`~/.gemini/antigravity-cli/settings.json`) has no `mcpServers` key. `agy` loads MCP servers from **plugins** (`agy plugin install <dir>`, where `<dir>` holds a `plugin.json` manifest). Empirically probing `agy plugin validate`: an inline `mcpServers` key in `plugin.json` is ignored, and a `.mcp.json` file is silently skipped (the Go plugin walker ignores the dotfile). The validator recognises the server only when it is in a file named exactly **`mcp_config.json`** (`{"mcpServers": {…}}`) — then it reports `mcpServers : 1 processed`.
**Decision:** Wire letterbox into Antigravity as a minimal local plugin directory containing two files: `plugin.json` (manifest — `name`/`version`/`description`) and `mcp_config.json` (the letterbox server, `command` = absolute `letterbox` path, `args` = `["mcp"]`, channel-agnostic — the launcher passes channel/identity by env per ADR-055). Install with `agy plugin install <dir>`; remove with `agy plugin uninstall letterbox`. The canonical scaffold lives at `~/.letterbox/agy-plugin/letterbox/`. This is documented in the README's per-harness Setup and in `skills/letterbox/SKILL.md` (the agent-facing self-wiring guide).
**Rationale:** The `mcp_config.json` filename is non-obvious and was the entire blocker — a `.mcp.json` (the Claude-plugin convention) validates "ok" while silently contributing no server, which would read as success and fail in use. Recording the exact filename, the install/uninstall commands, and the `agy`-uses-plugins-not-settings distinction turns a reverse-engineering session into a one-paste setup. Keeping the server file channel-agnostic preserves the same "launch is the only per-channel action" UX the other two harnesses already have.
**Supersedes:** None. Fills in the Antigravity half of ADR-054's per-adapter MCP-wiring contract (Claude = `--mcp-config`; Gemini = `settings.json` `mcpServers`; Antigravity = `agy` plugin with `mcp_config.json`).

## ADR-059 — `letterbox agy` is accepted as a CLI alias for the `antigravity` harness subcommand

**Date:** 2026-06-05
**Context:** Antigravity ships as the `agy` binary, and `agy` is how people actually launch it (the adapter already encodes this — `command = "agy"`, distinct from the `"antigravity"` registry key). But the launcher subcommand was only spelled `letterbox antigravity`, forcing the user to type the long form of a tool they invoke as `agy` everywhere else. The first live Antigravity bridge confirmed the PTY layer works end-to-end (notifications and message delivery both directions) and surfaced this ergonomic friction.
**Decision:** Add `agy` as an argparse alias on the `antigravity` subparser (`_HARNESS_ALIASES = {"antigravity": ("agy",)}`), so `letterbox agy …` and `letterbox antigravity …` are interchangeable. The alias is purely a surface spelling: `set_defaults(harness_name="antigravity")` pins the canonical registry key regardless of which form the user typed, so adapter lookup, the `[harness.antigravity]` config block, and `run_launcher` dispatch are all unchanged. The help text renders it as `antigravity (agy)`.
**Rationale:** An alias, not a rename, is the right tool. The registry key, config-block name, adapter `name`, and the tier-header/K4 tests all standardise on `"antigravity"`; renaming the subcommand to `agy` would split the subcommand from the config-block name and churn those invariants for no gain. The alias meets the user where their muscle memory already is (`agy`) while keeping one canonical internal identity. The dispatch is robust because `harness_name` comes from `set_defaults`, which is independent of the alias argparse records in `dest="command"`.
**Supersedes:** None. A pure CLI-ergonomics addition on top of the harness-subcommand surface (8a/9a).

## ADR-058 — The read marker self-maintains: it aligns to the launch watermark and `check_messages` advances it on a default catch-up read

**Date:** 2026-06-05
**Context:** A live two-agent run surfaced inbox clutter from two directions. (1) **Cross-session backlog** — a freshly relaunched endpoint's `check_messages` returned messages from a *previous* session (in the run, 3 unread where only 1 was new). The watcher already records a "start watermark" at init and only notifies about messages arriving strictly after it (ADR-024), so a restart is a calm "fresh start" on the *notification* path — but the read marker (`high_water_mark`, what `check_messages`/`channel_info` filter on) only ever moved on an explicit `acknowledge`. The two positions were out of sync: the watcher said "nothing new since launch", but a catch-up read replayed the whole backlog. (2) **Within-session pile-up** — `check_messages` was purely non-advancing, and agents in practice never call `acknowledge`, so every catch-up read re-returned everything it had already shown. The inbox never drained without a manual marker advance the agent never performed.
**Decision:** The read marker becomes self-maintaining via two coupled changes, both composed at the MCP layer from existing channel primitives (no channel-layer change). (1) **Launch alignment.** When `letterbox mcp` opens its channel, it advances this endpoint's `high_water_mark` to the newest message already on disk (`_align_read_marker` → `Channel.acknowledge(latest_stem)`), re-syncing the marker with the watcher's start watermark — so the inbox starts as "messages since I launched", not the whole channel history. Monotonic (clamps to `max(current, latest)`, never rewinds a marker a prior session pushed further via `since_id`) and a no-op on an empty channel. (2) **Read-is-acknowledge.** A *default* `check_messages` call (no `since_id`) advances the marker to the newest item it returns, so the next call resumes past this page and pagination is driven off the marker with no `since_id` threading — "keep calling until `has_more` is false to drain the inbox". A `since_id` call stays an explicit, non-advancing **history** query (the same falsy guard `list_unread` uses for its bound — `None` and `""` both fall through to the marker, a real id overrides without advancing), `check_latest_message` stays **peek-only**, and `acknowledge` remains for explicit single-message control. The peer's history is always reachable on demand via `check_messages(since_id=…)`. Whichever peer message reconciliation `_is_own_write` resolves to is unaffected — own writes are filtered downstream, so the launch watermark uses the newest filename regardless of sender (it means "everything up to launch is not new").
**Rationale:** The read marker and the watcher's notification watermark are two views of the same question — "what has this endpoint already seen?" — and they should not drift. ADR-024 made the *notification* path treat a restart as a fresh start; this extends the same intent to the *read* path, which had silently kept replaying. Reading-is-acknowledging matches a human mail client (opening a thread marks it read) and removes a footgun where an inbox could only be cleared by a marker advance the agent never issued. Keeping `since_id`/`check_latest_message`/`acknowledge` non-advancing preserves the full power surface: explicit history queries, low-cost peeks, and single-message control all stay available for the agent that wants them. Composing both fixes at the MCP layer from `list_unread` + `acknowledge` honours the channel layer's pure-query / mutating-command separation — the query primitives stay read-only; only the agent-facing tool composes a read with a marker advance.
**Supersedes:** The "`check_messages` is purely non-advancing; only `acknowledge` moves the marker" semantics, and refines ADR-024 (whose calm-restart guarantee now covers the read marker, not just the watcher) and ADR-012 (the default `check_messages` page now advances the marker; the `since_id` cursor's non-advancing contract is retained). `acknowledge` (ADR-021) is unchanged.

## ADR-057 — The line terminator is delayed past the harness's fast-return window for TUIs that gate submission on timing (Gemini, Antigravity)

**Date:** 2026-06-05
**Context:** A live two-agent run surfaced a one-sided failure: when Claude injected a notification to Gemini, the text landed in Gemini's input box but **never submitted** — the human had to press Enter by hand. The reverse direction (Gemini → Claude) worked. A first attempt — writing the text and the `\r` as two separate `os.write`s (mirroring the tower orchestrator's `send(text); send('\r')`) — did **not** fix it. Reading the Gemini CLI bundle (`packages/cli/src/ui/contexts/KeypressContext.tsx`) revealed why: its `bufferFastReturn` rewrites any Enter that arrives within `FAST_RETURN_TIMEOUT` (**30 ms**) of the previous key into shift+Enter — with the literal comment *"to make it a newline, not a submission."* Our two writes fire back-to-back (~0 ms apart), so the `\r` lands inside the 30 ms window and is demoted to a newline. The reason tower works is incidental: pexpect's default `delaybeforesend = 0.05` (50 ms) puts tower's `\r` *outside* the window. The root cause is **timing**, not write-separation.
**Decision:** The adapter property is a **delay**, not a boolean: a new `terminator_delay` class attribute (seconds). `0.0` (the base default, Claude) appends the terminator to the text in one write — Claude's proven path. A positive value (Gemini and Antigravity: `0.1`) writes the text, awaits the delay, then writes the terminator alone — so the terminator clears the harness's fast-return window and registers as a real Enter (a submit). 100 ms sits comfortably above Gemini's 30 ms threshold with margin for scheduling jitter, and is imperceptible to a human. Antigravity inherits `0.1` by inference (Gemini-family TUI) pending a live `agy` run; the knob makes tuning it a one-line change. Claude keeps `0.0`, so its verified behavior is untouched.
**Rationale:** Encoding the fix as a delay matches the actual mechanism (a timing gate read straight from Gemini's source), where the earlier "separate write" framing was a symptom-level guess that the live test falsified. A declarative numeric class attribute keeps the adapter substrate honest — a future harness states *how long* it needs between text and submit, alongside its command, args, template, terminator, and MCP-wiring style. The byte-content the child receives is unchanged; the new behavior is verified by asserting the writes are split (`[b"hi", b"\r"]` vs `[b"hi\r"]`) and that the delay is awaited between them.
**Supersedes:** None. Refines ADR-018 (the `b"\r"` terminator that wakes the agent) for harnesses that gate submission on inter-key timing.

## ADR-056 — The `letterbox mcp` server runs *dormant* when spawned with no channel, and `channel_info` is the agent's bridge-state oracle

**Date:** 2026-06-05
**Context:** Now that a settings-wired harness loads its letterbox MCP server from its own user-level settings (ADR-055), that server starts on *every* session of that harness — including plain sessions the user never intended as a bridge (no `letterbox <harness>` launch, so no `LETTERBOX_CHANNEL`). The server previously required all three join keys and exited non-zero when they were absent, which a host renders as a failed/`Disconnected` MCP server in every ordinary session — noise that violates letterbox's "calm surface, quiet success" accessibility contract. Separately, an agent that *has* the letterbox tools available had no way to know its own situation: whether a bridge is even active, on what channel, as whom, who the peer is, or whether the peer has shown any life. It would see `send_message` and assume it works.
**Decision:** Two coupled changes. (1) **Dormant mode.** `letterbox mcp` resolves its join keys leniently (flags OR environment — ADR-055); when a value is absent from both, `run()` branches on `stdin.isatty()`. A non-TTY spawn (an MCP host — the common case) starts the server **dormant**: it connects identically (same six tools, same schemas, so the host shows a calm "connected"), but the four messaging tools (`send_message`, `check_latest_message`, `check_messages`, `acknowledge`) fail loud the instant they are called, and `channel_info` returns `{"bridged": false, "detail": …}`. A TTY spawn (a human ran `letterbox mcp` by hand — which the README says never to do) still fails loud to stderr with exit 2. `list_channels` works in either mode (filesystem enumeration, not a per-channel operation). (2) **State oracle.** `channel_info` is reshaped into the agent's situational-awareness tool: when bridged it returns `{bridged, channel, sender, unread, peer, peer_has_spoken, last_peer_activity}`, where `peer` and `last_peer_activity` are read from the *most recent peer message* (a reverse scan that stops at the first non-own-write) — the honest, observed-from-traffic answers to "who am I talking to?" and "is the peer alive?", since the launch-time `recipient_label` is `""` in v1.
**Rationale:** "Never silently dark" (design spec §7.1) and "calm surface" pull in opposite directions for a server that lives in user settings: failing loud on startup spams every plain session, but starting silently could hide a *genuinely* misconfigured bridge. Splitting on `isatty()` and deferring the loudness to *tool-use time* resolves both — a deliberate plain session stays quiet (it simply never calls a messaging tool), while a misconfigured bridge surfaces the moment the agent actually tries to talk. The state oracle extends letterbox's "files as the source of truth" ethos to the *agent*, not only the human inspecting the directory: the agent can now reason — "peer last spoke 90 s ago → likely live" vs "never → tell the human to launch the peer" — instead of guessing. The peer fields are peer-sourced and so share the trust class of `check_messages` bodies (informational, never fed into a trusted notification). Mirrors how the Forge tower MCP server starts gracefully without its channel.
**Supersedes:** The prior "`letterbox mcp` requires all three join-key flags and exits 2 when any is missing" behavior, and the four-field `channel_info` wire shape (`channel`/`sender_label`/`recipient_label`/`unread_count`).

## ADR-055 — A settings-wired harness receives its per-launch channel and identity via environment variables, keeping its MCP-server config channel-agnostic

**Date:** 2026-06-05
**Context:** ADR-054 stops letterbox from injecting `--mcp-config` into Gemini/Antigravity; those harnesses instead load the letterbox MCP server from their own settings (e.g. `~/.gemini/settings.json`). But that server still needs to know *which channel* and *which identity* each launch is for — and baking `--channel X --as Y` into the static settings file would mean editing config for every new channel, destroying the "just launch both sides on the same `--channel` and they talk" simplicity the Forge orchestrators already provide. The Forge tower orchestrator solves exactly this: its settings entry carries no channel argument; the orchestrator sets `FORGE_DIALOGUE_CHANNEL` in the spawned agent's environment and the MCP server reads it at runtime.
**Decision:** The launcher exports the per-launch join keys into the harness's spawn environment — `LETTERBOX_CHANNEL`, `LETTERBOX_SENDER`, and `LETTERBOX_INSTANCE_ID` (alongside the existing `LETTERBOX_HOME`) — and the `letterbox mcp` argument parser falls back to each of these when the corresponding flag (`--channel` / `--as` / `--instance-id`) is absent. Flags win over the environment when both are present. The harness forwards its environment to the MCP child it spawns (verified live for Gemini), so a settings entry as minimal as `{"command": ".../letterbox", "args": ["mcp"]}` picks up the correct channel, label, and instance-id for *every* launch with no per-channel editing. Because the launcher exports its *own* freshly-generated instance-id, the MCP child and the watcher now share it — own-write recognition works on both the sender-label and instance-id halves (ADR-022), so the spurious identity-collision warning a static instance-id would have produced never fires.
**Rationale:** This is a faithful port of the working Forge pattern (env-passed channel), not a new invention. It restores the exact UX the user relies on — change the channel on the launch command, nothing else — for harnesses that cannot take a per-launch config flag. The environment is the right transport because it is the one channel a harness reliably forwards to a stdio MCP child (letterbox already depends on this for `LETTERBOX_HOME`). Flags-win-over-env keeps the self-contained Claude path (ADR-054) authoritative: its `--mcp-config` bakes in explicit flags that override the env, so the two wiring styles coexist without conflict.
**Supersedes:** None. Builds on ADR-054 (per-adapter MCP-config wiring) and the `LETTERBOX_HOME` join-key precedent.

## ADR-054 — MCP-config wiring is per-adapter: `--mcp-config` is Claude-only; Gemini and Antigravity load letterbox from their own settings

**Date:** 2026-06-05
**Context:** The launcher unconditionally prepended `--mcp-config <temp-path>` to *every* harness's argv as its mechanism for wiring the letterbox MCP server. This was only ever exercised against Claude Code (whose CLI has the flag) and the fake test harness. Against the real Gemini CLI (0.45.0) and Antigravity CLI (`agy` 1.0.5), `--mcp-config` is an **unknown flag**: Gemini aborts with `Unknown arguments: mcp-config` and exit 1 — the harness never starts. Making the wiring style an explicit, declarative adapter property keeps the elegant Claude path (self-contained, no manual config) while letting Gemini/Antigravity use the only mechanism their CLIs actually offer. It also matches the user's real, working setup, where the Gemini-side tools have always come from `~/.gemini/settings.json`. The fake-harness suite missed this because it never parses real CLI flags; the fix is grounded in live probes of both real CLIs.
**Supersedes:** The launcher's prior "prepend `--mcp-config <temp-path>` for every harness" behavior.

## ADR-053 — The harness launch path renders its expected startup errors as one-line vectors, scoped to the harness dispatch (not the whole `main()` boundary)

**Date:** 2026-06-01
**Context:** Launching a harness (`letterbox claude --channel …`) runs a startup-validation chain before spawning the agent: it checks the state directory's permissions, resolves the adapter and its configuration, validates the notification template, and confirms the harness command is on PATH. Each check raises a typed error carrying a clean, human-facing message. But the command line only caught one kind of error at its top-level boundary — a malformed config file (see ADR-050). Every other expected startup failure escaped as a raw Python traceback: a command that is not on PATH, an unknown or unconfigured harness, an existing world-accessible state directory, or an invalid configured notification template all printed a stack trace before exiting. A traceback on a normal setup mistake is a wall, not a path forward — the same blemish ADR-050 fixed for config files and ADR-051 fixed for a missing state directory, left open for the rest of the launch path.
**Decision:** The four expected pre-spawn startup errors — command-not-on-PATH, unknown/unconfigured harness, world-accessible state directory, and invalid notification template — are caught where the launch is driven and each is printed as a single `letterbox: <message>` line on stderr with exit code 1, never a traceback. The catch is scoped to **exactly four enumerated error types** — there is deliberately **no catch-all**: a genuine, unexpected bug still surfaces as a full traceback so it can be diagnosed, never silently flattened into a one-liner. The catch lives at the **harness dispatch**, not the top-level boundary where the config-file error is caught. This is a deliberate narrowing: a config-file error can arise from every command (read commands, harness launch, the agent's server child all read config), so it belongs at the one shared boundary; but these four errors are raised *only* by the harness launch path. Two of them (command-not-found and the unconfigured-harness lookup) are expressed with the same built-in error types that the read and maintenance commands raise during normal operation, so catching them at the shared top-level boundary would risk swallowing a genuine bug in one of those other commands. Scoping the catch to the launch path renders the launch errors cleanly while leaving every other command's error handling — and every genuine bug — untouched. One regression test per error type drives the real launch path through the command line and asserts a clean vector with no traceback on either stream.
**Rationale:** "Errors are vectors, not walls" (Framework P3) applies to the whole public entry point, not just its config-file door — a first-time user who mistypes a harness command or forgets to set a distinct label should see one clear sentence, not a stack trace. Enumerating the catchable types rather than using a catch-all is the load-bearing restraint: it keeps the diagnostic value of a real traceback for real bugs while taming the expected, user-facing failures. Scoping the catch to the path that actually raises these errors (rather than the broadest possible boundary) is the bulkhead discipline (Framework P7) — the narrowest catch that covers the real cases, with no collateral reach into unrelated commands. Surfaced by the same live smoke test that drove ADR-051; this closes the launch path's error-vector contract holistically.
**Supersedes:** None. Extends the launcher error-vector lineage of ADR-050 (config-file errors at the top-level boundary) and ADR-051 (missing state directory auto-created) to the remaining expected startup failures.

## ADR-052 — The PTY-Parent bridges the controlling terminal to the spawned agent

**Date:** 2026-05-31
**Context:** The launcher spawned the agent on its own PTY and held the master file descriptor, but the only code touching that descriptor wrote injected notifications and closed it at teardown. There was no reader and no input path: a human who ran `letterbox <harness>` saw a blank screen and could not type to the agent. An earlier decision had explicitly scoped the PTY-Parent to "spawn / watch / inject / teardown only," with no interactive stdin/stdout bridge. A live smoke test of the end-to-end experience exposed that this left the tool unusable for its primary interactive flow.
**Decision:** The PTY-Parent now bridges the controlling terminal to the spawned agent for the duration of an interactive session, reversing the earlier "deliberately no interactive bridge" decision. The bridge does three things: (1) **relays bytes both directions** — user keystrokes to the agent's PTY and agent output back to the user's screen — on a single dedicated thread running a `select` loop, because reads and writes on a PTY master descriptor are blocking and belong off the event loop; (2) puts the controlling terminal in **raw mode** so control characters and escape sequences pass through byte-faithfully, with the original terminal attributes **restored on every exit path** (clean exit, agent exit, signal, or exception); and (3) propagates **window size** — the agent's PTY is sized to the terminal at startup and re-sized when the terminal is resized (`SIGWINCH`). The bridge engages **only when both the input and output file descriptors are real terminals**; under a pipe, a redirect, or a test harness it is skipped entirely and the launcher behaves exactly as the non-interactive lifecycle-only path. Two writers touch the agent's PTY without a lock — the relay (keystrokes) and the notification injector — which is safe because injected notifications are far smaller than the atomic-write boundary, so a notification cannot tear a keystroke burst; and the agent echoing injected bytes back through its PTY is precisely what makes a notification visible on the user's screen. In raw mode the interrupt signal is disabled, so the user's Ctrl-C is relayed to the agent (the conventional `script`/`pexpect` behavior); teardown is instead driven by the agent exiting or by an external signal, both already handled by the launcher's existing shutdown path.
**Rationale:** Without a bridge the tool's headline use — sitting in a terminal, watching an agent, and typing to it — does not work at all, so this is load-bearing rather than an enhancement. A dedicated relay thread keeps blocking terminal I/O off the asyncio event loop, which stays free for the watcher and the notification loop; the relay is pure byte transport and never participates in lifecycle decisions. Guaranteed raw-mode restoration on every exit path means a crash or a signal never leaves the user staring at a wedged terminal (Self-Healing). Restricting the bridge to real terminals keeps `letterbox <harness> > log` and the entire automated test suite working unchanged. The change adds no third-party dependency — only the standard library's terminal primitives. A real pseudo-terminal-pair test harness accompanies the bridge and automates the previously manual interactive smoke test, so this capability can never silently regress.
**Supersedes:** The earlier "deliberately no interactive stdin/stdout bridge" scoping of the PTY-Parent.

## ADR-051 — A missing state directory is auto-created at launch, not refused

**Date:** 2026-05-31
**Context:** On a first run with no `~/.letterbox` state directory yet, launching a harness (`letterbox claude --channel … --as …`) printed a raw Python traceback and exited. The launcher's startup chain validated the state directory's permissions, and a missing directory raised `FileNotFoundError`, which propagated uncaught to the console. The launcher had hand-crafted a "run init to create it" message, but nothing rendered it as a clean one-liner, so it surfaced as a wall. This was inconsistent on two counts: (1) the channel layer (`Channel.get_or_create`) already auto-creates the channel subtree with `mkdir(parents=True)`, so the launcher refusing to create the parent state directory was an internal contradiction; and (2) the "run init" hint named a command that would not have fixed it — a plain `init` writes a project-local config and does not create the state directory.
**Decision:** When the state directory is absent at launch, create it at mode `0700` and continue, rather than refusing. The directory's mode is set explicitly after creation (the OS umask masks `mkdir`'s mode argument), and this only ever applies to a directory letterbox itself just created. The security gate is unchanged: an *existing* world-accessible state directory is still refused with a clear `chmod 0700` vector.
**Rationale:** Self-healing a missing directory (Framework P5) removes an entire first-run failure mode instead of merely prettifying its error message, and aligns the launcher with the channel layer's existing auto-create behavior. A user's first command now simply works. The change is narrow — a missing directory is created; every other startup check (permissions on an existing directory, adapter availability, command-on-PATH, template validity) is untouched. Surfaced by the live manual smoke test, which a fresh-state run exposed.
**Supersedes:** The launcher's prior "refuse a missing state directory and instruct the user to run init" behavior.

## ADR-050 — A malformed `--config` file produces a one-line error vector, not a traceback (`ConfigError` caught at the `main()` boundary)

## ADR-047 — `letterbox prune` is dry-run by default with a four-row action/safety matrix (`--yes-i-am-sure` executes, `--delete` upgrades move→delete), ages by the filename timestamp (NOT mtime), takes the *minimum* high-water-mark for `--acknowledged-by-all`, and is provably the sole user-message deletion path (L8)

**Date:** 2026-05-29
**Context:** Phase 9d replaces `cli.py`'s last transitional `prune` stub with the real retention command — the **only** way a user reclaims space in an unbounded channel (design spec §3.5, which ships no automatic retention) and the **single destructive command** in the entire codebase. Four design questions had to be resolved, two of which are genuine contradictions/imprecisions in the source docs. (1) **Dry-run vs. move.** design spec §3.5 says prune *both* "defaults to dry-run (touches nothing)" *and* "moves matching messages to a `cold/` subdirectory" — a bare `prune` cannot do both. The documented flag surface shows only `[--delete --yes-i-am-sure]` (no `--apply`/`--execute`). (2) **Age by mtime vs. filename.** The planned test for this phase said "`--older-than 7d` matches by mtime," but the entire codebase time-doctrine is filename-authoritative (§3.2 / ADR-017; `list_messages`'s own docstring states mtime is fragile under `cp -r`/syncthing/NFS). (3) **`--acknowledged-by-all` ceiling** — min or max high-water-mark across endpoints? (4) **Where the cold-store lives** and how the move/delete primitives stay L8-auditable. The phase also empties and removes the transitional utility-stub scaffold (prune was the last stub).
**Decision:** (a) **Four-row action/safety matrix, no new flag.** `--yes-i-am-sure` is the universal "stop previewing, actually do it" execute-gate; `--delete` upgrades the executed action from a `cold/` move to a real deletion. Bare selection = dry-run preview (matched ids → stdout, "would move N to cold/" → stderr, touch nothing, exit 0). `--yes-i-am-sure` alone = move matched files to `<channel>/cold/` via per-file `os.replace` (atomic, reversible). `--delete --yes-i-am-sure` = `os.unlink` each matched file (the one L8 deletion path). `--delete` *without* consent = explicit refusal (Framework-P3 vector naming `--yes-i-am-sure`, exit 1, nothing happens). The handler checks the refusal gate *before* resolving anything. Exactly one selection rule is required via a `required=True` mutually-exclusive argparse group, so a bare `prune --channel x` is an exit-2 usage error — there is no "prune everything by accident." (b) **`--older-than` ages by the message's filename timestamp**, parsed with `datetime.strptime(seg, "%Y%m%dT%H%M%S%f").replace(tzinfo=utc)` (4-digit-year-safe — avoids the glibc `strftime("%Y")` zero-pad bug 2b/3d hit) and compared `msg_time < now - duration`; only called on `list_messages`-returned paths, which are `is_valid_message_filename`-gated so the segment is well-formed. Duration grammar is `\d+[dhms]`, wired as the argparse `type=` callable so junk is an exit-2 vector. (c) **`--acknowledged-by-all` takes the MINIMUM high-water-mark** across all `.read/*.json` endpoints (read directly via `json.loads(path.read_bytes())`, **never** `channel.read_state` — whose corruption-recovery rename side-effect would mutate read-state), matching ids whose stem is `<=` that ceiling (inclusive). Two fail-safes: zero `.read/*.json` → match nothing + informational note; any empty/unreadable mark → min drops to `""` → match nothing. (d) **`cold/` is a child of the channel dir**, created at mode `0700` (mkdir + explicit `os.chmod` umask-defense, only when absent — the 3a/4d/9c idiom) on first need; `list_messages` skips it (it's a dir), so cold messages vanish from `tail`/`check_messages` but persist on disk. (e) **`prune` joins `_REJECTS_UNKNOWN`** (stray flags → exit 2) and the utility-stub scaffold (`_UTILITY_STUBS`, `_STUB_EXIT_CODE`, `_make_stub_handler`, the registration loop, and `TestUtilityStubs`) is **removed** — prune was the last stub. Matched ids go to **stdout** (data — pipeable), every summary/diagnostic/refusal to **stderr** (10a output discipline); no ANSI, no `--format`/`--color`. A meta-test (`TestIroncladInvariant`) scans every `letterbox/*.py` for deletion primitives (`os.unlink|os.remove|.unlink(|shutil.rmtree|os.rmdir`) and binds each hit to a curated `(module, actual-call-form, justification)` allowlist — NOT a bare count — and separately asserts the prune deletion site exists.
**Rationale:** The matrix reconciliation fits the documented `[--delete --yes-i-am-sure]` surface byte-for-byte while making §3.5's contradiction coherent and keeping "no surprise deletion ever" literally true — reusing `--yes-i-am-sure` as the execute-gate (vs. inventing an `--apply` flag the design outline doesn't show) is the minimal faithful disambiguation. Filename-time aging is the only choice consistent with the doctrine the rest of the system is built on; aging by mtime would give wrong results on any `cp`/backup/syncthing/NFS-migrated channel — this is a deliberate push-back where the literal "by mtime" wording would weaken an established invariant. The *minimum* high-water-mark is correct because "prune only what *everyone* has consumed" is the highest-confidence rule precisely when uncertainty must *reduce* the prune set, never expand it (a `max`/union would prune messages a lagging endpoint hasn't seen — data loss). Reading `.read/*.json` directly keeps prune's filesystem footprint exactly "move/delete `msg-*.json` in the channel root, nothing else" — which is what makes L8 auditable; calling `read_state` would pull in its rename side-effect (wrong for a dry-run-by-default command) and a Tier-2 dependency prune doesn't need. `cold/` as a child (vs. a top-level `state_dir/cold/<channel>/`) keeps a channel's data in one subtree, user-inspectable and cleanly `rm -rf`-able (Sovereignty). The move uses a single `os.replace` of an *existing* file — `os.replace`/`os.rename` are moves, not deletions, so the cold move and the `.broken` corruption-rename are L8-safe by construction and correctly absent from the deletion-primitive scan. The allowlist binds to the actual call-form (e.g. `path.unlink(missing_ok=True)`, not `os.unlink`) so a future swap to a different primitive at the same line fails loudly (a bare count would silently pass a swapped site).
**Resolution:** design spec §3.5's apparent dry-run-vs-move contradiction is resolved by the action/safety matrix this ADR documents — a bare `prune` previews, `--yes-i-am-sure` executes the `cold/` move, and `--delete` upgrades that to a real deletion. Aging is by the message's filename timestamp, never `mtime`, consistent with the codebase-wide time doctrine (ADR-017).
**Supersedes:** None — first record of the `prune` retention contract and the codebase's sole deletion path. Consumes ADR-017 (filename-timestamp ordering — the §3.2 doctrine `--older-than` honors), ADR-021/ADR-028 (the `.read/*.json` per-agent read-state shape + strict message-filename regex that make the stem-vs-stem comparison and `*.json` glob safe), the 2c atomic-rename idiom (`os.replace`), and the 3a/4d/9c mkdir(0o700)+chmod umask-defense pattern (cold/ creation). Builds on ADR-045 (9a's `_REJECTS_UNKNOWN` gate prune now joins; the utility-stub scaffold 9a introduced and 9d retires) and ADR-046 (9c's existence-gate + P3-vector + exit-1 refusal shape, mirrored for prune's preamble). Completes the `Filled in: Phase 9a/9b/9c/9d` promise in `cli.py`'s tier-header — the CLI surface is now feature-complete.

## ADR-046 — `letterbox init` is two distinct steps (existence-precheck refuse + temp-then-`os.replace` publish), regenerates the trailing `[[channels]]` block by string-split for `--channel`, and writes to fixed conventional paths (NOT the `--config`/`LETTERBOX_CONFIG` lever)

**Date:** 2026-05-29
**Context:** Phase 9c replaces `cli.py`'s transitional `init` stub with the real handler — the **first write command** in the CLI surface (9a/9b were dispatch + read-only). Three design choices needed recording. (1) design spec §13.4 / Kernel L6 explicitly names "config files written by `letterbox init`" as bound to write-temp-then-rename, but the obvious one-liner `open(path, "x")` (atomic exclusive create) *looks* like it satisfies both refuse-overwrite and crash-safety in a single call — and doesn't: a crash mid-`"x"`-write leaves a partial file that `init` then politely refuses to re-scaffold over forever, the exact dead-end §13.4 exists to prevent. (2) `--channel NAME` must produce a config *about* `NAME`, not the sample's `debate-01` placeholder plus a leftover — but there is no stdlib TOML *writer* (`tomllib` is read-only; `tomli_w` is not a dependency), so a parse→mutate→re-serialize round-trip is impossible. (3) The 9b read handlers resolve their target via the `--config`/`LETTERBOX_CONFIG` lever (`_resolve_state_dir`); it was tempting to reuse that for init's write target.
**Decision:** (a) **Refuse and durability are two separate steps, never collapsed.** The handler does an existence pre-check (`target.exists()` → Framework-P3 vector citing the path + a "remove it to re-scaffold" hint, exit 1) — that is the refuse-overwrite covenant; then `_atomic_write_new` writes the content to a sibling `<name>.<pid>.tmp`, `flush`+`os.fsync`, and `os.replace`s it onto the target — that is the crash-safe publish (the target never appears half-written; a crash leaves only the temp, which is cleaned up on any error path including `KeyboardInterrupt`). `open("x")` is explicitly rejected. The residual check→replace race (a concurrent `init` could create the target between the check and the replace, which then clobbers it) is documented and accepted: both writes are identical content, and concurrent `init` of the same path is not a real human scenario — locking would be over-engineering (Framework P12). (b) **`--channel NAME` regenerates the trailing block by string-split.** `_build_config_content` reads the 1c bundled sample via `importlib.resources` (UTF-8, so the 📬 `notification_template` emoji round-trips), asserts `sample.count("[[channels]]") == 1` (sample-drift guard), and for `--channel` takes the sample text up to its sole `[[channels]]` token and appends a freshly-templated block. This is robust because TOML array-of-tables must conventionally come **last** (a scalar table after them would re-bind to the last array element) and the sample places it last. `--channel` is validated by `protocol.is_valid_channel_name` *before* templating, so the name is `[a-z0-9_-]` (alphanumeric first char) → a `name = "{NAME}"` string template carries zero TOML-injection risk. Without `--channel`, the sample is written **verbatim** (byte-equal). (c) **init writes to fixed conventional paths** — `Path.cwd() / "letterbox.toml"` (default, ADR-010 project-local) or `Path.home() / ".letterbox" / "config.toml"` (`--global`, HOME-derived, mirroring `config._user_global_config_path`), creating `~/.letterbox/` at mode `0700` (mkdir + explicit `os.chmod` to defeat umask) **only when it was absent** (never re-tightening a dir the user already configured). The `--config`/`LETTERBOX_CONFIG` lever is NOT consulted: that lever selects *which file to read*, not *where to scaffold*. `--global` uses `dest="is_global"` (`global` is a Python keyword), and init joins `_REJECTS_UNKNOWN` so `init --bogus` is a vector error (exit 2).
**Rationale:** The two-step split is the whole point of §13.4 for a *write* command: refuse protects the user's existing work, atomic-publish protects against a partial write becoming a permanent dead-end — `open("x")` satisfies the first while *creating* the second failure mode, so it is worse than the idiom it appears to replace. Reusing `protocol.write_message`'s established temp+`os.replace` shape (vs. an `os.link` no-clobber publish that would collapse the gate and the race) keeps `cli.py` legible against the codebase's existing durability idiom; the link-based race-free variant was available latitude but trades readability for a guarantee against a non-scenario. String-split regeneration is the only available mechanism given no stdlib TOML writer, and the array-of-tables-last convention plus the drift-guard assert make it safe rather than fragile. Writing to fixed paths (not the config lever) is correct because init's job is to create the conventional file `load_config` will later *discover* — routing it through the read-lever would let `--config ./elsewhere.toml init` scaffold somewhere `load_config` won't look by default, defeating the on-ramp. Parent-dir fsync after `os.replace` (2c's durability extension) is intentionally skipped for this one-shot human command — the file fsync is the core §13.4 requirement; a one-shot scaffold does not warrant the directory-fsync cost.
**Supersedes:** None — first record of the `init` write contract. Consumes ADR-010 (project-local-by-default scope), the 2c atomic-write idiom (`protocol.write_message`), and the 3a/3b/4d mkdir(0o700)+chmod umask-defense pattern. Builds on ADR-045 (9a's `_REJECTS_UNKNOWN` gate, which init now joins) and ADR-026 (the per-launch-identity NOTE comment carried into the regenerated `[[channels]]` block — identity is never per-channel-config).

## ADR-045 — Phase 9a CLI dispatch: pre-argparse `mcp` intercept + `parse_known_args` leftover-routing (NOT `argparse.REMAINDER`), with `--config` as a global top-level flag

**Date:** 2026-05-29
**Context:** Phase 9a fills `letterbox/cli.py`'s stub with the top-level argparse dispatch — harness subcommands (`claude`/`gemini`/`antigravity`) → `launcher.run_launcher` under `asyncio.run`; `mcp` → raw-argv forward to `mcp_server.run`; `list-channels`/`tail`/`init`/`prune` → transitional fail-loud stubs (bodies land in 9b–9d). The original design sketched the `mcp` route and the utility-stub catch-alls using `argparse.REMAINDER` positionals, with `parse_args` as the dispatch spine. **Two argparse-3.13 realities broke that sketch.** (1) `argparse.REMAINDER` as a subparser's first positional does NOT capture a *leading optional-looking* token: `letterbox mcp --channel a …` and `letterbox tail --channel x` both bubble `--channel` up as `error: unrecognized arguments: --channel` (exit 2) instead of feeding it to REMAINDER. An early code survey flagged this exact fragility ("REMAINDER is order-sensitive on 3.13; the pre-argparse intercept is the more robust mechanism") and a known requirement is that `letterbox tail --channel x` must NOT crash with "unrecognized arguments". (2) Success criterion #5 (`letterbox --config ./my.toml claude --channel c`) places `--config` *before* the subcommand, which argparse only supports for a flag on the *top-level* parser — not a per-subparser (parent-parser) flag, which would have to come after the subcommand (`letterbox claude --config … ` — an alternative shape that contradicts criterion #5).
**Decision:** (a) **`mcp` is intercepted before argparse** — `if raw and raw[0] == "mcp": return _dispatch_mcp(raw[1:])` — forwarding everything after the `mcp` token byte-for-byte to `mcp_server.run`. This is the canonical agent invocation shape (`generate_mcp_config` always emits `["mcp", "--channel", …]` with `mcp` leading, §7.1), so the intercept handles 100% of real usage and never lets argparse touch the join-key flag spellings. (b) **The dispatch spine uses `parse_known_args`, not `parse_args`.** The `mcp` subparser (registered for `--help`/invalid-choice coherence) and the four utility stubs declare NO flags of their own; `parse_known_args` returns leftover argv in `unknown`, which `_handle_mcp` forwards raw and the stubs ignore. (c) **Harness flag-surface strictness is preserved manually**: after `parse_known_args`, `if args.command in _HARNESSES and unknown: parser.error(…)` — a stray flag before any `--` is a vector error (P3), since the harness flags are a fixed contract, not transitional. (d) **`--config` is a global top-level flag** (sets `os.environ["LETTERBOX_CONFIG"]` to `os.path.abspath(os.path.expanduser(value))` — the environment lever, not `cli_overrides`); per-harness `--channel`/`--as`(`dest="as_label"`)/`--cwd` are added to each harness subparser. (e) **Stub exit code is `1`** (a recognized-but-unimplemented command, distinct from argparse's `2` usage errors). The `--` → harness `extra_args` split stays a manual pre-argparse `_split_passthrough` on the first `--`, as designed.
**Rationale:** `parse_known_args` is the only argparse mechanism that reliably absorbs leading-dash leftovers on 3.13 (REMAINDER is broken for this, `nargs="*"` rejects optionals); it is *simpler* than a fragile REMAINDER positional and honors the actual contract ("raw forward, no reparse" for `mcp`; "don't crash on transitional flags" for stubs) rather than its illustrative mechanism (the design explicitly grants latitude here: the contract is raw forward, not the mechanism). The pre-argparse `mcp` intercept is defense-in-depth + the byte-exact-forward guarantee for the one invocation that actually matters; the registered (flag-less) `mcp` subparser keeps `letterbox --help` and the invalid-choice options list coherent and forwards the non-canonical `letterbox --config x mcp …` ordering through the same `_dispatch_mcp` helper. Global `--config` is dictated by criterion #5's placement; making it top-level (vs. the §10-latitude per-harness shape) is the only way `letterbox --config … claude …` parses. Manual harness-leftover rejection restores the strict-validation that `parse_known_args` relaxes, scoped to the subcommands whose flag surface is a real contract.
**Supersedes:** None. Refines the original REMAINDER sketch (mechanism only — the "raw forward, never reparse" contract is honored unchanged) and resolves the tension over `--config` placement. Consumes ADR-035 (`mcp_server.run(argv)` owns the join-key argparse — 9a forwards to it untouched), Phase 8c (`run_launcher` flat-`0` exit code mapped to `sys.exit` here), and ADR-026 (identity is per-launch; 9a passes `--as` through as `as_label=` and resolves no identity itself). Documented here rather than silently changed (a planned mechanism was replaced for a grounded reason).

## ADR-044 — Phase 8d realizes the two-process exchange as ONE receiver `run_launcher` (B) + a real stdio-client `send_message` (A), and drives A's server body directly (the console `mcp` route is not wired until 9a)

**Date:** 2026-05-28
**Context:** Phase 8d is the launcher-half integration test that finally drives the whole chain end-to-end across two real endpoints over the real stdio transport — closing the gap that the in-memory SDK client is NOT the real stdio transport, and the launcher half of the two-process end-to-end test. The original 8d sketch and design spec §9.2 describe that test as "two letterbox processes, each running fake-harness." Two problems surfaced when realizing that literal shape. **(1) The two-`run_launcher` shape is mechanically unrealizable as a *triggerable send*.** `fake_harness` only echoes stdin and `subprocess.Popen`s+reaps its `--mcp-config` MCP child — it never *calls* a tool, and the test cannot reach that child's stdin (fake_harness owns the pipe). So in a two-`run_launcher` design, neither endpoint's agent ever calls `send_message`; the "send" would have to be faked by a direct `write_message`, which does NOT exercise the MCP tool over stdio and would leave the real-transport gap open (it proves nothing the 8c `TestRunLauncherComposition` test didn't). **(2) The real `letterbox mcp` console-script route is not wired until Phase 9a.** The original design assumed A could drive "the real `letterbox` console script's `mcp` subcommand," but the `mcp` dispatch lives in `cli.py`, whose body is a Phase-9a stub (`cli.main()` prints "CLI not yet implemented (Phase 9a)" and returns 1). 9a comes *after* 8d, so `letterbox mcp …` never reaches `mcp_server.run()` and `stdio_client.initialize()` fails with `McpError: Connection closed`.
**Decision:** Realize the two-process test's launcher half as **one receiver `run_launcher` (B = claude-b) + a real MCP stdio-client `send_message` (A = claude-a) driven by the test acting as A's agent.** The two coordinating letterbox processes are then (1) B's PTY-Parent runtime and (2) the A-side `letterbox mcp` server — coordinating ONLY via the filesystem (the shared `channels/test-e2e` dir), exactly the §2.1 / CLAUDE.md architecture. A's send spawns a genuine stdio MCP server subprocess over the real `mcp.client.stdio` transport and invokes the real `send_message` body (7b), which `write_message`s into the shared channel; B's live watcher detects the peer write, renders, and injects the notification into B's PTY (proven via B's echo file). Because the console `mcp` route is unwired, A spawns the **Phase-7 server body directly** — `command=sys.executable, args=["-c", "import sys; from letterbox.mcp_server import run; run(sys.argv[1:])", "--channel", …]` — the exact `run(sys.argv[1:])` invocation `mcp_server.run`'s own docstring names as supported. A's `StdioServerParameters.env` carries `LETTERBOX_HOME=str(state_dir)` explicitly (the mcp SDK forwards only `['HOME','LOGNAME','PATH','SHELL','TERM','USER']`, NOT `LETTERBOX_HOME`), plus an isolated `HOME` and clean `cwd` (the directory-agreement join key). The process-tree test (design spec §9.2) is **separate** and keeps the *real* `generate_mcp_config` + console-script child — it asserts spawn-tree *position* (PPID == fake_harness, not the PTY-Parent), which holds regardless of whether the CLI stub short-circuits.
**Rationale:** The one-receiver + real-stdio-send shape is *more* faithful to the contract being proven ("A sends via the MCP `send_message` tool over real stdio"), not less — it exercises the genuine transport and server body, which is the entire point of the real-transport requirement; the rejected two-`run_launcher`+`write_message` alternative is cheaper but proves nothing new and leaves that gap open. Driving `mcp_server.run` directly (vs. the console `mcp` route) keeps the transport + server 100% real — only the process *launcher* differs from the not-yet-wired console route, and that route is 9a's concern; this honors the original intent (8d proves the thing 9a will route to actually works end-to-end): 8d proves the server works over real stdio, 9a wires the CLI route to it. Splitting the topology assertion into its own test (real child) from the roundtrip (sleeper child, PATH-independent) keeps each test's flake surface minimal and maps cleanly onto the two distinct design spec §9.2 contracts. **Forward note for 9a:** once `cli.main` dispatches `mcp`→`mcp_server.run`, a follow-up may switch Test 1's A-side `command` to the `letterbox` console script to also exercise the CLI route end-to-end (or 10b's full E2E suite covers it).
**Supersedes:** None — first record of the two-process end-to-end launcher-half realization. Refines the original "two run_launchers" sketch and design spec §9.2's literal wording (documented here rather than silently changed, exactly as 8c flagged its three-way race). Builds on ADR-043 (`run_launcher` composition + cancellation-as-teardown), ADR-040 (the `LETTERBOX_HOME` join key), and ADR-035 (`mcp_server.run(argv)` owns the join-key argparse — the directly-invoked body here).

## ADR-043 — `run_launcher` composes the PTY-Parent runtime via a three-way race, converging every exit path on one idempotent teardown ladder in a `finally`

**Date:** 2026-05-28
**Context:** Phase 8c adds the public blocking `run_launcher` to `letterbox/launcher.py` — the composer that turns `letterbox <harness>` into a runnable, interruptible command (the final piece of the launcher integration). 8a built `setup_launcher` (the started session); 8b built `_injection_loop` (the consumer that drains the watcher queue and injects). Neither could *stop*, and nothing tied them into one thing you can run and Ctrl-C. The §2.1 contract is explicit — *"on clean exit: terminate the harness gracefully, stop the watcher, remove the temp MCP config file."* The Phase 8b→8c transition notes sketched a **two-way** race (injection loop ∨ signal). That is insufficient: between notifications the injection loop is parked on `await queue.get()`, so a harness that exits *quietly* (the common case — the user just quits the agent with no peer message pending) would never wake the loop, and `run_launcher` would hang past the harness's death until a signal arrived.
**Decision:** `run_launcher(harness, channel, *, …, teardown_timeout=5.0) -> int` is a public coroutine (9a owns the `asyncio.run(...)` boundary and `sys.exit`s the returned int). It `await`s `setup_launcher`, installs SIGINT/SIGTERM handlers that set an `asyncio.Event`, then blocks on a **THREE-way** `asyncio.wait({injection_task, proc_task, signal_task}, FIRST_COMPLETED)` — adding an explicit harness-process-exit waiter (`_await_process_exit`, a pure-asyncio `Popen.poll()` + `asyncio.sleep(0.1)` loop) as the third racer. Whatever wakes the race, a `finally` runs the idempotent teardown ladder (`_teardown_runtime`): (1) `remove_signal_handler` first; (2) cancel-and-settle all racers via `asyncio.gather(*racers, return_exceptions=True)` (never re-raises, so a child `CancelledError` or an unexpected loop bug comes back as a result element — logged, never allowed to skip cleanup); (3) `watcher.stop()` → (4) `adapter.teardown(handle, timeout=teardown_timeout)` → (5) `cleanup_mcp_config(path)`, the resource steps nested in `finally`s so each runs even if an earlier one raises. Because teardown lives in the `finally`, **cancelling the `run_launcher` task runs the identical teardown** — so 8d and any multi-launcher embedding tear down via `task.cancel()`, NOT real signals (`add_signal_handler` is one handler per signal per loop). Returns `0` on every clean teardown. Signal installation is wrapped to tolerate `NotImplementedError`/`RuntimeError`/`ValueError` (non-POSIX / off-main-thread): warn and continue, since the process waiter + cancellation still provide teardown.
**Rationale:** The process-exit waiter is what lets us *observe* the clean exit §2.1 promises — the difference between a launcher that returns to the shell and one that hangs; it is correctness, not scope creep (the injection-loop racer still catches a harness dying mid-inject, surfacing 8b's §12 diagnostic; the signal racer catches Ctrl-C; all three converge on one `finally`). Poll over `asyncio.to_thread(process.wait)`: the `sleep` is a clean cancellation point, so teardown cancels the waiter instantly — a `to_thread` wait would keep its worker thread blocked until the process actually died, lingering past the cancel and tripping `filterwarnings=["error"]`. `gather(return_exceptions=True)` over a bare `with suppress(CancelledError): await task`: the injection loop only catches `OSError` and returns (per the Phase 8b design), so any *other* exception (a real bug) would propagate out of a bare-suppress await and **abort the resource-critical teardown**, leaking the harness child + temp file (the L6-forbidden failure); gathering guarantees steps 3-5 still run. The harness teardown reaps the whole process group including the MCP child via `killpg` (per the Phase 6d teardown design), so 8c writes no separate MCP-child reaping. No `queue.task_done()`/`join()`: the notification queue is disposable inference (per the Phase 8b design) — undrained events are dropped because the messages remain on disk (L1). Return code is a flat `0` for all clean teardowns (the recommended v1 minimum; `128 + signum` was available latitude but adds a branch for no v1 consumer benefit — 9a maps the int to `sys.exit`). **Explicit non-goal:** no interactive stdin/stdout I/O bridge — §2.1's PTY-Parent duties are spawn / watch / inject / teardown only; `run_launcher` blocks on lifecycle events, not terminal bytes.
**Supersedes:** None — first record of the `run_launcher` composition + three-way race + teardown-ordering + signal/cancellation model. Improves on the Phase 8b→8c transition notes' two-way race sketch (documented here rather than silently changed). Builds on ADR-042 (the injection loop's cancel-clean, return-on-OSError shape 8c cancels and settles), the 4c cancel-and-settle discipline, and ADR-040/041 (8a's launcher policies). Realizes the `Filled in: Phase 8a/8b/8c` promise in `launcher.py`'s tier-header.

## ADR-042 — The injection loop surfaces a dead-PTY `OSError` to stderr and *stops* (return), rather than swallowing, continuing, or re-raising

**Date:** 2026-05-28
**Context:** Phase 8b adds `_injection_loop(session)` to `letterbox/launcher.py` — the coroutine that drains the watcher's `asyncio.Queue[WatcherEvent]`, renders each event, and injects it via `adapter.inject`. `Adapter.inject` propagates `OSError` (`EIO`/`EBADF`, base.py:210-212) uncaught when the harness's slave fd has closed — i.e. the harness process is gone. design spec §12 (line 853) names *silent injection failure* as a top risk: "the message is lost from the agent's perspective even though it's still on disk; surface this as a visible error, not swallow it." But §12 stops there — it does not say what the loop should then *do* (keep going? stop? re-raise?), and that choice has real consequences. A closed PTY is **permanent**: 5a's `inject_to_pty` loops on short writes, so backpressure *blocks* rather than raises; an `OSError` out of `inject` is never a transient hiccup. The launcher tests run under `filterwarnings = ["error"]` (1b), which escalates an un-retrieved task exception's `RuntimeWarning` into a hard failure.
**Decision:** On `OSError` from the single `inject` call, the loop (a) emits one **`_LOGGER.error(...)`** line naming the harness and the cause (Python's `logging.lastResort` writes WARNING+ to stderr even with zero handler configuration — matching the watcher/protocol `_LOGGER` idiom and not coupling the launcher to a print site), then (b) **returns** — terminating the loop cleanly. It does **not** swallow-and-continue (a closed PTY would spew identical `EIO` on every subsequent event) and does **not** re-raise (a bare-`create_task` coroutine that raises leaves an un-retrieved task exception that `filterwarnings=["error"]` escalates unless 8c retrieves it; returning sidesteps that while still satisfying "do not swallow," because the ERROR line is already out). The `except` clause is **`except OSError` only** — narrow, so `CancelledError` (8c's cancel point) and any genuine bug (e.g. a `TypeError` from a malformed render) propagate loudly.
**Rationale:** Returning lets 8c observe loop completion and proceed to teardown — the natural "the harness died, wind down" signal — without a second exception-plumbing path. Re-raise was rejected not because it is wrong in principle but because it shifts a retrieval obligation onto 8c that returning makes unnecessary; an implementer who prefers re-raise must confirm 8c retrieves the exception (a documented area of latitude). `print(..., file=sys.stderr)` was considered and rejected: the `_LOGGER` path is the house idiom and is zero-config-correct. The narrow `except` is load-bearing — broadening it would swallow `CancelledError` and break 8c's cancel-and-suppress under `filterwarnings=["error"]`. **Testing note (lastResort semantics):** asserting the surfacing uses `caplog` for the ERROR *record*, not `capsys` for stderr bytes — pytest installs a root logging handler, which suppresses `logging.lastResort`, so nothing reaches stderr *under test* even though production (no handlers) writes to stderr. The "reaches stderr in production" property follows from lastResort semantics, not from a capsys assertion.
**Supersedes:** None — first record of the injection-loop error-handling policy. Extends design spec §12 ("surface, don't swallow") with the loop's termination + mechanism choice that §12 left implicit. Complements ADR-041/ADR-040 (8a's other launcher policies) and the 4c cancel-and-suppress pattern that 8c will use against the cancel-clean coroutine shape 8b authors.

## ADR-041 — The launcher resolves the spawned `command`/`default_args`/`notification_template` from `config.harnesses`, not the adapter class attrs

**Date:** 2026-05-28
**Context:** Phase 8a fills `letterbox/launcher.py`, the first consumer of `config.harnesses` (built in 1c, no consumer until now). The adapter registry (5b) returns an `Adapter` instance whose `command`/`default_args`/`notification_template` are `ClassVar` defaults equal to `DEFAULT_HARNESSES` (e.g. `ClaudeAdapter.command == "claude"`). But design spec §6.4 (line 563) is explicit — *"spawn the configured harness CLI with the configured args; both come from `letterbox.toml`"* — and §5.3/§8.2 frame the adapter attrs as TOML *defaults* the user overrides. If the launcher spawned from the adapter class attrs, a user `[harness.claude] command = "/opt/claude"` block in `letterbox.toml` would be silently ignored — a documented v1 feature failing silently, the exact class of failure the framework forbids. The design outline left the config-vs-adapter source-of-truth implicit.
**Decision:** `setup_launcher` reads `harness_config = config.harnesses[harness_name]` and assigns the resolved values as **instance attributes** onto the fresh adapter (`adapter.command = harness_config.command`; `adapter.default_args = list(harness_config.default_args)`), which shadow the `ClassVar` defaults for that one instance. `Adapter.spawn` then composes `[self.command, *self.default_args, *extra_args]` from them. The resolved `notification_template` is validated at startup (ADR-039 chain) and stashed on `LauncherSession.notification_template` for 8b to render. Config is the source of truth; the adapter class attrs are the fallback that lives in `DEFAULT_HARNESSES`.
**Rationale:** `get_adapter` returns a fresh, stateless instance per call (per the Phase 5b design), so per-instance attribute mutation is safe and localized — no need to reopen the sealed 5b spawn API or thread config through it. For the default case the instance attrs equal `DEFAULT_HARNESSES`, so behaviour is unchanged; the override case is the whole point. The alternative (spawn from class attrs) was rejected because it silently drops user config. Assumption (v1): harness names are exactly the three built-ins, so `get_adapter(name)` and `config.harnesses[name]` always agree on the key — `load_config` seeds `harnesses` from `DEFAULT_HARNESSES` and only adds/replaces, never removes. A registered-but-unconfigured harness (a custom adapter with no `[harness.X]` block) is a clean `KeyError` with a vector message, not a crash.
**Supersedes:** None — first record of the config-overrides-adapter-defaults policy. Complements ADR-040 (the launcher's other join-key wiring) and the 5b adapter contract (ADR-018/spawn API), and discharges the §6.4/§5.3/§8.2 "users override the defaults" promise that had no consumer until 8a.

## ADR-040 — The launcher propagates the resolved `state_dir` to the agent + its MCP child via `LETTERBOX_HOME` (the cross-process join key)

**Date:** 2026-05-28
**Context:** Phase 8a spawns the harness, which in turn spawns its own MCP-server child (`letterbox mcp …`, ADR-009/ADR-025). The three processes coordinate ONLY via the filesystem — specifically, all three must operate on the *same* channel directory under the same `state_dir`. The launcher resolves `state_dir` via `config.load_config(cli_overrides).state_dir` (full §8.1 precedence) and opens the channel under it. But the MCP argv (`mcp --channel --as --instance-id`, from Phase 5c) carries **no `--state-dir` token** — so the MCP child re-resolves `state_dir` itself via `load_config()`. If the child's cwd holds a different project-local `letterbox.toml`, or its env differs, it could resolve a *different* directory: both sides would write to different dirs and the conversation would go silently dark with no error (the silent-failure class L1/P3 forbid). Phase 7a flagged this as a forward concern for 8a.
**Decision:** `setup_launcher` builds the adapter spawn env as `{**os.environ, "LETTERBOX_HOME": str(state_dir)}`, where `state_dir` is the exact value it passed to `Channel.get_or_create(..., state_dir=state_dir)`. `LETTERBOX_HOME` is config Level 5 — the highest-precedence override (config.py:594) — so the MCP child's `load_config()` pins to the launcher's value regardless of any project-local `letterbox.toml` or other env in the child's cwd. The spawn env is fully specified (per the Phase 5a design — the adapter does not merge `os.environ`), so the launcher forwards the parent environment explicitly and adds the join key. ADR-038's `list_channels` derives the same dir from the open handle, so all three views agree.
**Rationale:** Env propagation, not a new `--state-dir` flag, is the mechanism — it keeps the MCP argv unchanged and pins the child without widening the join-key surface. `.resolve()` is idempotent on an already-absolute path, so launcher and child agree byte-for-byte. The cost is one env entry; the benefit is structural prevention of the silent channel-split. The join key has a second consumer past 8a (9a's CLI also consumes the resolved `state_dir`).
**Supersedes:** None — first record of the `LETTERBOX_HOME`-as-cross-process-join-key policy. Complements ADR-035/ADR-038 (the MCP child resolves `state_dir` at launch / derives operating paths from the open handle) by pinning *what value* that resolution lands on.

## ADR-039 — Adapter registration via a call-time `load_builtin_adapters()` bootstrap, not package-import side effects

**Date:** 2026-05-28
**Context:** The adapter registry (`letterbox.adapters.base._REGISTRY`) is populated by the `@register_adapter` decorator, which runs as an import side effect of each concrete module (`claude`/`gemini`/`antigravity`, 6a–6c). `get_adapter("claude")` therefore only resolves *after* `letterbox.adapters.claude` has been imported. `letterbox/adapters/__init__.py` was deliberately left empty (6a–6d) so that importing the `letterbox.adapters` package — which sibling modules like `mcp_config` trigger — has **no side effect** and drags in no concrete adapters. Phases 6a–6d and 7a all carried the same forward concern: the green in-process test suite hides the unwired state because 6d imports all three concretes directly; *something* at launch must trigger registration, or the launcher's `get_adapter` raises `KeyError` against an empty registry.
**Decision:** Add `load_builtin_adapters() -> None` to `letterbox/adapters/__init__.py`. It imports the three concrete adapter modules **inside the function body** (call-time), never at module level, and the launcher calls it once during `setup_launcher` before `get_adapter`. The package docstring is updated to state the preserved property explicitly. The bootstrap is verified by a **fresh-interpreter subprocess smoke test** (Phase 8a §10 criterion 10) — the only proof that the bootstrap, not some unrelated test import, does the registration.
**Rationale:** A function-body import (a) preserves the load-bearing "no side effect on `import letterbox.adapters`" property — only *calling* the function registers; (b) keeps the concrete adapter module names out of `launcher.py`, honouring its Tier-4 header ("concrete adapters via the registry only, never direct imports of sibling adapter modules"); (c) co-locates the built-in roster with the adapter package. Module-level imports in `__init__.py` were rejected (they reverse the no-side-effect property, pulling every concrete into any `import letterbox.adapters.mcp_config`); direct concrete imports in `launcher.py` were rejected (they violate the launcher tier-header). Gotcha for tests: the bootstrap fires via the import cache, so after a `reset_registry` empties `_REGISTRY`, calling `load_builtin_adapters()` again does NOT re-register (modules already cached) — registry-resetting tests must register their own adapter.
**Supersedes:** None — first record of the registration-bootstrap policy. Discharges the 6a–6d/7a forward concern (empty `adapters/__init__.py`, registration triggered at the launcher).

## ADR-038 — `list_channels` derives its operating `state_dir` from the open channel handle, not by re-resolving config

**Date:** 2026-05-28
**Context:** Phase 7d fills the `list_channels` MCP tool body in `letterbox/mcp_server.py`. The tool must enumerate every channel under the operating state directory, delegating to the 3d free function `letterbox.channel.list_channels(*, state_dir: Path)`. But `channel.py` cannot import `letterbox.config` (Tier 2 leaf discipline), so the *caller* must supply the absolute `state_dir`. The `_build_server(channel, instance_id)` closure captures the open `Channel` handle and the launch `instance_id` — but **not** `state_dir`. Two sources are available to recover it: (a) re-resolve via `load_config().state_dir` — the same full-precedence resolver `_open_channel` used at launch (the earlier join-key precedent); or (b) derive it from the already-open handle: `channel.path.parent.parent`. `Channel.get_or_create` locks the invariant `channel.path == state_dir / "channels" / <name>` (channel.py:180), so the handle's grandparent path *is* the exact state directory the channel lives in.
**Decision:** `list_channels` derives `state_dir = channel.path.parent.parent` from the open channel handle and passes it to the 3d free function. It does **not** re-resolve config. The derivation is inlined at the call site with a comment naming the `get_or_create` invariant it relies on.
**Rationale:** The open handle is the single authoritative source for the world this channel belongs to. Re-resolving config (option a) re-reads `letterbox.toml` and the `LETTERBOX_HOME`/`LETTERBOX_STATE_DIR` environment on every tool call; if config or env changed between launch and the call (an edited TOML, a mutated env in a long-lived agent process), `list_channels` would enumerate a *different* directory than the one the open channel lives in — a subtle cross-world inconsistency where the agent's "list of channels" excludes the very channel it is talking on. The config resolver is authoritative *at launch* (that is what `_open_channel` correctly uses to decide where to open); once the channel is open, the handle is authoritative for "where am I." Deriving from the handle also avoids a second filesystem/TOML read on a read-only hot path (the 10c `list-channels` budget is P95 < 200 ms) and eliminates a TOCTOU window between resolve and enumerate. The cost is a dependency on the `get_or_create` path invariant, which is already locked by tests and documented in the channel layer — a cheap, durable coupling.
**Supersedes:** None — first record of the derive-operating-paths-from-the-open-handle policy. Complements ADR-035 (`_open_channel` resolves `state_dir` via `load_config().state_dir` *at launch*) by drawing the line: the config resolver decides where to open; the open handle decides where everything else operates.

## ADR-037 — `acknowledge` validates `message_id` wire-format at the MCP boundary; `since_id` is trusted (validate-the-write, trust-the-read)

**Date:** 2026-05-28
**Context:** Phase 7c fills the `check_messages` and `acknowledge` MCP tool bodies in `letterbox/mcp_server.py`. `acknowledge` advances a **persisted** per-agent read marker via `Channel.acknowledge`, whose implementation is `high_water_mark = max(current, message_id)` using a *lexical* string comparison (correct because microsecond-precise filenames sort chronologically — ADR-027). The `Channel.acknowledge` docstring (`channel.py:220`) deliberately defers wire-format validation: *"the message_id is trusted verbatim — no regex check … the MCP boundary at Phase 7c owns wire-format validation."* This leaves a live silent-corruption path: a hallucinated `message_id` that sorts lexically **above** every real id (e.g. `"zzz"`, since every real id begins `"msg-"` and `"msg-…" < "zzz"`) would jump the high-water-mark past the entire channel and **silently blank the agent's whole inbox** — `check_messages`/`check_latest_message` would thereafter return nothing, with no error anyone thinks to look for (the exact silent-failure class L1 and P3 forbid). `check_messages`' other agent-supplied input, `since_id`, is structurally different: `list_unread` uses it only as a one-call `path.stem > since_id` comparison bound; it is never persisted and never advances the marker (ADR-012).
**Decision:** `acknowledge` guards `is_valid_message_filename(f"{message_id}.json")` (the ADR-028 strict regex) **before** delegating to `Channel.acknowledge`, and raises a *vector* `ValueError` naming the rejected value and the expected stem shape on failure — letting FastMCP wrap it as a tool error (the same raise-and-let-the-SDK-wrap surface as `send_message`'s `MessageTooLarge`). `message_id` arrives FastMCP-coerced as `str`, so only its *format* is checked (no type-guard needed at this boundary, unlike the 3a `Channel.get_or_create` sites that accept arbitrary input). `since_id` is **not** validated — it is passed through verbatim. This codifies a boundary policy: **validate the input that writes persisted state; trust the input that only reads.** `limit` is likewise unguarded at the tool (its type-guard + `[1,100]` clamp already live in `list_unread`).
**Rationale:** The validated-write / trusted-read split mirrors the existing `send_message` asymmetry — `in_reply_to` (7b) is passed through unvalidated because it is inert payload, whereas `message_id` mutates a persisted cursor, so the blast radius differs in kind, not degree. Validating `since_id` was rejected: a garbage `since_id` can at worst return wrong results for a single non-mutating call, and the regex would reject legitimate stems the agent paginates with. Returning `{"ok": False, "error": …}` instead of raising was rejected as softer than the established raise-and-wrap pattern and easier for an agent to ignore. The guard lives at the MCP boundary (Tier 4) rather than in `Channel.acknowledge` (Tier 2) by the channel layer's explicit deferral: keeping the regex at the agent-facing edge means a non-MCP caller (a future CLI `ack`, a test) is free to choose its own surface, while the one untrusted-input path that can corrupt state is hard-gated. `acknowledge` is the *only* MCP-boundary input that mutates persisted state, so this is the correctly-scoped single site for the policy (the 7d tools take no agent input).
**Supersedes:** None — first record of the MCP-boundary wire-format-validation policy. Complements ADR-022 (own-write filter; the join-key/identity half of boundary discipline) and ADR-028 (the strict filename regex this guard reuses), and discharges the validation responsibility `Channel.acknowledge`'s docstring deferred to Phase 7c.

## ADR-036 — `check_latest_message` uses a new `Channel.latest_unread` reverse-scan, not `list_unread`-tail

**Date:** 2026-05-28
**Context:** Phase 7b fills the `check_latest_message` MCP tool, whose design spec §6.1 contract is "the single **most recent** unread peer message (or null), minimal context cost, does NOT advance the read marker." The Phase 7a→7b transition notes previewed "reuse `Channel.list_unread`, don't modify Tier 1/2." But `Channel.list_unread` (3c) is a *forward page* — it returns unread messages **oldest-first, capped at `limit` (max 100)**. Taking `list_unread(limit=100).messages[-1]` returns the wrong message whenever unread > 100, and that is not a rare edge: because `check_latest_message` deliberately never advances the `high_water_mark` (§6.1), a peek-only agent (never calls `acknowledge`/`check_messages`) accumulates unbounded unread, so `messages[-1]` would return an ever-staler ~100th-oldest message forever; a peer burst of >100 messages between acknowledges hits the same bug. The correct algorithm is a reverse scan from the newest filename stopping at the first unread peer message (O(trailing own-writes) reads, usually one) — but the own-write filter `_is_own_write` it needs is module-private to `channel.py` (the friend-import contract, ADR-022).
**Decision:** Add `Channel.latest_unread(*, self_instance_id: str) -> Message | None` to Tier-1 `channel.py` (a deliberate, documented edit to a Phase-3-"sealed" module) and route `check_latest_message` through it. The method reverse-scans `reversed(list_messages(self.path))`, breaks at the first stem `<= high_water_mark` (chronological filename sort ⇒ all older are also read), skips prune-race `FileNotFoundError` and `ParseError` files **silently** (no per-file WARN in an unbounded scan, diverging from `list_unread`), and returns the first surviving non-own-write `Message` via the existing `_is_own_write`. The MCP tool body becomes a 2-line serialize wrapper over `_message_to_dict`. Two alternatives rejected: (a) duplicating the OR-filter inline in the Tier-4 tool body — forks the ADR-022 own-write semantic into a second site that would silently drift if the pending OR/AND own-write-filter reconciliation resolves to AND; (b) promoting `_is_own_write` to public and scanning in the tool body — spreads filesystem+filter loop logic into Tier 4 (orchestration) and breaks the `test_is_own_write_is_module_private` lock.
**Rationale:** Putting the scan on `Channel` keeps all scan+filter logic in the tier that owns it and **reuses** `_is_own_write` directly, so `latest_unread` automatically tracks whatever the own-write-filter reconciliation resolves to — single-source-of-truth (Framework "the principle wins over the convenient shortcut"). The cost is ~25 LOC + unit tests in a module Phase 3 sealed; the benefit is correctness for unbounded unread and zero filter duplication. This is the architecturally honest fix for a gap the "reuse `list_unread`" preview did not anticipate. `latest_unread` is the sole "newest unread" tail accessor — `check_messages` (7c) correctly wants `list_unread`'s forward page, and `channel_info` (3d) inlines its own count, so there are no hidden sibling consumers tempted to mis-reuse it.
**Supersedes:** None — first record of the latest-unread accessor. Refines the Phase 7a→7b transition notes' "7b does not modify Tier 1/2" preview (sanctioned override) and complements ADR-022 (combined own-write filter) by adding a second in-module consumer of `_is_own_write`.

## ADR-035 — MCP server: `run(argv)` owns the join-key argparse; high-level `FastMCP`

**Date:** 2026-05-28
**Context:** Phase 7a fills `letterbox/mcp_server.py`, the stdio MCP server the agent spawns as its own child (`letterbox mcp --channel … --as … --instance-id …`, design spec §6.3, ADR-009). Two architectural choices needed pinning. **(a)** The four-token launch invocation is a cross-process join key: 5c's `generate_mcp_config` (locked by a Phase 6d producer-side test) emits `["mcp", "--channel", ch, "--as", sl, "--instance-id", iid]`, and *something* downstream must parse those three flags back out. The risk is the silent-failure mode named in design spec §7.1 — if the parser's flag spellings drift from the producer's (a `--sender` for `--as`, an `--instance_id` underscore, a renamed subcommand), the agent's MCP child parse-errors the instant it spawns and the channel goes dark with no error anyone thinks to look for. Where the spellings live determines how many places can drift. **(b)** The `mcp` SDK exposes both a high-level `FastMCP` (decorator-driven tool registration, auto-generated input JSON-schema from signatures) and a low-level `mcp.server.lowlevel.Server` (hand-written `list_tools`/`call_tool` dispatch). ADR-029 noted both exist; 7a must choose.
**Decision:** **(a) `run(argv)` owns the join-key argparse, and 9a forwards raw argv to it.** `mcp_server._parse_args` defines the three flag spellings `--channel` / `--as` (mapped to `dest="sender_label"` because `as` is a Python keyword) / `--instance-id` in **one** place. 9a's `cli.py` recognizes the `mcp` subcommand token and calls `mcp_server.run(sys.argv[2:])` — everything after `mcp` — letting 7a parse the rest. 9a does **not** define a `mcp` argparse subparser with its own copy of the three flags. **(b) High-level `FastMCP("letterbox")` with `@server.tool()` registration.** Each tool's input schema is generated from the function signature + type hints, so locking the six signatures at 7a locks the agent-facing schema and 7b–7d only fill bodies. The six tools are registered at 7a as fail-loud `NotImplementedError` stubs (a stub raises, naming its filling phase, rather than returning a silent placeholder).
**Rationale:** **(a)** Join-key discipline (§13.3) says a cross-process contract must have exactly one authoritative spelling site, mirrored by a consumer-side lock test. The alternative — a `mcp` subparser in `cli.py` — would duplicate the three spellings across `cli.py` and `mcp_server.py`, creating exactly the two-places-to-drift hazard the contract exists to prevent; 9a forwarding raw argv keeps cli.py ignorant of the join key (it routes on the `mcp` token alone). The Phase 6d test is the producer-side lock; 7a's `TestParseArgs` (canonical spellings parse; `--sender`/`--instance_id`/`--instanceid`/`--channel-name` all rejected) is the consumer-side lock. **(b)** The low-level `Server` would require hand-writing `list_tools`/`call_tool` dispatch and the six input schemas — more surface for zero v1 benefit (ADR-009's stdio transport is satisfied either way; no v1 tool needs the streaming/progress affordances only the low-level API exposes). High-level `FastMCP` is the boring-at-the-core choice (P19); defer the low-level API to a future phase only if a tool ever needs what the decorator can't express.
**Supersedes:** None — first record of the MCP-server bootstrap contract. Refines ADR-009 (stdio MCP transport) and ADR-025 (PTY-Parent ↔ MCP-child coordinate only via the filesystem) by pinning *how* the server parses its launch invocation and *which* `mcp` API it builds on. Supersedes the early `async run(channel, sender_label, instance_id) -> int` interface sketch (which predates the join-key-ownership analysis) — the `run(argv)` shape is what lets 9a forward raw argv.

## ADR-034 — Adapter lifecycle-hook ordering contract (locked end-to-end)

**Date:** 2026-05-28
**Context:** Phase 5b wired four lifecycle hooks (`pre_spawn`/`post_spawn`/`pre_inject`/`pre_teardown`) into the concrete `Adapter.spawn`/`inject`/`teardown` methods (`letterbox/adapters/base.py`) and tested that each hook *fires*. What 5b did not lock is the **relative ordering** of those hooks across a whole session, nor the exact argument each receives. The concrete adapters 6a-c are about to start *relying* on this ordering — a future `post_spawn` startup-context injection assumes the handle already exists; a `pre_inject` per-harness escape assumes it runs before the UTF-8 encode-and-terminate step that ADR-018 mandates. An ordering that is "true by happenstance of how `spawn` is currently written" is a promise written in sand: the day a maintainer reorders `spawn` (calls `post_spawn` before `spawn_pty`, or moves `pre_inject` after the encode) every adapter that trusted the order breaks silently, and the agent goes quiet with no error anyone thinks to look for.
**Decision:** The four-hook lifecycle ordering is a **documented, tested contract**, not an implementation detail. Across one `spawn`→`inject`→`teardown` session the hooks fire in exactly this sequence, each `pre_*` hook *before* the primitive it wraps and `post_spawn` *after*:

```
spawn:    pre_spawn(argv)     → spawn_pty(...)      → post_spawn(handle)
inject:   pre_inject(message) → encode + terminator → inject_to_pty(...)
teardown: pre_teardown(handle)                      → close_pty_handle(...)
```

`pre_spawn` receives the **complete** assembled argv `[command, *default_args, *extra_args]` (command at index 0, not just `extra_args`); `post_spawn` and `pre_teardown` receive the **same** `PTYHandle` object `spawn` returns; `pre_inject` receives the **exact** `str` passed to `inject` (before the encode). All four defaults are identity/no-op, so a zero-override adapter completes a full lifecycle untouched. The contract lives in two complementary homes: the discoverable *what* in the `Adapter` class docstring (where a 6a-c adapter author looks), and the durable *why* here. Phase 5d adds the locking tests (`TestHookOrdering` in `tests/test_adapters_base.py`) but **no functional change** — 5b's composition was verified already correct.
**Rationale:** The ordering is load-bearing for the entire config-driven adapter model (L4): adding a harness must be a config change riding on a stable base, and the base's hook contract is the rail it rides on. Documenting the contract converts an implicit "it happens to work this way" into an explicit guarantee, so the next maintainer who reorders `spawn` gets a red test and a clear ADR instead of a 2 a.m. mystery — small code, long shadow. Two homes rather than one because the class docstring is where an adapter author reads the contract (and `DECISIONS.md` is gitignored until 12c, so less discoverable to them), while the ADR is the permanent decision trail (§13.9). Test-only because making 5d a behavioural edit would mean 5b shipped a bug — it did not.
**Supersedes:** None — first explicit record of the lifecycle-hook ordering contract. Refines (does not contradict) design spec §5.4's "default no-op" hook list by pinning the order and the per-hook argument shape. Note Phase 5b's design forward-guessed "5d does NOT modify base.py"; this ADR documents the deliberate, sanctioned override (the docstring-only edit, blessed by the Phase 5d "documents the hook ordering" deliverable and base.py's own "Filled in: Phase 5b/5d" tier-header).

## ADR-033 — MCP config generation: unified `mcpServers` shape, resolved command path, caller-driven cleanup

**Date:** 2026-05-28
**Context:** Phase 5c lands `letterbox/adapters/mcp_config.py`, the generator for the temp MCP config file the PTY-Parent writes at launch and points the agent's `--mcp-config` flag at (design spec §6.3). The agent reads this file to spawn its own stdio MCP child (`letterbox mcp --channel … --as … --instance-id …`). It is the only handshake between the human-launched `letterbox` process and the agent-launched MCP server, and the two halves coordinate only via the filesystem — so the four identity tokens and the file's mode/shape are a cross-process contract, not a local detail. The design spec phrase "per-harness conventions" needed pinning down: does it mean the *file content* differs per harness, or the *invocation flag*?
**Decision:** (1) **One unified `{"mcpServers": {"letterbox": {command, args}}}` shape for all three v1 harnesses.** Per-harness divergence is the `--mcp-config` *flag* carried by each concrete adapter (6a-c), NOT the file content — the MCP child argv is identical regardless of which harness spawns it. A `_build_payload(harness, …)` internal keeps `harness` as the single documented seam where a *future* harness needing a different file shape would branch; v1 ignores it. (2) **`command = shutil.which("letterbox")`, falling back to the bare string `"letterbox"` (with a WARN) when `which` returns `None`** — embeds the absolute path the PTY-Parent resolved so the agent-spawned MCP child does not silently depend on the agent's own `PATH` (per the Phase 5a design: the spawn env is fully specified, not inherited). (3) **Cleanup is caller-driven via a companion `cleanup_mcp_config(path)` free function**, wired by the launcher at 8c, NOT a self-registered `atexit` — design spec §6.3 step 6 assigns removal to the PTY-Parent's own teardown sequence, which a self-`atexit` would make non-deterministic and invisible. (4) **System-temp `mkstemp` + explicit `os.chmod(0o600)`** (umask defense, 3a precedent), JSON written `ensure_ascii=False` + `sort_keys=True` (ADR-030 deterministic-JSON precedent); the four launch values flow as discrete JSON-array argv tokens, never concatenated into a shell string (consistent with ADR-032's no-shell invariant). **No atomic-rename** — §13.4 governs durable protocol/`letterbox init` writes, not transient runtime config; `mkstemp`'s `O_EXCL` create + read-after-agent-spawn means no partial-visibility window regardless.
**Rationale:** Claude Code's `--mcp-config` consumes the `mcpServers` envelope; Gemini CLI and agy are Gemini-lineage and accept the same envelope — so fabricating three identical per-harness content branches now would be speculative while still honoring the contract structurally through the `_build_payload` seam. Embedding the resolved absolute `command` decouples the MCP-child spawn from whatever `PATH` 8a happens to construct for the agent; the bare-name fallback (with a WARN surfacing the latent `PATH`-coupling risk operationally) keeps behavior no worse than the literal design spec §6.3 text when `which` can't resolve — unusual, since the launcher is itself a `letterbox` invocation. The free-function cleanup pair mirrors the 3a/5a/5b "primitive built here, composed at the launcher" template and fits a signal-driven `finally` better than a context-manager `__exit__`. The explicit `os.chmod` restores the exact `0600` regardless of a pathological umask that could strip owner bits from `mkstemp`'s default. **Open assumption (to confirm in the Phase 13c smoke test):** all three real CLIs accept the unified `mcpServers` envelope via `--mcp-config` — verified for Claude Code's documented format, assumed-compatible for the Gemini-lineage CLIs; if a real CLI rejects it, that harness becomes the first user of the `_build_payload` divergence seam.
**Supersedes:** None — first record of the MCP-config-generation contract. Clarifies (does not contradict) design spec §6.3's "per-harness conventions" as flag-level, not content-level.

## ADR-032 — No-shell invariant: argv-only spawn + CI lint

**Date:** 2026-05-28
**Context:** design spec §6.4 forbids letterbox from `exec`-ing, `eval`-ing, or `subprocess`-shelling any message body or metadata field. Phase 5a (`pty_common.spawn_pty`) is the first and only `subprocess.Popen` call site in the `letterbox/` package; every future spawn site (the launcher in Phase 8a, anything the MCP server might shell out for) inherits the same invariant. Without an explicit structural enforcement, a future contributor "adding shell convenience" could open the no-execution gate without anyone noticing on review — and a peer message body could, in principle, end up reaching `subprocess(body, shell=True)` via a permissive call signature elsewhere.
**Decision:** Two-layer structural enforcement, recorded as a committed invariant:
1. **Function signature.** `spawn_pty(cmd: list[str], ...)` accepts ONLY an argv list. A type-guard at function entry rejects `str` with vector `TypeError`, rejects empty list with `ValueError`, and rejects non-`str` elements with vector `TypeError`. Same shape as 4a's `render_notification` keyword-only signature making prompt-injection via peer payload structurally impossible — 5a's signature makes shell-execution via any input path structurally impossible. A peer's message body cannot become `cmd` without an explicit, deliberate call site that splits it into argv tokens.
2. **CI lint.** `scripts/lint_no_shell_true.sh` (cloned from `lint_ensure_ascii.sh` following the established lint-script pattern, with the "match-and-fail" shape variant) greps for `shell\s*=\s*True` across `letterbox/**.py` and exits non-zero on any hit. Wired into `.github/workflows/test.yml` as the "Lint no shell=True discipline" step. No `# noqa` exemption mechanism (an established anti-pattern); if a future phase ever legitimately needs `shell=True` (it shouldn't given §6.4), that phase amends the lint with an explicit ADR superseding this one.
**Rationale:** Defense in depth. Either layer alone closes most paths to a violation, but the function signature catches what a developer might add locally without re-running the lint, and the lint catches what a developer might add at a site distant from `spawn_pty` (e.g., a future Tier-4 module shelling out for ergonomic reasons). Both layers are structurally cheap — the signature is one type-guard call, the lint is ~40 lines of bash. Together they convert §6.4 from prose into an invariant the Python language and CI gate enforce. Documents the no-shell rule explicitly so it cannot be silently relaxed; any change requires a new ADR superseding this one.
**Supersedes:** None — first explicit committed record of the no-shell invariant. Operationalizes design spec §6.4's prose.

## ADR-031 — PTY primitive: `os.openpty` + `subprocess.Popen` + `start_new_session=True`

**Date:** 2026-05-28
**Context:** Phase 5a lands `letterbox/adapters/pty_common.py`, the leaf module that opens a pseudo-terminal pair and spawns the harness CLI attached to it. The Phase 5a outline sketches the primitive as `pty.fork` / `os.forkpty`, but design spec §5.2's typed `PTYHandle.process: subprocess.Popen` field is the load-bearing source of truth — neither `pty.fork()` nor `os.forkpty()` returns a `Popen`. The implementation must reconcile these, and the choice of spawn primitive determines the teardown shape, signal semantics, and lifecycle observability for every adapter that follows.
**Decision:** `spawn_pty` uses `os.openpty()` to allocate the master/slave fd pair, then `subprocess.Popen(cmd, stdin=slave_fd, stdout=slave_fd, stderr=slave_fd, cwd=cwd, env=env, start_new_session=True, close_fds=True)` to spawn the child attached to the slave. All three child standard streams share one slave fd (PTY semantics). `start_new_session=True` is the default and the production-path value — the child becomes its own session/process group leader so `close_pty_handle` can `os.killpg(handle.pid, ...)` and reach the entire process tree (agent + tools + grandchildren) without signaling letterbox itself. Tests may pass `start_new_session=False` for inspection; production callers do not.
**Rationale:** design spec §5.2 mandates `PTYHandle.process: subprocess.Popen`. `pty.fork()` returns `(pid, master_fd)` only — no `Popen`, no `.wait()` / `.terminate()` / `.poll()` / `.returncode` accessors. `pty.spawn()` is interactive-and-blocks-the-caller, unusable for the async-injection model. The `openpty` + `Popen` pair is the stdlib-only textbook combination (the same primitive `ptyprocess`/`pexpect` use under the hood); using it directly avoids the P19 dependency-skepticism cost of pulling in another library to express what stdlib already offers. `start_new_session=True` is the load-bearing kwarg for safe teardown — without it, `killpg(handle.pid, SIGTERM)` would resolve through letterbox's own pgid and signal letterbox itself, an outright self-kill. The documented caveat is that `setsid` revokes the child's controlling TTY (since Python's `subprocess` does not perform a subsequent `TIOCSCTTY` ioctl); v1's three target harnesses (Claude Code, Gemini CLI, agy) do not require one — they read/write the slave fd directly and never query TTY-layer signal state. A future adapter that does need a controlling TTY would acquire it via a `preexec_fn` doing `os.setsid()` + `os.open(os.ttyname(slave_fd), os.O_RDWR)` (the ptyprocess shape), but that's a future phase's job per Framework L8 frozen-artifact discipline.
**Supersedes:** The "using stdlib `pty.fork`/`os.forkpty`" wording in the Phase 5a outline (which predates the `subprocess.Popen` integration mandate in design spec §5.2). The design spec wins over the outline per the Phase 4a precedent (design-spec/framework principle wins over the planned outline).

## ADR-030 — JSON serialization uses sorted keys for stable round-trip equality

**Date:** 2026-05-27
**Context:** Phase 2a lands `protocol.to_json_bytes` and `from_json_bytes`. Two equal `Message` values (same fields by structural equality) must produce byte-identical JSON output — otherwise refactors that reorder dataclass fields, or differences in dict insertion order across runs (including JSON parses from differently-ordered wire bytes), silently change the on-disk file content. Round-trip byte equality is load-bearing for any future change-detection / hash-based comparison, for diffability of two messages with `cmp`/`diff`, and for the planned round-trip test category (`to_json_bytes(from_json_bytes(to_json_bytes(m))) == to_json_bytes(m)`).
**Decision:** `to_json_bytes` passes `sort_keys=True` to `json.dumps`. This sorts keys alphabetically at every nesting level — top-level message keys, the two `metadata` sub-keys, and any nested objects inside `metadata.ext`. Combined with `ensure_ascii=False` (design spec §13.2), `separators=(",", ":")` (compact, no spaces), and no trailing newline, the output is fully deterministic for equal Messages.
**Rationale:** Without recursive sort, two equal Messages whose `metadata.ext` dicts differ only by key insertion order serialize to different bytes — breaking the deterministic-bytes property this decision rests on. Dict equality is order-independent in Python; the wire format must be equally order-independent. Sorting recursively costs nothing at `protocol.py`'s scale (no message exceeds 5 MB) and prevents an entire class of "two equal Messages produced different bytes" bugs. The wire-key order documented in design spec §3.2 happens to already be alphabetical, so `sort_keys=True` produces exactly the documented order with no special-casing.
**Supersedes:** Implicit "dataclass field order" assumption from design spec §3.2 (which the §3.2 example showed in alphabetical order without naming a mechanism). Locks the mechanism explicitly. Also makes the `metadata.ext` "preserved verbatim across round-trips" promise stronger — the *bytes* may differ (sub-key order may be sorted vs. original), but the parsed Python value is structurally equal in both directions, which is the contract callers care about.

## ADR-028 — Strict filename validation; malformed names ignored

**Date:** 2026-05-27
**Context:** Notification template renders `{message_id}` (and possibly other filename-derived fields) into the text injected into the PTY. If the watcher accepted any file matching `msg-*.json` without strict format validation, a malformed filename like `msg-123-$(ls).json` (or filenames with path traversal sequences) could land in the template substitution path. Even though the injected text goes to an agent (not a shell), the broader principle is: never trust what the filesystem hands you.
**Decision:** All readers and the watcher validate channel-directory filenames against the strict regex `^msg-\d{8}T\d{6}\d{6}-[0-9a-f]{32}\.json$` (updated for UUID4). Files failing the regex are completely ignored — never parsed, never rendered into templates, never returned by `check_messages`. Novel malformed filenames are logged at `WARN` level.
**Rationale:** Defense in depth. Even though the practical injection risk into a PTY-bound LLM agent is low (the LLM doesn't execute strings), strict input validation at the filesystem boundary is cheap insurance against future misuse and against bugs in downstream consumers (e.g., a `letterbox tail --format=html` that might one day render to a browser).
**Supersedes:** Implicit "match anything that looks like a message file" reader behavior.

## ADR-027 — Filenames use full UUID4 (not 4-hex-char) for collision safety

**Date:** 2026-05-27
**Context:** Earlier filename format used `secrets.token_hex(2)` = 4 hex chars = 16-bit randomness = 65,536 combinations. With microsecond timestamp precision, collisions require both same-µs writes and same hex — rare but possible, particularly under burst load or across endpoints with skewed clocks. `os.rename` silently overwrites on POSIX, so a collision = a silently lost message, violating L6 (atomic writes, no silent loss).
**Decision:** Filename random component is `uuid.uuid4().hex` (32 hex chars = 128 bits). Birthday paradox 50% collision at ~2^64 messages — effectively impossible across any realistic history.
**Rationale:** "Collisions are impossible by design" is famous-last-words reasoning when the math says otherwise. UUID4 costs ~28 extra bytes per filename — irrelevant for any practical channel size. Removes the dependence on serialization invariants for safety. Once-and-done.
**Supersedes:** 4-hex-char `secrets.token_hex(2)` in earlier drafts.

## ADR-026 — Identity is per-launch (CLI/env), not per-channel-config

**Date:** 2026-05-27
**Context:** Earlier drafts had `default_sender` / `default_recipient` declared per-channel in `letterbox.toml`. In a shared-project-directory scenario (the most common case — both endpoints in the same repo, reading the same toml), both letterbox processes would pick up the same `default_sender`, collapsing identity. Two Claudes in the same project directory would both write `sender: "claude"`.
**Decision:** Drop `default_sender` and `default_recipient` from per-channel config entirely. Identity resolution at launch: **(1)** `--as <label>` CLI flag (highest priority); **(2)** `LETTERBOX_SENDER` env var; **(3)** harness name (`claude`, `gemini`, `antigravity`) as default. PTY-Parent resolves identity once and propagates to the MCP child via `letterbox mcp --as <label>` arg.
**Rationale:** Identity is a property of the *launch*, not the *channel*. Two terminals on the same channel can (and often do) have different identities — that's the whole point of dialogue. Config files describing channels should describe the channel, not assert which side of it any particular launch represents. The cross-harness case (Claude+Gemini) just works by default because the harness names differ. The same-harness case requires `--as` — empirically how it's always worked in the planning-loop bridge.
**Supersedes:** Per-channel `default_sender`/`default_recipient` config fields. Amends ADR-005's channel config description.

## ADR-025 — MCP server is a separate subcommand spawned by the agent (not co-resident with PTY-Parent)

**Date:** 2026-05-27
**Context:** Initial spec described letterbox as a single process that spawns the harness CLI as a PTY child AND runs the MCP server "co-resident" using stdio transport. This is topologically impossible: stdio-transport MCP servers must be **spawned by** the consuming agent (the agent owns the stdin/stdout the server uses), but in the initial design the letterbox process was the PARENT of the agent — opposite direction.
**Decision:** Split letterbox into two coordinated processes per terminal. **PTY-Parent process** (`letterbox claude/gemini/antigravity`) spawns the harness CLI as a PTY child and runs the filesystem watcher. **MCP server process** (`letterbox mcp` subcommand) is spawned BY the agent as its stdio MCP subprocess child, after the PTY-Parent writes a temp MCP config and passes `--mcp-config` (or equivalent) to the harness. The two processes coordinate ONLY via the filesystem (channel directory + read-state files) — no IPC. The MCP child inherits channel, sender_label, and instance_id from CLI args the PTY-Parent injects into the temp config.
**Rationale:** Honors the actual MCP architecture. Lets us keep stdio transport (ADR-009) without violating process-tree direction. The filesystem-only coordination is consistent with L1 (files as source of truth) and means the two processes are independently testable. From the user's perspective, one terminal still hosts one logical letterbox endpoint — the two-process split is hidden inside the launch.
**Supersedes:** "MCP server is co-resident with the PTY/watcher" framing in earlier drafts. Amends L5 in the framework to clarify "one logical letterbox per terminal, two coordinated processes."

## ADR-024 — Watcher does NOT auto-surface backlog on startup; restart = fresh start

**Date:** 2026-05-27
**Context:** Earlier design had the watcher scan the channel directory at startup and emit notifications for every unread peer message — interpreted via the endpoint's `high_water_mark`. With a long-running channel and a recent crash, this could mean 30+ notifications firing immediately on startup, bombing the agent's context with stale messages and forcing it to respond to historical state when the user may have wanted a clean restart.
**Decision:** Watcher records a "start watermark" at init (the newest filename present, or current UTC timestamp if empty) and fires notifications **only for messages that arrive strictly after that point**. Backlog is not surfaced automatically. The agent has two MCP tools — `check_latest_message()` (returns single newest unread, minimal context cost) and `check_messages(limit, since_id)` (paginated catch-up) — that surface backlog explicitly when the user prompts the agent to check.
**Rationale:** "Wake the Agent" (L3) is about *new arrivals*, not historical replay. A restart often means "fresh start," not "catch me up on everything I missed." User agency is load-bearing: the user is in the loop, and they decide whether catch-up is wanted. Auto-flooding violates that by imposing a response to history. `check_latest_message` was the affordance pattern in the original planning-loop bridge and is reintroduced for the same reason — it gives the user a low-cost way to opt into catch-up without paying the full pagination cost.
**Supersedes:** Earlier implicit assumption that the watcher would rescan-and-fire on startup. Also reframes a portion of ADR-022's cross-restart concern: with no startup rescan, the cross-restart self-notification flood is structurally impossible regardless of filter logic. The combined `(sender OR instance_id)` filter still governs `check_messages` result classification (so the agent doesn't see its own past-session messages when explicitly catching up), but the watcher's auto-notify path no longer needs it.

## ADR-023 — Harness input-queueing is an assumption, not a check

**Date:** 2026-05-27
**Context:** Gemini's review raised concern that PTY injection during a tool-call subprocess could land in the subprocess's stdin (potentially corrupting bash scripts, vim sessions, etc.). The theoretical fix is checking `os.tcgetpgrp(handle.master_fd)` before each injection and deferring when a subprocess is in the foreground.
**Decision:** Document the harness's input-queueing behavior as an **assumption** in letterbox's interoperability contract, not a runtime check letterbox enforces. The three v1 adapters target harnesses (Claude Code, Gemini CLI, Antigravity CLI) that all queue PTY input correctly during tool calls — input received while the agent is busy is buffered and processed once the agent's main loop is free. Adapters for harnesses that *don't* queue properly are responsible for their own coordination.

Adding tcgetpgrp scaffolding to letterbox would solve a problem the v1 adapters don't have. If a future adapter targets a non-queueing harness, that's an adapter-level concern (it can implement queue checks in its `inject` override). The frozen-artifact discipline (L8) favors documenting clear assumptions over building defensive scaffolding for hypothetical cases.

## ADR-022 — Combined own-write filter: `sender` OR `instance_id`

**Date:** 2026-05-27
**Context:** Initial design filtered own-writes by `sender` label (would deadlock same-harness conversations where both endpoints share a sender). ADR-011 switched to `instance_id` filtering (works for same-harness in-session, but on letterbox restart the new process has a new instance_id and would see its own past-session messages as peer messages, flooding notifications). Neither filter alone covers all four cells of the matrix (same/different session × same/different label).
**Decision:** Filter own-writes by `(sender == self.sender_label) OR (instance_id == self.instance_id)`. The sender-label half catches cross-restart self-recognition (the durable label survives a restart that hands the process a new `instance_id`). The `instance_id` half catches the agent's own **same-session** writes (the durable label may legitimately be shared on a same-harness run). The OR's real wins are therefore (a) cross-restart self-recognition via the durable label and (b) clean same-harness operation **when the two endpoints use distinct labels**. When the labels are *identical* (the `--as`-forgotten misconfiguration), an incoming peer message has `msg.sender == self.sender_label`, so it is classified as own-write and dropped — the conversation pauses, and the loud identity-collision WARN (§3.4.1) is the recovery path, not this filter.
**Rationale:** Identity in dialogue systems has two axes — durable (label) and ephemeral (process). Both serve filtering needs. Combining them is cheap (a boolean OR on each incoming message): the OR prevents the **restart-flood** failure mode and enables clean same-harness operation **when labels are distinct**. The **identical-label** case is not silently solved by the filter — when both endpoints share a label, the OR still drops the peer message. The loud identity-collision WARN (§3.4.1) is the recovery there: it prompts the user to set a distinct `--as` label on at least one endpoint.
**Supersedes:** ADR-011's "instance_id is THE Join Key" framing — instance_id is now half of a combined filter, not the sole mechanism.

## ADR-021 — Per-agent read state replaces shared archive

**Date:** 2026-05-27
**Context:** Initial spec had `archive(message_id)` as a physical `os.rename` from the live channel directory to a shared `archive/` subdirectory. Because the channel directory is shared between both peers, archiving was a **global mutation** — when one peer archived a message, that message disappeared from the other peer's live view as well. For dialogue systems this is wrong: "read state" is fundamentally per-agent. If Gemini consumes and archives a message from Claude, that should not delete the message from Claude's `letterbox tail` history or from a third-peer's view (in future n-way topologies).
**Decision:** Replace shared archive with **per-agent read-state files** at `channel/.read/{sender_label}.json`. Each endpoint stores its own `high_water_mark` — the most recent message filename it has acknowledged. Messages remain in the live channel directory for everyone; only the per-agent marker advances. The MCP tool is renamed from `archive(message_id)` to `acknowledge(message_id)`, reflecting that the operation is a local read-state update, not a global filesystem mutation.
**Rationale:** Three benefits. (1) Preserves the agent-facing inbox-clearing affordance (check_messages won't return acknowledged messages for that endpoint). (2) Eliminates the shared-mutation footgun (peer's view is unaffected). (3) Naturally extends to n-way topologies (each endpoint independently tracks its read position). Physical removal of old messages moves to `letterbox prune`, which is user-controlled and never automatic. The high_water_mark optimization (one filename per endpoint, leveraging filename-as-chronological-sort) means read-state files are tiny regardless of channel length.
**Supersedes:** Shared `archive/` directory; the original `archive(message_id)` MCP tool. Obsoletes ADR-019.

## ADR-020 — `in_reply_to` is trusted blindly; no referential integrity check

**Date:** 2026-05-27
**Context:** The `in_reply_to` field references another message ID. A naive implementation would validate that the target exists before accepting the write. This is expensive (channel scan), racy (target may be archiving in flight), and wrong in cross-process scenarios (peer may have just archived the referenced message).
**Decision:** `send_message(body, in_reply_to=...)` accepts any string value for `in_reply_to` without validation. If the target ID doesn't exist or no longer exists, that's a readability issue downstream, not a protocol error. Implementing Claudes are explicitly forbidden from adding a validation scanner.
**Rationale:** Async file-based systems can't have strong referential integrity without locking and centralization, both of which violate L1/L7. Referential integrity at write time would also break in legitimate cases (peer archives before reply lands). Treating `in_reply_to` as informational metadata that consumers MAY use, rather than a foreign key the protocol enforces, is the only consistent model.
**Supersedes:** No prior decision; codifies the implicit stance against validation.

## ADR-018 — PTY injection always appends `\r` to force agent consumption

**Date:** 2026-05-27
**Context:** Initial spec described notification injection as "write to PTY master fd." This is necessary but not sufficient: a terminal agent reads stdin line-buffered, and bytes without a trailing carriage return just sit in the prompt buffer waiting for a human Enter key. The agent never wakes — violating L3 (Wake the Agent), the load-bearing kernel principle.
**Decision:** Every `Adapter.inject()` post-processes the rendered notification by appending `\r`. Enforced in the base class — adapters cannot skip it. If a future harness needs a different terminator, that's a per-adapter `line_terminator` class attribute override, not optional behavior.
**Rationale:** Forgetting the CR is the kind of bug that ships and then mysteriously fails in production. Baking it into the base class as non-optional makes it impossible to forget. Carriage-return is the canonical PTY input terminator; line discipline converts it to `\n` for the consuming process as needed.
**Supersedes:** "Write to PTY master fd" (incomplete — missing the terminator).

## ADR-017 — Microsecond precision in filenames; chronological sort by filename, not `mtime`

**Date:** 2026-05-27
**Context:** Initial filename format was `msg-YYYYMMDD-HHMMSS-XXXX.json`. Two messages in the same second would tie on the timestamp portion, and the random hex `XXXX` would order them incorrectly. The proposed fix was to sort by `mtime` instead — but `mtime` is fragile: `cp -r` (without `-p`), syncthing, rsync, NFS clock skew, and `tar` extraction can all reset or shuffle mtime, silently mangling conversation history.
**Decision:** Filename becomes `msg-YYYYMMDDTHHMMSSffffff-XXXX.json` with 6-digit microseconds. Lexical filename sort is chronologically correct independent of any filesystem metadata. All sorting in letterbox uses filename order, never `mtime`.
**Rationale:** Conversation history must survive standard filesystem operations (backups, syncs, archives). Microsecond precision is sufficient because same-endpoint writes are serialized and cross-endpoint collisions at 1µs resolution are impossible across any meaningful comm link. The cost is 7 extra chars per filename — irrelevant.
**Supersedes:** Second-precision filename + `mtime`-based sort.

## ADR-016 — `.tmp` files: ignored on read, reaped only on startup with mtime > 1h

**Date:** 2026-05-27
**Context:** Initial spec said orphaned `msg-*.json.tmp` files would be "deleted on next channel access." This is a textbook race condition: a `.tmp` file may be a live write in progress, and aggressive reaping by one process would silently corrupt writes by another.
**Decision:** Two-part rule. **(1)** Readers never look at `.tmp` files — the message-file glob (`msg-*.json`) structurally excludes them. **(2)** A reaper sweep runs only on `letterbox` startup (not on every access) and deletes only `.tmp` files with `mtime` older than 1 hour. Fresh `.tmp` files are preserved.
**Rationale:** Atomic-rename writes mean a `.tmp` file is fleeting under normal operation (microseconds between write+rename). A `.tmp` older than 1 hour is almost certainly orphaned. The 1-hour window protects against accidentally killing a slow legitimate write while still cleaning up real garbage.
**Supersedes:** "Delete on next access" reaper policy.

## ADR-015 — UTC timestamps everywhere; local time forbidden

**Date:** 2026-05-27
**Context:** Initial spec said filenames use `YYYYMMDD-HHMMSS` without specifying timezone. Local time would silently mangle chronological sorting across timezone changes, daylight-saving transitions, and machines in different zones (NFS-shared channels).
**Decision:** All timestamps — filenames AND the `timestamp` JSON field — are UTC. Generated via `datetime.now(timezone.utc)`, never `datetime.now()`. Filenames carry no offset suffix (to keep them portable across filesystems that disallow `:` or `+`); the JSON `timestamp` field includes explicit `+00:00`.
**Rationale:** Chronological order across letterbox files is load-bearing — `check_messages` returns newest-first, archives sort by filename, recovery semantics depend on `mtime` and filename order. Local time in any of those produces silent data corruption when timezones shift. UTC is the only safe answer for filesystem-as-database systems.
**Supersedes:** Implicit local-time assumption in initial filename spec.

## ADR-014 — 5MB ceiling on message body size

**Date:** 2026-05-27
**Context:** Initial spec had no upper bound on message body size. A looping agent (or a malicious peer) could write arbitrarily large payloads — gigabyte JSON files would OOM the letterbox process when `check_messages` parsed them.
**Decision:** Hard ceiling of 5 MB on message body. Enforced on write (rejected with `MessageTooLarge` before disk I/O) and on read (oversized files surface as `parse_error: "oversized"` with `body: null`).
**Rationale:** Capping the worst-case parse cost is essential for DoS resistance. 5 MB is generous for human-readable text (≈1M words) but small enough to bound memory predictably. Larger payloads belong in shared filesystem locations referenced *by* message body, not in the body itself.
**Supersedes:** Unbounded body assumption in earlier drafts.

## ADR-013 — Watcher belt-and-suspenders (event observer + polling fallback)

**Date:** 2026-05-27
**Context:** Initial spec relied solely on `watchdog`'s event-driven observer (inotify on Linux, FSEvents on macOS) for new-message detection. Filesystem events are documented to be lossy under load, on certain filesystems (network mounts, FUSE), and during observer hiccups. A single dropped event silently breaks the conversation — peer agent sleeps forever — violating L3 (Wake the Agent) and Framework P5 (Self-Healing).
**Decision:** Watcher uses both paths in parallel. Primary: watchdog observer for low-latency detection (sub-second). Fallback: re-scan polling loop every 5 seconds. Both feed the same notification queue, deduplicated by filename + instance_id. If a watchdog event drops, the polling fallback catches the message within at most 5 seconds.
**Rationale:** Comms infrastructure cannot have silent-loss failure modes. The cost of belt-and-suspenders is a 5s background poll on a directory — negligible CPU. The benefit is that letterbox is self-healing against filesystem event flakiness, which is a documented real-world failure mode on every OS letterbox supports.
**Supersedes:** Event-observer-only design.

## ADR-012 — `check_messages` pagination (default `limit=20`, max `100`, `since_id` cursor)

**Date:** 2026-05-27
**Context:** Initial spec had `check_messages()` return all unarchived peer messages newest-first. With the §9.4 budget allowing up to 1000 unarchived messages per channel, a single tool call could return ~1000 fully-parsed message bodies — instantly destroying the consuming agent's context window and blowing the latency budget.
**Decision:** `check_messages(limit=20, since_id=None)`. Default returns at most 20 newest messages. Hard ceiling at 100 (above that, clamped with warning). `since_id` provides cursor-based pagination ("messages newer than this ID"). Response includes `has_more: bool`.
**Rationale:** Context windows are precious; an agent that swallows 1000 messages in one call is broken regardless of what letterbox returns. Pagination is the standard mechanism. 20 is large enough to handle typical "what did I miss since I last checked" cases and small enough to never bomb a context window. 100 ceiling handles the legitimate "I'm catching up after a long absence" case while still capping worst-case parse cost.
**Supersedes:** Initial unbounded `check_messages()` signature.

## ADR-011 — `instance_id` as the watcher's Join Key (not `sender`)

**Date:** 2026-05-27
**Context:** Initial spec implied the watcher would filter own-writes by matching the `sender` field against the watcher's own sender label. This works for cross-harness topologies (`claude` vs `gemini`) but breaks for **same-harness topologies** (two Claude Code tabs talking) where both endpoints share a sender label — the watcher would filter every message as its own and the conversation would deadlock silently.
**Decision:** Every `letterbox` process generates a per-instance `instance_id` at startup (`lb-{ISO8601}-{6-hex}`) and writes it on every outbound message. The watcher filters peer messages by `instance_id != self.instance_id`. The `sender` field becomes purely human-readable (what shows in `letterbox tail` and message logs); `instance_id` becomes the mechanical filter.
**Rationale:** Decouples human identity from mechanical identity. Enables any topology (cross-harness, same-harness, three-way) without configuration gymnastics. The cost is one JSON field per message and a startup UUID generation — negligible. Catches what would otherwise be a Day-1 footgun for anyone trying to run two tabs of the same harness.
**Supersedes:** Implicit "filter by sender label" assumption in earlier drafts. Later refined by ADR-022 into a combined filter.

## ADR-010 — `letterbox init` scope: project-local by default

**Date:** 2026-05-27
**Context:** `letterbox init` could scaffold a global config (`~/.letterbox/config.toml`) or a project-local one (`./letterbox.toml`).
**Decision:** Project-local by default. `--global` flag writes user-global instead.
**Rationale:** Project-local matches the per-project mental model letterbox is built around (one project, one channel set, one config). Global config is for cross-project defaults and is opt-in. Aligns with how `git init` and similar tools behave.
**Supersedes:** None.

## ADR-009 — MCP server transport: stdio

**Date:** 2026-05-27
**Context:** MCP supports stdio and HTTP transports. Choose for v1.
**Decision:** stdio.
**Rationale:** All three v1 harnesses (Claude Code, Gemini CLI, Antigravity) expect stdio-transport MCP servers as the default; stdio avoids the operational complexity of binding a local HTTP port (port conflicts, firewall rules, lifecycle management). HTTP transport remains a clean future adapter if needed.
**Supersedes:** None.

## ADR-008 — Notification template language: stdlib `str.format`

**Date:** 2026-05-27
**Context:** Notification templates need variable substitution. Considered Jinja2 (flexible, heavy), stdlib `str.format` (minimal, sufficient), and custom mini-DSL.
**Decision:** Stdlib `str.format` with a whitelist of allowed variable names (`{channel}`, `{sender}`, `{message_id}`, `{timestamp}`).
**Rationale:** The substitution needs are simple — fixed-name variables, no conditionals, no loops, no filters. Jinja2 adds a heavy dependency and an attack surface (template injection) for zero benefit at this scope. The whitelist in `notifications.py` enforces that only safe variables resolve.
**Supersedes:** None.

## ADR-007 — Python version floor: 3.10

**Date:** 2026-05-27
**Context:** Choose between Python 3.10 (broader compatibility, needs `tomli` fallback) and 3.11 (stdlib `tomllib`, narrower compatibility window).
**Decision:** 3.10 minimum, with `tomli` as a runtime dependency.
**Rationale:** 3.10 catches users on long-lived distros and corporate-managed Python installs. `tomli` is a single-purpose mature dependency that drops automatically when 3.11 becomes the floor.
**Supersedes:** None.

## ADR-006 — License: MIT

**Date:** 2026-05-27
**Context:** OSS license choice for the public release. Apache 2.0 and MIT both considered.
**Decision:** MIT.
**Rationale:** Permissive, minimal compliance overhead, well understood by every contributor and corporate user, shortest LICENSE file. Apache 2.0's patent grant is valuable but unnecessary for a small filesystem-comms primitive with no novel patentable mechanism. MIT keeps the legal surface trivial for forks.
**Supersedes:** None — first license decision.

## ADR-005 — Channels are first-class, declared in config (not env-var-only)

**Date:** 2026-05-26
**Context:** Current bridge implementation reads `FORGE_DIALOGUE_CHANNEL` env var with a hardcoded fallback. Adding channels feels like an afterthought.
**Decision:** Channels are a typed concept in the library (`Channel` class), declared in `letterbox.toml`, and explicitly named on the CLI (`letterbox claude --channel <name>`).
**Rationale:** Multiple concurrent projects sharing one machine need clean channel isolation. Making channels first-class also clarifies the mental model: a letterbox session is *between two endpoints on a named channel*, not "the bridge."
**Supersedes:** Current `FORGE_DIALOGUE_CHANNEL` convention — vestigial name, replaced by explicit `--channel` arg + config. Later amended by ADR-026 to remove `default_sender` / `default_recipient` from per-channel config (identity is per-launch, not per-channel).

## ADR-003 — Real-time injection, not polling-by-instruction

**Date:** 2026-05-26
**Context:** Considered making letterbox a pure MCP server with `check_messages` tool, leaving it to the agent to decide when to check.
**Decision:** Letterbox owns a launcher that spawns the agent CLI as a PTY child and injects "new message" notifications into stdin when a peer writes.
**Rationale:** Polling-by-instruction is slower than copy-paste between windows — verified empirically during planning-loop development. The agent has to be *woken* for real-time comms to feel real.
**Supersedes:** None.

## ADR-002 — Files are the source of truth (no daemon, no network)

**Date:** 2026-05-26
**Context:** Several alternatives considered for AI-to-AI messaging: HTTP server, message queue, named pipes, filesystem.
**Decision:** Filesystem only. A channel is a directory; a message is a file with a deterministic name.
**Rationale:** Two processes on one machine. Files give us durability, inspectability (`cat`, `tail -f`), trivial crash recovery, and atomic-rename as a write primitive. Anything heavier is accidental complexity at this scale.
**Supersedes:** None.

## ADR-001 — Frozen artifact, not maintained project

**Date:** 2026-05-26
**Context:** Letterbox is being extracted from a planning loop as a standalone OSS artifact.
**Decision:** Ship v1 as a frozen artifact. Complete, documented, MIT-licensed, but with no commitment to issues, PRs, or releases beyond v1.
**Rationale:** Vinga doesn't use social media [[feedback_no_social_media]] and has no bandwidth to nurture a community. Honest framing ("here is the system as of June 2026") beats implied promises of maintenance.
**Supersedes:** None — first decision.
