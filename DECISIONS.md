## ADR-049 — The `two-claudes-debating` sample ships its personas in BOTH `AGENTS.md` and `CLAUDE.md` (Phase 13c)

**Date:** 2026-05-29
**Context:** The sample's two personas were authored only as `claude-{a,b}/AGENTS.md`. Phase 13c's G2 probe (a live `claude -p` against throwaway dirs holding distinct codewords) established that **Claude Code 2.1.156 reads `CLAUDE.md` from its cwd and does NOT read `AGENTS.md`** — so an `AGENTS.md`-only persona silently fails to load for a real `letterbox claude` run, leaving the demo with no debate stances.
**Decision:** Each persona dir ships **both** files with identical content (`AGENTS.md` is the source; `CLAUDE.md` is a duplicate, `cp`'d). Duplicating rather than renaming keeps the sample harness-agnostic — it works whether a harness reads `CLAUDE.md` (Claude Code) or `AGENTS.md` (the broader cross-tool convention). The files are tiny and the artifact is frozen, so the sync cost is negligible. Re-verified post-fix: a `claude -p` from `claude-a/` reported *"My assigned stance is 'A hot dog IS a sandwich,' loaded from `CLAUDE.md`."*
**Rationale / `.gitignore` consequence:** The repo-wide ignore list carries an *unanchored* `CLAUDE.md` rule (to keep the private *root* collaboration file out of the public repo). That rule would also catch the persona `CLAUDE.md` files at any depth, so two explicit negation entries (`!examples/two-claudes-debating/claude-{a,b}/CLAUDE.md`) exempt them — the root `CLAUDE.md` stays private, the sample `CLAUDE.md` files ship. Verified with `git check-ignore`.
**Supersedes:** None. Refines the Phase 11a sample (the `debate-01` four-file artifact) so its personas actually load against the real `claude` harness.

## ADR-048 — Release packaging (Phase 13a): version `1.0.0` synced across two literals, PEP 639 SPDX license form (driven by the live build deprecation), `[project.urls]` → personal `dovahkiin-v` repo, Development Status `5 - Production/Stable`

**Date:** 2026-05-29
**Context:** Phase 13a — the first of three release-prep phases — discharges Vision §16 Done-Criterion #5 (*"LICENSE in place; `pyproject.toml` is publishable, even if not pushed to PyPI"*). The 1a `pyproject.toml` carried a `0.0.0` placeholder (in two lockstep literals — `pyproject.toml` and `letterbox/__init__.py`), no `keywords`, no `[project.urls]`, the `license = { text = "MIT" }` TOML-table form, and a `License :: OSI Approved :: MIT License` classifier. Four release decisions needed grounding — chiefly the license declaration form, which the plan (K3) mandated be driven by *actual* build output rather than guessed.
**Decision:** (a) **Release version `1.0.0`**, set in both `pyproject.toml` and `letterbox/__init__.py` as static literals kept in lockstep, with a new static guard test (`tests/test_packaging_meta.py`) asserting `version != "0.0.0"` and `version == letterbox.__version__`. The dynamic-`attr` single-source alternative (K1 latitude) was declined — two literals + a guard test is less build machinery for a frozen artifact, and the guard makes drift impossible anyway. `1.0.0` (not `0.x`) is the honest version for ADR-001's frozen, complete v1. (b) **PEP 639 SPDX license form**: `license = "MIT"` + `license-files = ["LICENSE"]`, `[build-system] requires` bumped `setuptools>=68 → >=77` (SPDX string needs ≥77), and the deprecated `License ::` classifier removed. This was **driven by the live build**: the first `python -m build` (with the table form) emitted two `SetuptoolsDeprecationWarning`s — *"`project.license` as a TOML table is deprecated … remove deprecated calls by 2027-Feb-18 or builds will no longer be supported"* and *"License classifiers are deprecated"* — so the table form has a hard sunset, not just a stylistic nit. The rebuilt wheel's METADATA carries `License-Expression: MIT` and is warning-clean. (c) **`[project.urls]`** Homepage/Repository/Documentation all point at `https://github.com/dovahkiin-v/letterbox` (the personal repo, never skyforge-sh — the positioning is load-bearing). (d) **`keywords`** = `["mcp", "ai-agents", "agent-communication", "ipc", "cli", "pty", "claude-code"]`; **classifiers** finalized to `Development Status :: 5 - Production/Stable` (an ADR-001 frozen-and-complete v1 is Production, not Beta) plus `Environment :: Console`, `Intended Audience :: Developers`, `Topic :: Communications`, and `Operating System :: POSIX :: Linux` / `MacOS`; **`authors`** name aligned `Vinga` → `Vinga Luksaite` for LICENSE consistency. Build proof (recorded in IMPLEMENTATION_NOTES): `python -m build` exits 0 producing both wheel + sdist; the wheel ships `letterbox/data/sample_letterbox.toml` **and** `letterbox/data/__init__.py`; a fresh `/tmp/lb-smoke` venv installs the wheel, `letterbox --help` + all eight subcommand `--help`s exit 0, a real `letterbox init` writes the 705-byte sample config (proving the bundled resource resolves post-install), and `pip check` is clean. The pytest/CI config (`addopts`, `--cov-fail-under=0`, the `budget` marker), `.github/workflows/test.yml`, and every `letterbox/*.py` except the one `__version__` line are byte-untouched (13b/13c territory).
**Rationale:** Grounding K3 in the observed warning (rather than the plan's prediction) is the "Agency, Anchored" discipline — the table form is valid *today* but carries an explicit 2027 sunset, so for a millennial-durability frozen artifact (Framework P19, L8) the SPDX form is the choice that ages without a future forced edit. Static literals + guard test over dynamic version keeps the build boring and the invariant enforced. Production/Stable over Beta is the honest signal for a v1 that ADR-001 declares complete.
**Flag for vision review:** Phase 13a deliberately did **not** fix two known `letterbox/*.py` logic issues, as they are out of a release-metadata phase's no-source-logic scope (plan K5/§11) and belong in a scoped remediation: (1) the unguarded `config.load_config()` → `ConfigError` *traceback* (instead of a Framework-P3 stderr vector) on a bad `--config` path, present in **both** `cli.py` (`_resolve_state_dir`) and `launcher.py:194` (`setup_launcher`); and (2) the `--strict-fsync` phantom-flag docstrings referencing an unimplemented CLI flag in `protocol.py:649` + `channel.py:820`. Neither is architecture-breaking; both are re-flagged here and in the 13a handoff for vision-review → `start_remediation`.
**Supersedes:** None. Resolves the `0.0.0` placeholder the 12a README flagged for 13a. Consumes ADR-001 (frozen complete v1 → `1.0.0`), ADR-006 (MIT), ADR-007 (Python 3.10 floor), and the 1c bundled-data packaging (`[tool.setuptools.package-data]` + `letterbox/data/__init__.py`) whose wheel-presence this phase proves. Does not touch ADR-029 (the `mcp` pin) or any pytest/CI decision (13b).

## ADR-047 — `letterbox prune` is dry-run by default with a four-row action/safety matrix (`--yes-i-am-sure` executes, `--delete` upgrades move→delete), ages by the filename timestamp (NOT mtime), takes the *minimum* high-water-mark for `--acknowledged-by-all`, and is provably the sole user-message deletion path (L8)

**Date:** 2026-05-29
**Context:** Phase 9d replaces `cli.py`'s last transitional `prune` stub with the real retention command — the **only** way a user reclaims space in an unbounded channel (Vision §3.5, which ships no automatic retention) and the **single destructive command** in the entire codebase. Four design questions had to be resolved, two of which are genuine contradictions/imprecisions in the source docs. (1) **Dry-run vs. move (K2).** Vision §3.5 says prune *both* "defaults to dry-run (touches nothing)" *and* "moves matching messages to a `cold/` subdirectory" — a bare `prune` cannot do both. The PHASE_INDEX flag surface shows only `[--delete --yes-i-am-sure]` (no `--apply`/`--execute`). (2) **Age by mtime vs. filename (K4).** The PHASE_INDEX *test row* says "`--older-than 7d` matches by mtime," but the entire codebase time-doctrine is filename-authoritative (§3.2 / ADR-017; `list_messages`'s own docstring states mtime is fragile under `cp -r`/syncthing/NFS). (3) **`--acknowledged-by-all` ceiling** — min or max high-water-mark across endpoints? (4) **Where the cold-store lives** and how the move/delete primitives stay L8-auditable. The phase also empties and removes the transitional utility-stub scaffold (prune was the last stub).
**Decision:** (a) **Four-row action/safety matrix, no new flag.** `--yes-i-am-sure` is the universal "stop previewing, actually do it" execute-gate; `--delete` upgrades the executed action from a `cold/` move to a real deletion. Bare selection = dry-run preview (matched ids → stdout, "would move N to cold/" → stderr, touch nothing, exit 0). `--yes-i-am-sure` alone = move matched files to `<channel>/cold/` via per-file `os.replace` (atomic, reversible). `--delete --yes-i-am-sure` = `os.unlink` each matched file (the one L8 deletion path). `--delete` *without* consent = explicit refusal (Framework-P3 vector naming `--yes-i-am-sure`, exit 1, nothing happens). The handler checks the refusal gate *before* resolving anything. Exactly one selection rule is required via a `required=True` mutually-exclusive argparse group, so a bare `prune --channel x` is an exit-2 usage error — there is no "prune everything by accident." (b) **`--older-than` ages by the message's filename timestamp**, parsed with `datetime.strptime(seg, "%Y%m%dT%H%M%S%f").replace(tzinfo=utc)` (4-digit-year-safe — avoids the glibc `strftime("%Y")` zero-pad bug 2b/3d hit) and compared `msg_time < now - duration`; only called on `list_messages`-returned paths, which are `is_valid_message_filename`-gated so the segment is well-formed. Duration grammar is `\d+[dhms]`, wired as the argparse `type=` callable so junk is an exit-2 vector. (c) **`--acknowledged-by-all` takes the MINIMUM high-water-mark** across all `.read/*.json` endpoints (read directly via `json.loads(path.read_bytes())`, **never** `channel.read_state` — whose corruption-recovery rename side-effect would mutate read-state), matching ids whose stem is `<=` that ceiling (inclusive). Two fail-safes: zero `.read/*.json` → match nothing + informational note; any empty/unreadable mark → min drops to `""` → match nothing. (d) **`cold/` is a child of the channel dir**, created at mode `0700` (mkdir + explicit `os.chmod` umask-defense, only when absent — the 3a/4d/9c idiom) on first need; `list_messages` skips it (it's a dir), so cold messages vanish from `tail`/`check_messages` but persist on disk. (e) **`prune` joins `_REJECTS_UNKNOWN`** (stray flags → exit 2) and the utility-stub scaffold (`_UTILITY_STUBS`, `_STUB_EXIT_CODE`, `_make_stub_handler`, the registration loop, and `TestUtilityStubs`) is **removed** — prune was the last stub. Matched ids go to **stdout** (data — pipeable), every summary/diagnostic/refusal to **stderr** (10a output discipline); no ANSI, no `--format`/`--color`. A meta-test (`TestIroncladInvariant`) scans every `letterbox/*.py` for deletion primitives (`os.unlink|os.remove|.unlink(|shutil.rmtree|os.rmdir`) and binds each hit to a curated `(module, actual-call-form, justification)` allowlist — NOT a bare count — and separately asserts the prune deletion site exists.
**Rationale:** The matrix reconciliation fits the documented `[--delete --yes-i-am-sure]` surface byte-for-byte while making §3.5's contradiction coherent and keeping "no surprise deletion ever" literally true — reusing `--yes-i-am-sure` as the execute-gate (vs. inventing an `--apply` flag the PHASE_INDEX doesn't show) is the minimal faithful disambiguation. Filename-time aging is the only choice consistent with the doctrine the rest of the system is built on; aging by mtime would give wrong results on any `cp`/backup/syncthing/NFS-migrated channel — this is an "Agency, Anchored" push-back where the literal PHASE_INDEX "by mtime" wording would weaken an established invariant. The *minimum* high-water-mark is correct because "prune only what *everyone* has consumed" is the highest-confidence rule precisely when uncertainty must *reduce* the prune set, never expand it (a `max`/union would prune messages a lagging endpoint hasn't seen — data loss). Reading `.read/*.json` directly keeps prune's filesystem footprint exactly "move/delete `msg-*.json` in the channel root, nothing else" — which is what makes L8 auditable; calling `read_state` would pull in its rename side-effect (wrong for a dry-run-by-default command) and a Tier-2 dependency prune doesn't need (K1). `cold/` as a child (vs. a top-level `state_dir/cold/<channel>/`) keeps a channel's data in one subtree, user-inspectable and cleanly `rm -rf`-able (Sovereignty). The move uses a single `os.replace` of an *existing* file — `os.replace`/`os.rename` are moves, not deletions, so the cold move and the `.broken` corruption-rename are L8-safe by construction and correctly absent from the deletion-primitive scan. The allowlist binds to the actual call-form (e.g. `path.unlink(missing_ok=True)`, not `os.unlink`) so a future swap to a different primitive at the same line fails loudly (PLANNING_NOTES versioned-assertion lesson — a bare count silently passes a swapped site).
**Flag for vision review:** two source-doc reconciliations are recorded here as conscious decisions, not loose ends — (1) Vision §3.5's dry-run-vs-move contradiction (resolved via the `--yes-i-am-sure`/`--delete` matrix) and (2) the PHASE_INDEX 9d test row's "matches by mtime" wording (the implementation and tests assert filename-time behavior; the index row should be reconciled to say "filename timestamp").
**Supersedes:** None — first record of the `prune` retention contract and the codebase's sole deletion path. Consumes ADR-017 (filename-timestamp ordering — the §3.2 doctrine `--older-than` honors), ADR-021/ADR-028 (the `.read/*.json` per-agent read-state shape + strict message-filename regex that make the stem-vs-stem comparison and `*.json` glob safe), the 2c atomic-rename idiom (`os.replace`), and the 3a/4d/9c mkdir(0o700)+chmod umask-defense pattern (cold/ creation). Builds on ADR-045 (9a's `_REJECTS_UNKNOWN` gate prune now joins; the utility-stub scaffold 9a introduced and 9d retires) and ADR-046 (9c's existence-gate + P3-vector + exit-1 refusal shape, mirrored for prune's preamble). Completes the `Filled in: Phase 9a/9b/9c/9d` promise in `cli.py`'s tier-header — the CLI surface is now feature-complete.

## ADR-046 — `letterbox init` is two distinct steps (existence-precheck refuse + temp-then-`os.replace` publish), regenerates the trailing `[[channels]]` block by string-split for `--channel`, and writes to fixed conventional paths (NOT the `--config`/`LETTERBOX_CONFIG` lever)

**Date:** 2026-05-29
**Context:** Phase 9c replaces `cli.py`'s transitional `init` stub with the real handler — the **first write command** in the CLI surface (9a/9b were dispatch + read-only). Three design choices needed recording. (1) Vision §13.4 / Kernel L6 explicitly names "config files written by `letterbox init`" as bound to write-temp-then-rename, but the obvious one-liner `open(path, "x")` (atomic exclusive create) *looks* like it satisfies both refuse-overwrite and crash-safety in a single call — and doesn't: a crash mid-`"x"`-write leaves a partial file that `init` then politely refuses to re-scaffold over forever, the exact dead-end §13.4 exists to prevent. (2) `--channel NAME` must produce a config *about* `NAME`, not the sample's `debate-01` placeholder plus a leftover — but there is no stdlib TOML *writer* (`tomllib` is read-only; `tomli_w` is not a dependency), so a parse→mutate→re-serialize round-trip is impossible. (3) The 9b read handlers resolve their target via the `--config`/`LETTERBOX_CONFIG` lever (`_resolve_state_dir`); it was tempting to reuse that for init's write target.
**Decision:** (a) **Refuse and durability are two separate steps, never collapsed.** The handler does an existence pre-check (`target.exists()` → Framework-P3 vector citing the path + a "remove it to re-scaffold" hint, exit 1) — that is the refuse-overwrite covenant; then `_atomic_write_new` writes the content to a sibling `<name>.<pid>.tmp`, `flush`+`os.fsync`, and `os.replace`s it onto the target — that is the crash-safe publish (the target never appears half-written; a crash leaves only the temp, which is cleaned up on any error path including `KeyboardInterrupt`). `open("x")` is explicitly rejected. The residual check→replace race (a concurrent `init` could create the target between the check and the replace, which then clobbers it) is documented and accepted: both writes are identical content, and concurrent `init` of the same path is not a real human scenario — locking would be over-engineering (Framework P12). (b) **`--channel NAME` regenerates the trailing block by string-split.** `_build_config_content` reads the 1c bundled sample via `importlib.resources` (UTF-8, so the 📬 `notification_template` emoji round-trips), asserts `sample.count("[[channels]]") == 1` (sample-drift guard), and for `--channel` takes the sample text up to its sole `[[channels]]` token and appends a freshly-templated block. This is robust because TOML array-of-tables must conventionally come **last** (a scalar table after them would re-bind to the last array element) and the sample places it last. `--channel` is validated by `protocol.is_valid_channel_name` *before* templating, so the name is `[a-z0-9_-]` (alphanumeric first char) → a `name = "{NAME}"` string template carries zero TOML-injection risk. Without `--channel`, the sample is written **verbatim** (byte-equal). (c) **init writes to fixed conventional paths** — `Path.cwd() / "letterbox.toml"` (default, ADR-010 project-local) or `Path.home() / ".letterbox" / "config.toml"` (`--global`, HOME-derived, mirroring `config._user_global_config_path`), creating `~/.letterbox/` at mode `0700` (mkdir + explicit `os.chmod` to defeat umask) **only when it was absent** (never re-tightening a dir the user already configured). The `--config`/`LETTERBOX_CONFIG` lever is NOT consulted: that lever selects *which file to read*, not *where to scaffold*. `--global` uses `dest="is_global"` (`global` is a Python keyword), and init joins `_REJECTS_UNKNOWN` so `init --bogus` is a vector error (exit 2).
**Rationale:** The two-step split is the whole point of §13.4 for a *write* command: refuse protects the user's existing work, atomic-publish protects against a partial write becoming a permanent dead-end — `open("x")` satisfies the first while *creating* the second failure mode, so it is worse than the idiom it appears to replace. Reusing `protocol.write_message`'s established temp+`os.replace` shape (vs. an `os.link` no-clobber publish that would collapse the gate and the race) keeps `cli.py` legible against the codebase's existing durability idiom; the link-based race-free variant was available latitude but trades readability for a guarantee against a non-scenario. String-split regeneration is the only available mechanism given no stdlib TOML writer, and the array-of-tables-last convention plus the drift-guard assert make it safe rather than fragile. Writing to fixed paths (not the config lever) is correct because init's job is to create the conventional file `load_config` will later *discover* — routing it through the read-lever would let `--config ./elsewhere.toml init` scaffold somewhere `load_config` won't look by default, defeating the on-ramp. Parent-dir fsync after `os.replace` (2c's durability extension) is intentionally skipped for this one-shot human command — the file fsync is the core §13.4 requirement; a one-shot scaffold does not warrant the directory-fsync cost.
**Supersedes:** None — first record of the `init` write contract. Consumes ADR-010 (project-local-by-default scope), the 2c atomic-write idiom (`protocol.write_message`), and the 3a/3b/4d mkdir(0o700)+chmod umask-defense pattern. Builds on ADR-045 (9a's `_REJECTS_UNKNOWN` gate, which init now joins) and ADR-026 (the per-launch-identity NOTE comment carried into the regenerated `[[channels]]` block — identity is never per-channel-config).

## ADR-045 — Phase 9a CLI dispatch: pre-argparse `mcp` intercept + `parse_known_args` leftover-routing (NOT `argparse.REMAINDER`), with `--config` as a global top-level flag

**Date:** 2026-05-29
**Context:** Phase 9a fills `letterbox/cli.py`'s stub with the top-level argparse dispatch — harness subcommands (`claude`/`gemini`/`antigravity`) → `launcher.run_launcher` under `asyncio.run`; `mcp` → raw-argv forward to `mcp_server.run`; `list-channels`/`tail`/`init`/`prune` → transitional fail-loud stubs (bodies land in 9b–9d). The plan (§3 illustration, K2, §6 Gotcha #3) sketched the `mcp` route and the utility-stub catch-alls using `argparse.REMAINDER` positionals, with `parse_args` as the dispatch spine. **Two argparse-3.13 realities broke that sketch.** (1) `argparse.REMAINDER` as a subparser's first positional does NOT capture a *leading optional-looking* token: `letterbox mcp --channel a …` and `letterbox tail --channel x` both bubble `--channel` up as `error: unrecognized arguments: --channel` (exit 2) instead of feeding it to REMAINDER. The scout brief flagged this exact fragility ("REMAINDER is order-sensitive on 3.13; the pre-argparse intercept is the more robust mechanism") and Gotcha #9 explicitly requires `letterbox tail --channel x` to NOT crash with "unrecognized arguments". (2) Success criterion #5 (`letterbox --config ./my.toml claude --channel c`) places `--config` *before* the subcommand, which argparse only supports for a flag on the *top-level* parser — not a per-subparser (parent-parser) flag, which would have to come after the subcommand (`letterbox claude --config … ` — the §10-latitude shape that contradicts criterion #5).
**Decision:** (a) **`mcp` is intercepted before argparse** — `if raw and raw[0] == "mcp": return _dispatch_mcp(raw[1:])` — forwarding everything after the `mcp` token byte-for-byte to `mcp_server.run`. This is the canonical agent invocation shape (`generate_mcp_config` always emits `["mcp", "--channel", …]` with `mcp` leading, §7.1), so the intercept handles 100% of real usage and never lets argparse touch the W13 join-key spellings (K2). (b) **The dispatch spine uses `parse_known_args`, not `parse_args`.** The `mcp` subparser (registered for `--help`/invalid-choice coherence) and the four utility stubs declare NO flags of their own; `parse_known_args` returns leftover argv in `unknown`, which `_handle_mcp` forwards raw and the stubs ignore. (c) **Harness flag-surface strictness is preserved manually**: after `parse_known_args`, `if args.command in _HARNESSES and unknown: parser.error(…)` — a stray flag before any `--` is a vector error (P3), since the harness flags are a fixed contract, not transitional. (d) **`--config` is a global top-level flag** (sets `os.environ["LETTERBOX_CONFIG"]` to `os.path.abspath(os.path.expanduser(value))` — the K5 env lever, not `cli_overrides`); per-harness `--channel`/`--as`(`dest="as_label"`)/`--cwd` are added to each harness subparser. (e) **Stub exit code is `1`** (a recognized-but-unimplemented command, distinct from argparse's `2` usage errors). The `--` → harness `extra_args` split (K1) stays a manual pre-argparse `_split_passthrough` on the first `--`, as planned.
**Rationale:** `parse_known_args` is the only argparse mechanism that reliably absorbs leading-dash leftovers on 3.13 (REMAINDER is broken for this, `nargs="*"` rejects optionals); it is *simpler* than a fragile REMAINDER positional and honors the plan's actual contract ("raw forward, no reparse" for `mcp`; "don't crash on transitional flags" for stubs) rather than its illustrative mechanism (the plan §10 explicitly grants latitude here: "the contract is raw forward, not the mechanism"). The pre-argparse `mcp` intercept is defense-in-depth + the byte-exact-forward guarantee for the one invocation that actually matters; the registered (flag-less) `mcp` subparser keeps `letterbox --help` and the invalid-choice options list coherent and forwards the non-canonical `letterbox --config x mcp …` ordering through the same `_dispatch_mcp` helper. Global `--config` is dictated by criterion #5's placement; making it top-level (vs. the §10-latitude per-harness shape) is the only way `letterbox --config … claude …` parses. Manual harness-leftover rejection restores the strict-validation that `parse_known_args` relaxes, scoped to the subcommands whose flag surface is a real contract.
**Supersedes:** None. Refines the plan's §3/K2/§6-Gotcha-#3 REMAINDER sketch (mechanism only — the K2 "raw forward, never reparse" contract is honored unchanged) and resolves the §10-latitude-vs-criterion-#5 tension on `--config` placement. Consumes ADR-035 (`mcp_server.run(argv)` owns the join-key argparse — 9a forwards to it untouched), 8c K5 (`run_launcher` flat-`0` exit code mapped to `sys.exit` here), and ADR-026 (identity is per-launch; 9a passes `--as` through as `as_label=` and resolves no identity itself). Recorded for vision-review visibility per "Agency, Anchored" (a planned mechanism was replaced for a grounded reason).

## ADR-044 — Phase 8d realizes T1's two-process exchange as ONE receiver `run_launcher` (B) + a real stdio-client `send_message` (A), and drives A's server body directly (the console `mcp` route is not wired until 9a)

**Date:** 2026-05-28
**Context:** Phase 8d is the launcher-half integration test that finally drives the whole chain end-to-end across two real endpoints over the real stdio transport — closing **W14** ("the in-memory SDK client is NOT the real stdio transport") and the launcher half of **T1**. The PHASE_INDEX 8d sketch and Vision §9.2 (line 71) describe T1 as "two letterbox processes, each running fake-harness." Two problems surfaced when realizing that literal shape. **(1) The two-`run_launcher` shape is mechanically unrealizable as a *triggerable send*.** `fake_harness` only echoes stdin and `subprocess.Popen`s+reaps its `--mcp-config` MCP child — it never *calls* a tool, and the test cannot reach that child's stdin (fake_harness owns the pipe). So in a two-`run_launcher` design, neither endpoint's agent ever calls `send_message`; the "send" would have to be faked by a direct `write_message`, which does NOT exercise the MCP tool over stdio and would leave W14 open (it proves nothing the 8c `TestRunLauncherComposition` test didn't). **(2) The real `letterbox mcp` console-script route is not wired until Phase 9a.** The plan's §3/§9 assumed A could drive "the real `letterbox` console script's `mcp` subcommand," but the `mcp` dispatch lives in `cli.py`, whose body is a Phase-9a stub (`cli.main()` prints "CLI not yet implemented (Phase 9a)" and returns 1). 9a comes *after* 8d, so `letterbox mcp …` never reaches `mcp_server.run()` and `stdio_client.initialize()` fails with `McpError: Connection closed`.
**Decision:** Realize T1's launcher half as **one receiver `run_launcher` (B = claude-b) + a real MCP stdio-client `send_message` (A = claude-a) driven by the test acting as A's agent.** The two coordinating letterbox processes are then (1) B's PTY-Parent runtime and (2) the A-side `letterbox mcp` server — coordinating ONLY via the filesystem (the shared `channels/test-e2e` dir), exactly the §2.1 / CLAUDE.md architecture. A's send spawns a genuine stdio MCP server subprocess over the real `mcp.client.stdio` transport and invokes the real `send_message` body (7b), which `write_message`s into the shared channel; B's live watcher detects the peer write, renders, and injects the notification into B's PTY (proven via B's echo file). Because the console `mcp` route is unwired, A spawns the **Phase-7 server body directly** — `command=sys.executable, args=["-c", "import sys; from letterbox.mcp_server import run; run(sys.argv[1:])", "--channel", …]` — the exact `run(sys.argv[1:])` invocation `mcp_server.run`'s own docstring names as supported. A's `StdioServerParameters.env` carries `LETTERBOX_HOME=str(state_dir)` explicitly (the mcp SDK forwards only `['HOME','LOGNAME','PATH','SHELL','TERM','USER']`, NOT `LETTERBOX_HOME`), plus an isolated `HOME` and clean `cwd` (W18/K5 directory-agreement join key). The process-tree test (Vision §9.2 line 65) is **separate** and keeps the *real* `generate_mcp_config` + console-script child — it asserts spawn-tree *position* (PPID == fake_harness, not the PTY-Parent), which holds regardless of whether the CLI stub short-circuits.
**Rationale:** The one-receiver + real-stdio-send shape is *more* faithful to the contract being proven ("A sends via the MCP `send_message` tool over real stdio"), not less — it exercises the genuine transport and server body, which is the entire point of W14; the rejected two-`run_launcher`+`write_message` alternative is cheaper but proves nothing new and leaves W14 open. Driving `mcp_server.run` directly (vs. the console `mcp` route) keeps the transport + server 100% real — only the process *launcher* differs from the not-yet-wired console route, and that route is 9a's concern; this honors the plan's §1 INTENT ("8d proves the thing 9a will route to actually works end-to-end"): 8d proves the server works over real stdio, 9a wires the CLI route to it. Splitting the topology assertion into its own test (real child) from the roundtrip (sleeper child, PATH-independent) keeps each test's flake surface minimal and maps cleanly onto the two distinct Vision §9.2 contracts. **Forward note for 9a:** once `cli.main` dispatches `mcp`→`mcp_server.run`, a follow-up may switch Test 1's A-side `command` to the `letterbox` console script to also exercise the CLI route end-to-end (or 10b's full E2E suite covers it).
**Supersedes:** None — first record of the T1 launcher-half realization. ## ADR-043 — `run_launcher` composes the PTY-Parent runtime via a three-way race, converging every exit path on one idempotent teardown ladder in a `finally`

**Date:** 2026-05-28
**Context:** Phase 8c adds the public blocking `run_launcher` to `letterbox/launcher.py` — the composer that turns `letterbox <harness>` into a runnable, interruptible command (closes Wiring Ledger entry-002 / W15). 8a built `setup_launcher` (the started session); 8b built `_injection_loop` (the consumer that drains the watcher queue and injects). Neither could *stop*, and nothing tied them into one thing you can run and Ctrl-C. The §2.1 contract is explicit — *"on clean exit: terminate the harness gracefully, stop the watcher, remove the temp MCP config file."* The 8b→8c handoff sketched a **two-way** race (injection loop ∨ signal). That is insufficient: between notifications the injection loop is parked on `await queue.get()`, so a harness that exits *quietly* (the common case — the user just quits the agent with no peer message pending) would never wake the loop, and `run_launcher` would hang past the harness's death until a signal arrived.
**Decision:** `run_launcher(harness, channel, *, …, teardown_timeout=5.0) -> int` is a public coroutine (9a owns the `asyncio.run(...)` boundary and `sys.exit`s the returned int). It `await`s `setup_launcher`, installs SIGINT/SIGTERM handlers that set an `asyncio.Event`, then blocks on a **THREE-way** `asyncio.wait({injection_task, proc_task, signal_task}, FIRST_COMPLETED)` — adding an explicit harness-process-exit waiter (`_await_process_exit`, a pure-asyncio `Popen.poll()` + `asyncio.sleep(0.1)` loop) as the third racer. Whatever wakes the race, a `finally` runs the idempotent teardown ladder (`_teardown_runtime`): (1) `remove_signal_handler` first; (2) cancel-and-settle all racers via `asyncio.gather(*racers, return_exceptions=True)` (never re-raises, so a child `CancelledError` or an unexpected loop bug comes back as a result element — logged, never allowed to skip cleanup); (3) `watcher.stop()` → (4) `adapter.teardown(handle, timeout=teardown_timeout)` → (5) `cleanup_mcp_config(path)`, the resource steps nested in `finally`s so each runs even if an earlier one raises. Because teardown lives in the `finally`, **cancelling the `run_launcher` task runs the identical teardown** — so 8d and any multi-launcher embedding tear down via `task.cancel()`, NOT real signals (`add_signal_handler` is one handler per signal per loop). Returns `0` on every clean teardown. Signal installation is wrapped to tolerate `NotImplementedError`/`RuntimeError`/`ValueError` (non-POSIX / off-main-thread): warn and continue, since the process waiter + cancellation still provide teardown.
**Rationale:** The process-exit waiter is what lets us *observe* the clean exit §2.1 promises — the difference between a launcher that returns to the shell and one that hangs; it is correctness, not scope creep (the injection-loop racer still catches a harness dying mid-inject, surfacing 8b's §12 diagnostic; the signal racer catches Ctrl-C; all three converge on one `finally`). Poll over `asyncio.to_thread(process.wait)`: the `sleep` is a clean cancellation point, so teardown cancels the waiter instantly — a `to_thread` wait would keep its worker thread blocked until the process actually died, lingering past the cancel and tripping `filterwarnings=["error"]`. `gather(return_exceptions=True)` over a bare `with suppress(CancelledError): await task`: the injection loop only catches `OSError` and returns (8b K3), so any *other* exception (a real bug) would propagate out of a bare-suppress await and **abort the resource-critical teardown**, leaking the harness child + temp file (the L6-forbidden failure); gathering guarantees steps 3-5 still run. The harness teardown reaps the whole process group including the MCP child via `killpg` (T10/6d), so 8c writes no separate MCP-child reaping. No `queue.task_done()`/`join()`: the notification queue is disposable inference (8b K5) — undrained events are dropped because the messages remain on disk (L1). Return code is a flat `0` for all clean teardowns (the recommended v1 minimum; `128 + signum` was available latitude but adds a branch for no v1 consumer benefit — 9a maps the int to `sys.exit`). **Explicit non-goal:** no interactive stdin/stdout I/O bridge — §2.1's PTY-Parent duties are spawn / watch / inject / teardown only; `run_launcher` blocks on lifecycle events, not terminal bytes.
**Supersedes:** None — first record of the `run_launcher` composition + three-way race + teardown-ordering + signal/cancellation model. Improves on the 8b→8c handoff's two-way race sketch (recorded here for vision-review visibility per "Agency, Anchored"). Builds on ADR-042 (the injection loop's cancel-clean, return-on-OSError shape 8c cancels and settles), the 4c cancel-and-settle discipline, and ADR-040/041 (8a's launcher policies). Realizes the `Filled in: Phase 8a/8b/8c` promise in `launcher.py`'s tier-header.

## ADR-042 — The injection loop surfaces a dead-PTY `OSError` to stderr and *stops* (return), rather than swallowing, continuing, or re-raising

**Date:** 2026-05-28
**Context:** Phase 8b adds `_injection_loop(session)` to `letterbox/launcher.py` — the coroutine that drains the watcher's `asyncio.Queue[WatcherEvent]`, renders each event, and injects it via `adapter.inject`. `Adapter.inject` propagates `OSError` (`EIO`/`EBADF`, base.py:210-212) uncaught when the harness's slave fd has closed — i.e. the harness process is gone. Vision §12 (line 853) names *silent injection failure* as a top risk: "the message is lost from the agent's perspective even though it's still on disk; surface this as a visible error, not swallow it." But §12 stops there — it does not say what the loop should then *do* (keep going? stop? re-raise?), and that choice has real consequences. A closed PTY is **permanent**: 5a's `inject_to_pty` loops on short writes, so backpressure *blocks* rather than raises; an `OSError` out of `inject` is never a transient hiccup. The launcher tests run under `filterwarnings = ["error"]` (1b), which escalates an un-retrieved task exception's `RuntimeWarning` into a hard failure.
**Decision:** On `OSError` from the single `inject` call, the loop (a) emits one **`_LOGGER.error(...)`** line naming the harness and the cause (Python's `logging.lastResort` writes WARNING+ to stderr even with zero handler configuration — matching the watcher/protocol `_LOGGER` idiom and not coupling the launcher to a print site), then (b) **returns** — terminating the loop cleanly. It does **not** swallow-and-continue (a closed PTY would spew identical `EIO` on every subsequent event) and does **not** re-raise (a bare-`create_task` coroutine that raises leaves an un-retrieved task exception that `filterwarnings=["error"]` escalates unless 8c retrieves it; returning sidesteps that while still satisfying "do not swallow," because the ERROR line is already out). The `except` clause is **`except OSError` only** — narrow, so `CancelledError` (8c's cancel point) and any genuine bug (e.g. a `TypeError` from a malformed render) propagate loudly.
**Rationale:** Returning lets 8c observe loop completion and proceed to teardown — the natural "the harness died, wind down" signal — without a second exception-plumbing path. Re-raise was rejected not because it is wrong in principle but because it shifts a retrieval obligation onto 8c that returning makes unnecessary; an implementer who prefers re-raise must confirm 8c retrieves the exception (Plan §13 latitude). `print(..., file=sys.stderr)` was considered and rejected: the `_LOGGER` path is the house idiom and is zero-config-correct. The narrow `except` is load-bearing — broadening it would swallow `CancelledError` and break 8c's cancel-and-suppress under `filterwarnings=["error"]`. **Testing note (lastResort semantics):** asserting the surfacing uses `caplog` for the ERROR *record*, not `capsys` for stderr bytes — pytest installs a root logging handler, which suppresses `logging.lastResort`, so nothing reaches stderr *under test* even though production (no handlers) writes to stderr. The "reaches stderr in production" property follows from lastResort semantics, not from a capsys assertion.
**Supersedes:** None — first record of the injection-loop error-handling policy. Extends Vision §12 ("surface, don't swallow") with the loop's termination + mechanism choice that §12 left implicit. Complements ADR-041/ADR-040 (8a's other launcher policies) and the 4c cancel-and-suppress pattern that 8c will use against the cancel-clean coroutine shape 8b authors.

## ADR-041 — The launcher resolves the spawned `command`/`default_args`/`notification_template` from `config.harnesses`, not the adapter class attrs

**Date:** 2026-05-28
**Context:** Phase 8a fills `letterbox/launcher.py`, the first consumer of `config.harnesses` (built in 1c, no consumer until now). The adapter registry (5b) returns an `Adapter` instance whose `command`/`default_args`/`notification_template` are `ClassVar` defaults equal to `DEFAULT_HARNESSES` (e.g. `ClaudeAdapter.command == "claude"`). But Vision §6.4 (line 563) is explicit — *"spawn the configured harness CLI with the configured args; both come from `letterbox.toml`"* — and §5.3/§8.2 frame the adapter attrs as TOML *defaults* the user overrides. If the launcher spawned from the adapter class attrs, a user `[harness.claude] command = "/opt/claude"` block in `letterbox.toml` would be silently ignored — a documented v1 feature failing silently, the exact class of failure the framework forbids. The PHASE_INDEX left the config-vs-adapter source-of-truth implicit.
**Decision:** `setup_launcher` reads `harness_config = config.harnesses[harness_name]` and assigns the resolved values as **instance attributes** onto the fresh adapter (`adapter.command = harness_config.command`; `adapter.default_args = list(harness_config.default_args)`), which shadow the `ClassVar` defaults for that one instance. `Adapter.spawn` then composes `[self.command, *self.default_args, *extra_args]` from them. The resolved `notification_template` is validated at startup (ADR-039 chain) and stashed on `LauncherSession.notification_template` for 8b to render. Config is the source of truth; the adapter class attrs are the fallback that lives in `DEFAULT_HARNESSES`.
**Rationale:** `get_adapter` returns a fresh, stateless instance per call (5b K5), so per-instance attribute mutation is safe and localized — no need to reopen the sealed 5b spawn API or thread config through it. For the default case the instance attrs equal `DEFAULT_HARNESSES`, so behaviour is unchanged; the override case is the whole point. The alternative (spawn from class attrs) was rejected because it silently drops user config. Assumption (v1): harness names are exactly the three built-ins, so `get_adapter(name)` and `config.harnesses[name]` always agree on the key — `load_config` seeds `harnesses` from `DEFAULT_HARNESSES` and only adds/replaces, never removes. A registered-but-unconfigured harness (a custom adapter with no `[harness.X]` block) is a clean `KeyError` with a vector message, not a crash.
**Supersedes:** None — first record of the config-overrides-adapter-defaults policy. Complements ADR-040 (the launcher's other join-key wiring) and the 5b adapter contract (ADR-018/spawn API), and discharges the §6.4/§5.3/§8.2 "users override the defaults" promise that had no consumer until 8a.

## ADR-040 — The launcher propagates the resolved `state_dir` to the agent + its MCP child via `LETTERBOX_HOME` (W18 join key)

**Date:** 2026-05-28
**Context:** Phase 8a spawns the harness, which in turn spawns its own MCP-server child (`letterbox mcp …`, ADR-009/ADR-025). The three processes coordinate ONLY via the filesystem — specifically, all three must operate on the *same* channel directory under the same `state_dir`. The launcher resolves `state_dir` via `config.load_config(cli_overrides).state_dir` (full §8.1 precedence) and opens the channel under it. But the W13 MCP argv (`mcp --channel --as --instance-id`, 5c) carries **no `--state-dir` token** — so the MCP child re-resolves `state_dir` itself via `load_config()`. If the child's cwd holds a different project-local `letterbox.toml`, or its env differs, it could resolve a *different* directory: both sides would write to different dirs and the conversation would go silently dark with no error (the silent-failure class L1/P3 forbid). 7a's IMPLEMENTATION_NOTES flagged this as a forward concern for 8a.
**Decision:** `setup_launcher` builds the adapter spawn env as `{**os.environ, "LETTERBOX_HOME": str(state_dir)}`, where `state_dir` is the exact value it passed to `Channel.get_or_create(..., state_dir=state_dir)`. `LETTERBOX_HOME` is config Level 5 — the highest-precedence override (config.py:594) — so the MCP child's `load_config()` pins to the launcher's value regardless of any project-local `letterbox.toml` or other env in the child's cwd. The spawn env is fully specified (5a K6 — the adapter does not merge `os.environ`), so the launcher forwards the parent environment explicitly and adds the join key. ADR-038's `list_channels` derives the same dir from the open handle, so all three views agree.
**Rationale:** Env propagation, not a new `--state-dir` flag, is the mechanism — it keeps the W13 argv unchanged and pins the child without widening the join-key surface. `.resolve()` is idempotent on an already-absolute path, so launcher and child agree byte-for-byte. The cost is one env entry; the benefit is structural prevention of the silent channel-split. The row stays OPEN past 8a (9a's CLI is the second consumer of the resolved `state_dir`).
**Supersedes:** None — first record of the `LETTERBOX_HOME`-as-cross-process-join-key policy. Complements ADR-035/ADR-038 (the MCP child resolves `state_dir` at launch / derives operating paths from the open handle) by pinning *what value* that resolution lands on.

## ADR-039 — Adapter registration via a call-time `load_builtin_adapters()` bootstrap, not package-import side effects

**Date:** 2026-05-28
**Context:** The adapter registry (`letterbox.adapters.base._REGISTRY`) is populated by the `@register_adapter` decorator, which runs as an import side effect of each concrete module (`claude`/`gemini`/`antigravity`, 6a–6c). `get_adapter("claude")` therefore only resolves *after* `letterbox.adapters.claude` has been imported. `letterbox/adapters/__init__.py` was deliberately left empty (6a–6d) so that importing the `letterbox.adapters` package — which sibling modules like `mcp_config` trigger — has **no side effect** and drags in no concrete adapters. The 6a–6d and 7a IMPLEMENTATION_NOTES all carried the same forward concern: the green in-process test suite hides the unwired state because 6d imports all three concretes directly; *something* at launch must trigger registration, or the launcher's `get_adapter` raises `KeyError` against an empty registry.
**Decision:** Add `load_builtin_adapters() -> None` to `letterbox/adapters/__init__.py`. It imports the three concrete adapter modules **inside the function body** (call-time), never at module level, and the launcher calls it once during `setup_launcher` before `get_adapter`. The package docstring is updated to state the preserved property explicitly. The bootstrap is verified by a **fresh-interpreter subprocess smoke test** (Phase 8a §10 criterion 10) — the only proof that the bootstrap, not some unrelated test import, does the registration.
**Rationale:** A function-body import (a) preserves the load-bearing "no side effect on `import letterbox.adapters`" property — only *calling* the function registers; (b) keeps the concrete adapter module names out of `launcher.py`, honouring its Tier-4 header ("concrete adapters via the registry only, never direct imports of sibling adapter modules"); (c) co-locates the built-in roster with the adapter package. Module-level imports in `__init__.py` were rejected (they reverse the no-side-effect property, pulling every concrete into any `import letterbox.adapters.mcp_config`); direct concrete imports in `launcher.py` were rejected (they violate the launcher tier-header). Gotcha for tests: the bootstrap fires via the import cache, so after a `reset_registry` empties `_REGISTRY`, calling `load_builtin_adapters()` again does NOT re-register (modules already cached) — registry-resetting tests must register their own adapter.
**Supersedes:** None — first record of the registration-bootstrap policy. Discharges the 6a–6d/7a forward concern (empty `adapters/__init__.py`, registration triggered at the launcher).

## ADR-038 — `list_channels` derives its operating `state_dir` from the open channel handle, not by re-resolving config

**Date:** 2026-05-28
**Context:** Phase 7d fills the `list_channels` MCP tool body in `letterbox/mcp_server.py`. The tool must enumerate every channel under the operating state directory, delegating to the 3d free function `letterbox.channel.list_channels(*, state_dir: Path)`. But `channel.py` cannot import `letterbox.config` (Tier 2 leaf discipline), so the *caller* must supply the absolute `state_dir`. The `_build_server(channel, instance_id)` closure captures the open `Channel` handle and the launch `instance_id` — but **not** `state_dir`. Two sources are available to recover it: (a) re-resolve via `load_config().state_dir` — the same full-precedence resolver `_open_channel` used at launch (the W18 precedent); or (b) derive it from the already-open handle: `channel.path.parent.parent`. `Channel.get_or_create` locks the invariant `channel.path == state_dir / "channels" / <name>` (channel.py:180), so the handle's grandparent path *is* the exact state directory the channel lives in.
**Decision:** `list_channels` derives `state_dir = channel.path.parent.parent` from the open channel handle and passes it to the 3d free function. It does **not** re-resolve config. The derivation is inlined at the call site with a comment naming the `get_or_create` invariant it relies on.
**Rationale:** The open handle is the single authoritative source for the world this channel belongs to. Re-resolving config (option a) re-reads `letterbox.toml` and the `LETTERBOX_HOME`/`LETTERBOX_STATE_DIR` environment on every tool call; if config or env changed between launch and the call (an edited TOML, a mutated env in a long-lived agent process), `list_channels` would enumerate a *different* directory than the one the open channel lives in — a subtle cross-world inconsistency where the agent's "list of channels" excludes the very channel it is talking on. The config resolver is authoritative *at launch* (that is what `_open_channel` correctly uses to decide where to open); once the channel is open, the handle is authoritative for "where am I." Deriving from the handle also avoids a second filesystem/TOML read on a read-only hot path (the 10c `list-channels` budget is P95 < 200 ms) and eliminates a TOCTOU window between resolve and enumerate. The cost is a dependency on the `get_or_create` path invariant, which is already locked by tests and documented in the channel layer — a cheap, durable coupling.
**Supersedes:** None — first record of the derive-operating-paths-from-the-open-handle policy. Complements ADR-035 / W18 (`_open_channel` resolves `state_dir` via `load_config().state_dir` *at launch*) by drawing the line: the config resolver decides where to open; the open handle decides where everything else operates.

## ADR-037 — `acknowledge` validates `message_id` wire-format at the MCP boundary; `since_id` is trusted (validate-the-write, trust-the-read)

**Date:** 2026-05-28
**Context:** Phase 7c fills the `check_messages` and `acknowledge` MCP tool bodies in `letterbox/mcp_server.py`. `acknowledge` advances a **persisted** per-agent read marker via `Channel.acknowledge`, whose implementation is `high_water_mark = max(current, message_id)` using a *lexical* string comparison (correct because microsecond-precise filenames sort chronologically — ADR-027). The `Channel.acknowledge` docstring (`channel.py:220`) deliberately defers wire-format validation: *"the message_id is trusted verbatim — no regex check … the MCP boundary at Phase 7c owns wire-format validation."* This leaves a live silent-corruption path: a hallucinated `message_id` that sorts lexically **above** every real id (e.g. `"zzz"`, since every real id begins `"msg-"` and `"msg-…" < "zzz"`) would jump the high-water-mark past the entire channel and **silently blank the agent's whole inbox** — `check_messages`/`check_latest_message` would thereafter return nothing, with no error anyone thinks to look for (the exact silent-failure class L1 and P3 forbid). `check_messages`' other agent-supplied input, `since_id`, is structurally different: `list_unread` uses it only as a one-call `path.stem > since_id` comparison bound; it is never persisted and never advances the marker (ADR-012).
**Decision:** `acknowledge` guards `is_valid_message_filename(f"{message_id}.json")` (the ADR-028 strict regex) **before** delegating to `Channel.acknowledge`, and raises a *vector* `ValueError` naming the rejected value and the expected stem shape on failure — letting FastMCP wrap it as a tool error (the same raise-and-let-the-SDK-wrap surface as `send_message`'s `MessageTooLarge`). `message_id` arrives FastMCP-coerced as `str`, so only its *format* is checked (no type-guard needed at this boundary, unlike the 3a `Channel.get_or_create` sites that accept arbitrary input). `since_id` is **not** validated — it is passed through verbatim. This codifies a boundary policy: **validate the input that writes persisted state; trust the input that only reads.** `limit` is likewise unguarded at the tool (its type-guard + `[1,100]` clamp already live in `list_unread`, K5).
**Rationale:** The validated-write / trusted-read split mirrors the existing `send_message` asymmetry — `in_reply_to` (7b) is passed through unvalidated because it is inert payload, whereas `message_id` mutates a persisted cursor, so the blast radius differs in kind, not degree. Validating `since_id` was rejected: a garbage `since_id` can at worst return wrong results for a single non-mutating call, and the regex would reject legitimate stems the agent paginates with. Returning `{"ok": False, "error": …}` instead of raising was rejected as softer than the established raise-and-wrap pattern and easier for an agent to ignore. The guard lives at the MCP boundary (Tier 4) rather than in `Channel.acknowledge` (Tier 2) by the channel layer's explicit deferral: keeping the regex at the agent-facing edge means a non-MCP caller (a future CLI `ack`, a test) is free to choose its own surface, while the one untrusted-input path that can corrupt state is hard-gated. `acknowledge` is the *only* MCP-boundary input that mutates persisted state, so this is the correctly-scoped single site for the policy (the 7d tools take no agent input).
**Supersedes:** None — first record of the MCP-boundary wire-format-validation policy. Complements ADR-022 (own-write filter; the join-key/identity half of boundary discipline) and ADR-028 (the strict filename regex this guard reuses), and discharges the validation responsibility `Channel.acknowledge`'s docstring deferred to Phase 7c.

## ADR-036 — `check_latest_message` uses a new `Channel.latest_unread` reverse-scan, not `list_unread`-tail

**Date:** 2026-05-28
**Context:** Phase 7b fills the `check_latest_message` MCP tool, whose Vision §6.1 contract is "the single **most recent** unread peer message (or null), minimal context cost, does NOT advance the read marker." The 7a→7b handoff previewed "reuse `Channel.list_unread`, don't modify Tier 1/2." But `Channel.list_unread` (3c) is a *forward page* — it returns unread messages **oldest-first, capped at `limit` (max 100)**. Taking `list_unread(limit=100).messages[-1]` returns the wrong message whenever unread > 100, and that is not a rare edge: because `check_latest_message` deliberately never advances the `high_water_mark` (§6.1), a peek-only agent (never calls `acknowledge`/`check_messages`) accumulates unbounded unread, so `messages[-1]` would return an ever-staler ~100th-oldest message forever; a peer burst of >100 messages between acknowledges hits the same bug. The correct algorithm is a reverse scan from the newest filename stopping at the first unread peer message (O(trailing own-writes) reads, usually one) — but the own-write filter `_is_own_write` it needs is module-private to `channel.py` (the friend-import contract, ADR-022).
**Decision:** Add `Channel.latest_unread(*, self_instance_id: str) -> Message | None` to Tier-1 `channel.py` (a deliberate, documented edit to a Phase-3-"sealed" module) and route `check_latest_message` through it. The method reverse-scans `reversed(list_messages(self.path))`, breaks at the first stem `<= high_water_mark` (chronological filename sort ⇒ all older are also read), skips prune-race `FileNotFoundError` and `ParseError` files **silently** (K5 — no per-file WARN in an unbounded scan, diverging from `list_unread`), and returns the first surviving non-own-write `Message` via the existing `_is_own_write`. The MCP tool body becomes a 2-line serialize wrapper over `_message_to_dict`. Two alternatives rejected: (a) duplicating the OR-filter inline in the Tier-4 tool body — forks the ADR-022 own-write semantic into a second site that would silently drift if the pending K7 OR/AND reconciliation resolves to AND; (b) promoting `_is_own_write` to public and scanning in the tool body — spreads filesystem+filter loop logic into Tier 4 (orchestration) and breaks the `test_is_own_write_is_module_private` lock.
**Rationale:** Putting the scan on `Channel` keeps all scan+filter logic in the tier that owns it and **reuses** `_is_own_write` directly, so `latest_unread` automatically tracks whatever K7 resolves to — single-source-of-truth (Framework "the principle wins over the convenient shortcut"). The cost is ~25 LOC + unit tests in a module Phase 3 sealed; the benefit is correctness for unbounded unread and zero filter duplication. This is the architecturally honest fix for a gap the handoff's "reuse `list_unread`" preview did not anticipate. `latest_unread` is the sole "newest unread" tail accessor — `check_messages` (7c) correctly wants `list_unread`'s forward page, and `channel_info` (3d) inlines its own count, so there are no hidden sibling consumers tempted to mis-reuse it.
**Supersedes:** None — first record of the latest-unread accessor. Refines the 7a→7b handoff's "7b does not modify Tier 1/2" preview (sanctioned override) and complements ADR-022 (combined own-write filter) by adding a second in-module consumer of `_is_own_write`.

## ADR-035 — MCP server: `run(argv)` owns the join-key argparse; high-level `FastMCP`

**Date:** 2026-05-28
**Context:** Phase 7a fills `letterbox/mcp_server.py`, the stdio MCP server the agent spawns as its own child (`letterbox mcp --channel … --as … --instance-id …`, Vision §6.3, ADR-009). Two architectural choices needed pinning. **(a)** The four-token launch invocation is a cross-process join key: 5c's `generate_mcp_config` (locked by 6d Family B) emits `["mcp", "--channel", ch, "--as", sl, "--instance-id", iid]`, and *something* downstream must parse those three flags back out. The risk is the silent-failure mode named in Vision §7.1 — if the parser's flag spellings drift from the producer's (a `--sender` for `--as`, an `--instance_id` underscore, a renamed subcommand), the agent's MCP child parse-errors the instant it spawns and the channel goes dark with no error anyone thinks to look for. Where the spellings live determines how many places can drift. **(b)** The `mcp` SDK exposes both a high-level `FastMCP` (decorator-driven tool registration, auto-generated input JSON-schema from signatures) and a low-level `mcp.server.lowlevel.Server` (hand-written `list_tools`/`call_tool` dispatch). ADR-029 noted both exist; 7a must choose.
**Decision:** **(a) `run(argv)` owns the join-key argparse, and 9a forwards raw argv to it.** `mcp_server._parse_args` defines the three flag spellings `--channel` / `--as` (mapped to `dest="sender_label"` because `as` is a Python keyword) / `--instance-id` in **one** place. 9a's `cli.py` recognizes the `mcp` subcommand token and calls `mcp_server.run(sys.argv[2:])` — everything after `mcp` — letting 7a parse the rest. 9a does **not** define a `mcp` argparse subparser with its own copy of the three flags. **(b) High-level `FastMCP("letterbox")` with `@server.tool()` registration.** Each tool's input schema is generated from the function signature + type hints, so locking the six signatures at 7a locks the agent-facing schema and 7b–7d only fill bodies. The six tools are registered at 7a as fail-loud `NotImplementedError` stubs (a stub raises, naming its filling phase, rather than returning a silent placeholder).
**Rationale:** **(a)** Join-key discipline (§13.3) says a cross-process contract must have exactly one authoritative spelling site, mirrored by a consumer-side lock test. The alternative — a `mcp` subparser in `cli.py` — would duplicate the three spellings across `cli.py` and `mcp_server.py`, creating exactly the two-places-to-drift hazard the contract exists to prevent; 9a forwarding raw argv keeps cli.py ignorant of the join key (it routes on the `mcp` token alone, per K1). 6d's Family B is the producer-side lock; 7a's `TestParseArgs` (canonical spellings parse; `--sender`/`--instance_id`/`--instanceid`/`--channel-name` all rejected) is the consumer-side lock. **(b)** The low-level `Server` would require hand-writing `list_tools`/`call_tool` dispatch and the six input schemas — more surface for zero v1 benefit (ADR-009's stdio transport is satisfied either way; no v1 tool needs the streaming/progress affordances only the low-level API exposes). High-level `FastMCP` is the boring-at-the-core choice (P19); defer the low-level API to a future phase only if a tool ever needs what the decorator can't express.
**Supersedes:** None — first record of the MCP-server bootstrap contract. Refines ADR-009 (stdio MCP transport) and ADR-025 (PTY-Parent ↔ MCP-child coordinate only via the filesystem) by pinning *how* the server parses its launch invocation and *which* `mcp` API it builds on. Supersedes the early CODEBASE_MAP `async run(channel, sender_label, instance_id) -> int` interface sketch (which predates the join-key-ownership analysis) — the `run(argv)` shape is what lets 9a forward raw argv.

## ADR-034 — Adapter lifecycle-hook ordering contract (locked end-to-end)

**Date:** 2026-05-28
**Context:** Phase 5b wired four lifecycle hooks (`pre_spawn`/`post_spawn`/`pre_inject`/`pre_teardown`) into the concrete `Adapter.spawn`/`inject`/`teardown` methods (`letterbox/adapters/base.py`) and tested that each hook *fires*. What 5b did not lock is the **relative ordering** of those hooks across a whole session, nor the exact argument each receives. The concrete adapters 6a-c are about to start *relying* on this ordering — a future `post_spawn` startup-context injection assumes the handle already exists; a `pre_inject` per-harness escape assumes it runs before the UTF-8 encode-and-terminate step that ADR-018 mandates. An ordering that is "true by happenstance of how `spawn` is currently written" is a promise written in sand: the day a maintainer reorders `spawn` (calls `post_spawn` before `spawn_pty`, or moves `pre_inject` after the encode) every adapter that trusted the order breaks silently, and the agent goes quiet with no error anyone thinks to look for.
**Decision:** The four-hook lifecycle ordering is a **documented, tested contract**, not an implementation detail. Across one `spawn`→`inject`→`teardown` session the hooks fire in exactly this sequence, each `pre_*` hook *before* the primitive it wraps and `post_spawn` *after*:

```
spawn:    pre_spawn(argv)     → spawn_pty(...)      → post_spawn(handle)
inject:   pre_inject(message) → encode + terminator → inject_to_pty(...)
teardown: pre_teardown(handle)                      → close_pty_handle(...)
```

`pre_spawn` receives the **complete** assembled argv `[command, *default_args, *extra_args]` (command at index 0, not just `extra_args`); `post_spawn` and `pre_teardown` receive the **same** `PTYHandle` object `spawn` returns; `pre_inject` receives the **exact** `str` passed to `inject` (before the encode). All four defaults are identity/no-op, so a zero-override adapter completes a full lifecycle untouched. The contract lives in two complementary homes: the discoverable *what* in the `Adapter` class docstring (where a 6a-c adapter author looks), and the durable *why* here. Phase 5d adds the locking tests (`TestHookOrdering` in `tests/test_adapters_base.py`) but **no functional change** — 5b's composition was verified already correct.
**Rationale:** The ordering is load-bearing for the entire config-driven adapter model (L4): adding a harness must be a config change riding on a stable base, and the base's hook contract is the rail it rides on. Documenting the contract converts an implicit "it happens to work this way" into an explicit guarantee, so the next maintainer who reorders `spawn` gets a red test and a clear ADR instead of a 2 a.m. mystery — small code, long shadow. Two homes rather than one because the class docstring is where an adapter author reads the contract (and `DECISIONS.md` is gitignored until 12c, so less discoverable to them), while the ADR is the permanent decision trail (§13.9). Test-only because making 5d a behavioural edit would mean 5b shipped a bug — it did not.
**Supersedes:** None — first explicit record of the lifecycle-hook ordering contract. Refines (does not contradict) Vision §5.4's "default no-op" hook list by pinning the order and the per-hook argument shape. Note 5b's plan §11 forward-guessed "5d does NOT modify base.py"; this ADR documents the deliberate, sanctioned override (the docstring-only edit, blessed by the PHASE_INDEX 5d "documents the hook ordering" deliverable and base.py's own "Filled in: Phase 5b/5d" tier-header).

## ADR-033 — MCP config generation: unified `mcpServers` shape, resolved command path, caller-driven cleanup

**Date:** 2026-05-28
**Context:** Phase 5c lands `letterbox/adapters/mcp_config.py`, the generator for the temp MCP config file the PTY-Parent writes at launch and points the agent's `--mcp-config` flag at (Vision §6.3). The agent reads this file to spawn its own stdio MCP child (`letterbox mcp --channel … --as … --instance-id …`). It is the only handshake between the human-launched `letterbox` process and the agent-launched MCP server, and the two halves coordinate only via the filesystem — so the four identity tokens and the file's mode/shape are a cross-process contract, not a local detail. The PHASE_INDEX/Vision phrase "per-harness conventions" needed pinning down: does it mean the *file content* differs per harness, or the *invocation flag*?
**Decision:** (1) **One unified `{"mcpServers": {"letterbox": {command, args}}}` shape for all three v1 harnesses.** Per-harness divergence is the `--mcp-config` *flag* carried by each concrete adapter (6a-c), NOT the file content — the MCP child argv is identical regardless of which harness spawns it. A `_build_payload(harness, …)` internal keeps `harness` as the single documented seam where a *future* harness needing a different file shape would branch; v1 ignores it. (2) **`command = shutil.which("letterbox")`, falling back to the bare string `"letterbox"` (with a WARN) when `which` returns `None`** — embeds the absolute path the PTY-Parent resolved so the agent-spawned MCP child does not silently depend on the agent's own `PATH` (5a K6: the spawn env is fully specified, not inherited). (3) **Cleanup is caller-driven via a companion `cleanup_mcp_config(path)` free function**, wired by the launcher at 8c, NOT a self-registered `atexit` — Vision §6.3 step 6 assigns removal to the PTY-Parent's own teardown sequence, which a self-`atexit` would make non-deterministic and invisible. (4) **System-temp `mkstemp` + explicit `os.chmod(0o600)`** (umask defense, 3a precedent), JSON written `ensure_ascii=False` + `sort_keys=True` (ADR-030 deterministic-JSON precedent); the four launch values flow as discrete JSON-array argv tokens, never concatenated into a shell string (consistent with ADR-032's no-shell invariant). **No atomic-rename** — §13.4 governs durable protocol/`letterbox init` writes, not transient runtime config; `mkstemp`'s `O_EXCL` create + read-after-agent-spawn means no partial-visibility window regardless.
**Rationale:** Claude Code's `--mcp-config` consumes the `mcpServers` envelope; Gemini CLI and agy are Gemini-lineage and accept the same envelope — so fabricating three identical per-harness content branches now would be speculative (PLANNING_NOTES "avoid speculative branching") while still honoring the contract structurally through the `_build_payload` seam. Embedding the resolved absolute `command` decouples the MCP-child spawn from whatever `PATH` 8a happens to construct for the agent; the bare-name fallback (with a WARN surfacing the latent `PATH`-coupling risk operationally) keeps behavior no worse than the literal Vision §6.3 text when `which` can't resolve — unusual, since the launcher is itself a `letterbox` invocation. The free-function cleanup pair mirrors the 3a/5a/5b "primitive built here, composed at the launcher" template and fits a signal-driven `finally` better than a context-manager `__exit__`. The explicit `os.chmod` restores the exact `0600` regardless of a pathological umask that could strip owner bits from `mkstemp`'s default. **Open assumption (flag for 13c / T6 smoke):** all three real CLIs accept the unified `mcpServers` envelope via `--mcp-config` — verified for Claude Code's documented format, assumed-compatible for the Gemini-lineage CLIs; if a real CLI rejects it, that harness becomes the first user of the `_build_payload` divergence seam.
**Supersedes:** None — first record of the MCP-config-generation contract. Clarifies (does not contradict) Vision §6.3 / PHASE_INDEX 5c "per-harness conventions" as flag-level, not content-level.

## ADR-032 — No-shell invariant: argv-only spawn + CI lint

**Date:** 2026-05-28
**Context:** Vision §6.4 forbids letterbox from `exec`-ing, `eval`-ing, or `subprocess`-shelling any message body or metadata field. Phase 5a (`pty_common.spawn_pty`) is the first and only `subprocess.Popen` call site in the `letterbox/` package; every future spawn site (the launcher in Phase 8a, anything the MCP server might shell out for) inherits the same invariant. Without an explicit structural enforcement, a future contributor "adding shell convenience" could open the no-execution gate without anyone noticing on review — and a peer message body could, in principle, end up reaching `subprocess(body, shell=True)` via a permissive call signature elsewhere.
**Decision:** Two-layer structural enforcement, recorded as a committed invariant:
1. **Function signature.** `spawn_pty(cmd: list[str], ...)` accepts ONLY an argv list. A type-guard at function entry rejects `str` with vector `TypeError`, rejects empty list with `ValueError`, and rejects non-`str` elements with vector `TypeError`. Same shape as 4a's `render_notification` keyword-only signature making prompt-injection via peer payload structurally impossible — 5a's signature makes shell-execution via any input path structurally impossible. A peer's message body cannot become `cmd` without an explicit, deliberate call site that splits it into argv tokens.
2. **CI lint.** `scripts/lint_no_shell_true.sh` (cloned from `lint_ensure_ascii.sh` per PATTERNS.md P1, with the "match-and-fail" shape variant) greps for `shell\s*=\s*True` across `letterbox/**.py` and exits non-zero on any hit. Wired into `.github/workflows/test.yml` as the "Lint no shell=True discipline" step. No `# noqa` exemption mechanism (PATTERNS.md P1 anti-pattern); if a future phase ever legitimately needs `shell=True` (it shouldn't given §6.4), that phase amends the lint with an explicit ADR superseding this one.
**Rationale:** Defense in depth. Either layer alone closes most paths to a violation, but the function signature catches what a developer might add locally without re-running the lint, and the lint catches what a developer might add at a site distant from `spawn_pty` (e.g., a future Tier-4 module shelling out for ergonomic reasons). Both layers are structurally cheap — the signature is one type-guard call, the lint is ~40 lines of bash. Together they convert §6.4 from prose into an invariant the Python language and CI gate enforce. Documents the no-shell rule explicitly so it cannot be silently relaxed; any change requires a new ADR superseding this one.
**Supersedes:** None — first explicit committed record of the no-shell invariant. Operationalizes Vision §6.4's prose.

## ADR-031 — PTY primitive: `os.openpty` + `subprocess.Popen` + `start_new_session=True`

**Date:** 2026-05-28
**Context:** Phase 5a lands `letterbox/adapters/pty_common.py`, the leaf module that opens a pseudo-terminal pair and spawns the harness CLI attached to it. PHASE_INDEX entry 5a sketches the primitive as `pty.fork` / `os.forkpty`, but Vision §5.2's typed `PTYHandle.process: subprocess.Popen` field is the load-bearing source of truth — neither `pty.fork()` nor `os.forkpty()` returns a `Popen`. The implementation must reconcile these, and the choice of spawn primitive determines the teardown shape, signal semantics, and lifecycle observability for every adapter that follows.
**Decision:** `spawn_pty` uses `os.openpty()` to allocate the master/slave fd pair, then `subprocess.Popen(cmd, stdin=slave_fd, stdout=slave_fd, stderr=slave_fd, cwd=cwd, env=env, start_new_session=True, close_fds=True)` to spawn the child attached to the slave. All three child standard streams share one slave fd (PTY semantics). `start_new_session=True` is the default and the production-path value — the child becomes its own session/process group leader so `close_pty_handle` can `os.killpg(handle.pid, ...)` and reach the entire process tree (agent + tools + grandchildren) without signaling letterbox itself. Tests may pass `start_new_session=False` for inspection; production callers do not.
**Rationale:** Vision §5.2 mandates `PTYHandle.process: subprocess.Popen`. `pty.fork()` returns `(pid, master_fd)` only — no `Popen`, no `.wait()` / `.terminate()` / `.poll()` / `.returncode` accessors. `pty.spawn()` is interactive-and-blocks-the-caller, unusable for the async-injection model. The `openpty` + `Popen` pair is the stdlib-only textbook combination (the same primitive `ptyprocess`/`pexpect` use under the hood); using it directly avoids the P19 dependency-skepticism cost of pulling in another library to express what stdlib already offers. `start_new_session=True` is the load-bearing kwarg for safe teardown — without it, `killpg(handle.pid, SIGTERM)` would resolve through letterbox's own pgid and signal letterbox itself, an outright self-kill. The documented caveat is that `setsid` revokes the child's controlling TTY (since Python's `subprocess` does not perform a subsequent `TIOCSCTTY` ioctl); v1's three target harnesses (Claude Code, Gemini CLI, agy) do not require one — they read/write the slave fd directly and never query TTY-layer signal state. A future adapter that does need a controlling TTY would acquire it via a `preexec_fn` doing `os.setsid()` + `os.open(os.ttyname(slave_fd), os.O_RDWR)` (the ptyprocess shape), but that's a future phase's job per Framework L8 frozen-artifact discipline.
**Supersedes:** The "using stdlib `pty.fork`/`os.forkpty`" wording in PHASE_INDEX 5a (which predates the `subprocess.Popen` integration mandate in Vision §5.2). The vision wins over the index per the 4a K2 precedent ("vision/framework principle wins over PHASE_INDEX wording").

## ADR-030 — JSON serialization uses sorted keys for stable round-trip equality

**Date:** 2026-05-27
**Context:** Phase 2a lands `protocol.to_json_bytes` and `from_json_bytes`. Two equal `Message` values (same fields by structural equality) must produce byte-identical JSON output — otherwise refactors that reorder dataclass fields, or differences in dict insertion order across runs (including JSON parses from differently-ordered wire bytes), silently change the on-disk file content. Round-trip byte equality is load-bearing for any future change-detection / hash-based comparison, for diffability of two messages with `cmp`/`diff`, and for the test category in plan §9.1 (`to_json_bytes(from_json_bytes(to_json_bytes(m))) == to_json_bytes(m)`).
**Decision:** `to_json_bytes` passes `sort_keys=True` to `json.dumps`. This sorts keys alphabetically at every nesting level — top-level message keys, the two `metadata` sub-keys, and any nested objects inside `metadata.ext`. Combined with `ensure_ascii=False` (Vision §13.2), `separators=(",", ":")` (compact, no spaces), and no trailing newline, the output is fully deterministic for equal Messages.
**Rationale:** Without recursive sort, two equal Messages whose `metadata.ext` dicts differ only by key insertion order serialize to different bytes — breaking the deterministic-bytes property that K5 names. Dict equality is order-independent in Python; the wire format must be equally order-independent. Sorting recursively costs nothing at `protocol.py`'s scale (no message exceeds 5 MB) and prevents an entire class of "two equal Messages produced different bytes" bugs. The wire-key order documented in Vision §3.2 happens to already be alphabetical, so `sort_keys=True` produces exactly the documented order with no special-casing.
**Supersedes:** Implicit "dataclass field order" assumption from Vision §3.2 (which the §3.2 example showed in alphabetical order without naming a mechanism). Locks the mechanism explicitly. Also makes the `metadata.ext` "preserved verbatim across round-trips" promise stronger — the *bytes* may differ (sub-key order may be sorted vs. original), but the parsed Python value is structurally equal in both directions, which is the contract callers care about.

## ADR-028 — Strict filename validation; malformed names ignored

**Date:** 2026-05-27
**Context:** Notification template renders `{message_id}` (and possibly other filename-derived fields) into the text injected into the PTY. If the watcher accepted any file matching `msg-*.json` without strict format validation, a malformed filename like `msg-123-$(ls).json` (or filenames with path traversal sequences) could land in the template substitution path. Even though the injected text goes to an agent (not a shell), the broader principle is: never trust what the filesystem hands you.
**Decision:** All readers and the watcher validate channel-directory filenames against the strict regex `^msg-\d{8}T\d{6}\d{6}-[0-9a-f]{32}\.json$` (updated for UUID4). Files failing the regex are completely ignored — never parsed, never rendered into templates, never returned by `check_messages`. Novel malformed filenames are logged at `WARN` level.
**Rationale:** Defense in depth. Even though the practical injection risk into a PTY-bound LLM agent is low (the LLM doesn't execute strings), strict input validation at the filesystem boundary is cheap insurance against future misuse and against bugs in downstream consumers (e.g., a `letterbox tail --format=html` that might one day render to a browser).
**Supersedes:** Implicit "match anything that looks like a message file" reader behavior.

## ADR-027 — Filenames use full UUID4 (not 4-hex-char) for collision safety

**Date:** 2026-05-27
**Context:** Earlier filename format used `secrets.token_hex(2)` = 4 hex chars = 16-bit randomness = 65,536 combinations. With microsecond timestamp precision, collisions require both same-µs writes and same hex — rare but possible, particularly under burst load or across endpoints with skewed clocks. `os.rename` silently overwrites on POSIX, so a collision = a silently lost message, violating L6 (atomic writes, no silent loss).
**Decision:** Filename random component is `uuid.uuid4().hex` (32 hex chars = 128 bits). Birthday paradox 50% collision at ~2^64 messages — effectively impossible across any realistic history.
**Rationale:** "Collisions are impossible by design" is famous-last-words reasoning when the math says otherwise. UUID4 costs ~28 extra bytes per filename — irrelevant for any practical channel size. Removes the dependence on serialization invariants for safety. Once-and-done.
**Supersedes:** 4-hex-char `secrets.token_hex(2)` in earlier drafts.

## ADR-026 — Identity is per-launch (CLI/env), not per-channel-config

**Date:** 2026-05-27
**Context:** Earlier drafts had `default_sender` / `default_recipient` declared per-channel in `letterbox.toml`. In a shared-project-directory scenario (the most common case — both endpoints in the same repo, reading the same toml), both letterbox processes would pick up the same `default_sender`, collapsing identity. Config files describing channels should describe the channel, not assert which side of it any particular launch represents. The cross-harness case (Claude+Gemini) just works by default because the harness names differ. The same-harness case requires `--as` — empirically how it's always worked in the Workshop bridge.
**Supersedes:** Per-channel `default_sender`/`default_recipient` config fields. Amends ADR-005's channel config description.

## ADR-025 — MCP server is a separate subcommand spawned by the agent (not co-resident with PTY-Parent)

**Date:** 2026-05-27
**Context:** Initial spec described letterbox as a single process that spawns the harness CLI as a PTY child AND runs the MCP server "co-resident" using stdio transport. This is topologically impossible: stdio-transport MCP servers must be **spawned by** the consuming agent (the agent owns the stdin/stdout the server uses), but in the initial design the letterbox process was the PARENT of the agent — opposite direction.
**Decision:** Split letterbox into two coordinated processes per terminal. **PTY-Parent process** (`letterbox claude/gemini/antigravity`) spawns the harness CLI as a PTY child and runs the filesystem watcher. **MCP server process** (`letterbox mcp` subcommand) is spawned BY the agent as its stdio MCP subprocess child, after the PTY-Parent writes a temp MCP config and passes `--mcp-config` (or equivalent) to the harness. The two processes coordinate ONLY via the filesystem (channel directory + read-state files) — no IPC. The MCP child inherits channel, sender_label, and instance_id from CLI args the PTY-Parent injects into the temp config.
**Rationale:** Honors the actual MCP architecture. Lets us keep stdio transport (ADR-009) without violating process-tree direction. The filesystem-only coordination is consistent with L1 (files as source of truth) and means the two processes are independently testable. From the user's perspective, one terminal still hosts one logical letterbox endpoint — the two-process split is hidden inside the launch.
**Supersedes:** "MCP server is co-resident with the PTY/watcher" framing in earlier drafts. Amends L5 in the framework to clarify "one logical letterbox per terminal, two coordinated processes."

## ADR-024 — Watcher does NOT auto-surface backlog on startup; restart = fresh start

**Date:** 2026-05-27
**Context:** Earlier design had the watcher scan the channel directory at startup and emit notifications for every unread peer message — interpreted via the endpoint's `high_water_mark`. With a long-running channel and a recent crash, this could mean 30+ notifications firing immediately on startup, bombing the agent's context with stale messages and forcing it to respond to historical state when the user may have wanted a clean restart.
**Decision:** Watcher records a "start watermark" at init (the newest filename present, or current UTC timestamp if empty) and fires notifications **only for messages that arrive strictly after that point**. Backlog is not surfaced automatically. The agent has two MCP tools — `check_latest_message()` (returns single newest unread, minimal context cost) and `check_messages(limit, since_id)` (paginated catch-up) — that surface backlog explicitly when the user prompts the agent to check.
**Rationale:** "Wake the Agent" (L3) is about *new arrivals*, not historical replay. A restart often means "fresh start," not "catch me up on everything I missed." User agency is load-bearing: the user is in the loop, and they decide whether catch-up is wanted. Auto-flooding violates that by imposing a response to history. `check_latest_message` was the affordance pattern in the original Workshop bridge and is reintroduced for the same reason — it gives the user a low-cost way to opt into catch-up without paying the full pagination cost.
**Supersedes:** Earlier implicit assumption that the watcher would rescan-and-fire on startup. Also reframes a portion of ADR-022's cross-restart concern: with no startup rescan, the cross-restart self-notification flood is structurally impossible regardless of filter logic. The combined `(sender OR instance_id)` filter still governs `check_messages` result classification (so the agent doesn't see its own past-session messages when explicitly catching up), but the watcher's auto-notify path no longer needs it.

## ADR-023 — Harness input-queueing is an assumption, not a check

**Date:** 2026-05-27
**Context:** Gemini's review raised concern that PTY injection during a tool-call subprocess could land in the subprocess's stdin (potentially corrupting bash scripts, vim sessions, etc.). The theoretical fix is checking `os.tcgetpgrp(handle.master_fd)` before each injection and deferring when a subprocess is in the foreground.
**Decision:** Document the harness's input-queueing behavior as an **assumption** in letterbox's interoperability contract, not a runtime check letterbox enforces. The three v1 adapters target harnesses (Claude Code, Gemini CLI, Antigravity CLI) that all queue PTY input correctly during tool calls — input received while the agent is busy is buffered and processed once the agent's main loop is free. Adapters for harnesses that *don't* queue properly are responsible for their own coordination.
**Rationale:** Empirically verified that all three v1 harnesses queue properly (Vinga's daily Workshop use over months). Adding tcgetpgrp scaffolding to letterbox would solve a problem the v1 adapters don't have. If a future adapter targets a non-queueing harness, that's an adapter-level concern (it can implement queue checks in its `inject` override). The frozen-artifact discipline (L8) favors documenting clear assumptions over building defensive scaffolding for hypothetical cases.
**Supersedes:** Earlier (briefly entertained) plan to bake foreground-process-group checks into the adapter base class.

## ADR-022 — Combined own-write filter: `sender` OR `instance_id`

**Date:** 2026-05-27
**Context:** Initial design filtered own-writes by `sender` label (would deadlock same-harness conversations where both endpoints share a sender). ADR-011 switched to `instance_id` filtering (works for same-harness in-session, but on letterbox restart the new process has a new instance_id and would see its own past-session messages as peer messages, flooding notifications). Neither filter alone covers all four cells of the matrix (same/different session × same/different label).
**Decision:** Filter own-writes by `(sender == self.sender_label) OR (instance_id == self.instance_id)`. Sender match catches cross-restart self-recognition (durable identity). Instance_id match catches the same-harness configuration-error case (identical sender labels). OR'ing them covers all four cells.
**Rationale:** Identity in dialogue systems has two axes — durable (label) and ephemeral (process). Both serve filtering needs. Combining them is cheap (a boolean OR on each incoming message) and prevents both the restart-flood failure mode and the same-harness-deadlock failure mode. The identity-collision warning (§3.4.1) prompts users to fix configuration errors that trigger the instance_id-only path.
**Supersedes:** ADR-011's "instance_id is THE Join Key" framing — instance_id is now half of a combined filter, not the sole mechanism.

## ADR-021 — Per-agent read state replaces shared archive

**Date:** 2026-05-27
**Context:** Initial spec had `archive(message_id)` as a physical `os.rename` from the live channel directory to a shared `archive/` subdirectory. Because the channel directory is shared between both peers, archiving was a **global mutation** — when one peer archived a message, that message disappeared from the other peer's live view as well. For dialogue systems this is wrong: "read state" is fundamentally per-agent. If Gemini consumes and archives a message from Claude, that should not delete the message from Claude's `letterbox tail` history or from a third-peer's view (in future n-way topologies).
**Decision:** Replace shared archive with **per-agent read-state files** at `channel/.read/{sender_label}.json`. Each endpoint stores its own `high_water_mark` — the most recent message filename it has acknowledged. Messages remain in the live channel directory for everyone; only the per-agent marker advances. The MCP tool is renamed from `archive(message_id)` to `acknowledge(message_id)`, reflecting that the operation is a local read-state update, not a global filesystem mutation.
**Rationale:** Three benefits. (1) Preserves the agent-facing inbox-clearing affordance (check_messages won't return acknowledged messages for that endpoint). (2) Eliminates the shared-mutation footgun (peer's view is unaffected). (3) Naturally extends to n-way topologies (each endpoint independently tracks its read position). Physical removal of old messages moves to `letterbox prune`, which is user-controlled and never automatic. The high_water_mark optimization (one filename per endpoint, leveraging filename-as-chronological-sort) means read-state files are tiny regardless of channel length.
**Supersedes:** Shared `archive/` directory; the original `archive(message_id)` MCP tool. Obsoletes ADR-019.

## ADR-020 — `in_reply_to` is trusted blindly; no referential integrity check

**Date:** 2026-05-27
**Context:** The `in_reply_to` field references another message ID. A naive implementation would validate that the target exists before accepting the write. This is expensive (channel scan), racy (target may be archiving in flight), and wrong in cross-process scenarios (peer may have just archived the referenced message).
**Decision:** `send_message(body, in_reply_to=...)` accepts any string value for `in_reply_to` without validation. If the target ID doesn't exist or no longer exists, that's a readability issue downstream, not a protocol error. Implementing Claudes are explicitly forbidden from adding a validation scanner.
**Rationale:** Async file-based systems can't have strong referential integrity without locking and centralization, both of which violate L1/L7. Referential integrity at write time would also break in legitimate cases (peer archives before reply lands). Treating `in_reply_to` as informational metadata that consumers MAY use, rather than a foreign key the protocol enforces, is the only consistent model.
**Supersedes:** No prior decision; codifies the implicit stance against validation.

## ADR-019 — ~~Concurrent `archive` is idempotent under FileNotFoundError~~ (OBSOLETED by ADR-021)

**Date:** 2026-05-27
**Status:** OBSOLETED 2026-05-27 by ADR-021.
**Context (preserved for history):** Initial spec said archive is "idempotent: re-archiving a message is a no-op." But `os.rename(live, archive)` raises `FileNotFoundError` when the source has already been moved by a concurrent process.
**Why obsoleted:** The shared-archive model itself was the wrong design — it mutated state across both peers. ADR-021 replaces shared archive with per-agent read state, which eliminates the concurrent-rename race entirely (each endpoint writes only its own read-marker file).

## ADR-018 — PTY injection always appends `\r` to force agent consumption

**Date:** 2026-05-27
**Context:** Initial spec described notification injection as "write to PTY master fd." This is necessary but not sufficient: a terminal agent reads stdin line-buffered, and bytes without a trailing carriage return just sit in the prompt buffer waiting for a human Enter key. The agent never wakes — violating L3 (Wake the Agent), the load-bearing kernel principle.
**Decision:** Every `Adapter.inject()` post-processes the rendered notification by appending `\r`. Enforced in the base class — adapters cannot skip it. If a future harness needs a different terminator, that's a per-adapter `line_terminator` class attribute override, not optional behavior.
**Rationale:** Forgetting the CR is the kind of bug that ships and then mysteriously fails in production. Baking it into the base class as non-optional makes it impossible to forget. Carriage-return is the canonical PTY input terminator; line discipline converts it to `\n` for the consuming process as needed.
**Supersedes:** "Write to PTY master fd" (incomplete — missing the terminator).

## ADR-017 — Microsecond precision in filenames; chronological sort by filename, not `mtime`

**Date:** 2026-05-27
**Context:** Initial filename format was `msg-YYYYMMDD-HHMMSS-XXXX.json`. Two messages in the same second would tie on the timestamp portion, and the random hex `XXXX` would order them incorrectly. The proposed fix was to sort by `mtime` instead — but `mtime` is fragile: `cp -r` (without `-p`), syncthing, rsync, NFS clock skew, and `tar` extraction can all reset or shuffle mtime, silently mangling conversation history.
**Decision:** Filename becomes `msg-YYYYMMDDTHHMMSSffffff-XXXX.json` with 6-digit microseconds. Lexical filename sort is chronologically correct independent of any filesystem metadata. All sorting in letterbox uses filename order, never `mtime`.
**Rationale:** Conversation history must survive standard filesystem operations (backups, syncs, archives). Microsecond precision is sufficient because same-endpoint writes are serialized and cross-endpoint collisions at 1µs resolution are impossible across any meaningful comm link. The cost is 7 extra chars per filename — irrelevant.
**Supersedes:** Second-precision filename + `mtime`-based sort.

## ADR-016 — `.tmp` files: ignored on read, reaped only on startup with mtime > 1h

**Date:** 2026-05-27
**Context:** Initial spec said orphaned `msg-*.json.tmp` files would be "deleted on next channel access." This is a textbook race condition: a `.tmp` file may be a live write in progress, and aggressive reaping by one process would silently corrupt writes by another.
**Decision:** Two-part rule. **(1)** Readers never look at `.tmp` files — the message-file glob (`msg-*.json`) structurally excludes them. **(2)** A reaper sweep runs only on `letterbox` startup (not on every access) and deletes only `.tmp` files with `mtime` older than 1 hour. Fresh `.tmp` files are preserved.
**Rationale:** Atomic-rename writes mean a `.tmp` file is fleeting under normal operation (microseconds between write+rename). A `.tmp` older than 1 hour is almost certainly orphaned. The 1-hour window protects against accidentally killing a slow legitimate write while still cleaning up real garbage.
**Supersedes:** "Delete on next access" reaper policy.

## ADR-015 — UTC timestamps everywhere; local time forbidden

**Date:** 2026-05-27
**Context:** Initial spec said filenames use `YYYYMMDD-HHMMSS` without specifying timezone. Local time would silently mangle chronological sorting across timezone changes, daylight-saving transitions, and machines in different zones (NFS-shared channels).
**Decision:** All timestamps — filenames AND the `timestamp` JSON field — are UTC. Generated via `datetime.now(timezone.utc)`, never `datetime.now()`. Filenames carry no offset suffix (to keep them portable across filesystems that disallow `:` or `+`); the JSON `timestamp` field includes explicit `+00:00`.
**Rationale:** Chronological order across letterbox files is load-bearing — `check_messages` returns newest-first, archives sort by filename, recovery semantics depend on `mtime` and filename order. Local time in any of those produces silent data corruption when timezones shift. UTC is the only safe answer for filesystem-as-database systems.
**Supersedes:** Implicit local-time assumption in initial filename spec.

## ADR-014 — 5MB ceiling on message body size

**Date:** 2026-05-27
**Context:** Initial spec had no upper bound on message body size. A looping agent (or a malicious peer) could write arbitrarily large payloads — gigabyte JSON files would OOM the letterbox process when `check_messages` parsed them.
**Decision:** Hard ceiling of 5 MB on message body. Enforced on write (rejected with `MessageTooLarge` before disk I/O) and on read (oversized files surface as `parse_error: "oversized"` with `body: null`).
**Rationale:** Capping the worst-case parse cost is essential for DoS resistance. 5 MB is generous for human-readable text (≈1M words) but small enough to bound memory predictably. Larger payloads belong in shared filesystem locations referenced *by* message body, not in the body itself.
**Supersedes:** Unbounded body assumption in earlier drafts.

## ADR-013 — Watcher belt-and-suspenders (event observer + polling fallback)

**Date:** 2026-05-27
**Context:** Initial spec relied solely on `watchdog`'s event-driven observer (inotify on Linux, FSEvents on macOS) for new-message detection. Filesystem events are documented to be lossy under load, on certain filesystems (network mounts, FUSE), and during observer hiccups. A single dropped event silently breaks the conversation — peer agent sleeps forever — violating L3 (Wake the Agent) and Framework P5 (Self-Healing).
**Decision:** Watcher uses both paths in parallel. Primary: watchdog observer for low-latency detection (sub-second). Fallback: re-scan polling loop every 5 seconds. Both feed the same notification queue, deduplicated by filename + instance_id. If a watchdog event drops, the polling fallback catches the message within at most 5 seconds.
**Rationale:** Comms infrastructure cannot have silent-loss failure modes. The cost of belt-and-suspenders is a 5s background poll on a directory — negligible CPU. The benefit is that letterbox is self-healing against filesystem event flakiness, which is a documented real-world failure mode on every OS letterbox supports.
**Supersedes:** Event-observer-only design.

## ADR-012 — `check_messages` pagination (default `limit=20`, max `100`, `since_id` cursor)

**Date:** 2026-05-27
**Context:** Initial spec had `check_messages()` return all unarchived peer messages newest-first. With the §9.4 budget allowing up to 1000 unarchived messages per channel, a single tool call could return ~1000 fully-parsed message bodies — instantly destroying the consuming agent's context window and blowing the latency budget.
**Decision:** `check_messages(limit=20, since_id=None)`. Default returns at most 20 newest messages. Hard ceiling at 100 (above that, clamped with warning). `since_id` provides cursor-based pagination ("messages newer than this ID"). Response includes `has_more: bool`.
**Rationale:** Context windows are precious; an agent that swallows 1000 messages in one call is broken regardless of what letterbox returns. Pagination is the standard mechanism. 20 is large enough to handle typical "what did I miss since I last checked" cases and small enough to never bomb a context window. 100 ceiling handles the legitimate "I'm catching up after a long absence" case while still capping worst-case parse cost.
**Supersedes:** Initial unbounded `check_messages()` signature.

## ADR-011 — `instance_id` as the watcher's Join Key (not `sender`)

**Date:** 2026-05-27
**Context:** Initial spec implied the watcher would filter own-writes by matching the `sender` field against the watcher's own sender label. This works for cross-harness topologies (`claude` vs `gemini`) but breaks for **same-harness topologies** (two Claude Code tabs talking) where both endpoints share a sender label — the watcher would filter every message as its own and the conversation would deadlock silently.
**Decision:** Every `letterbox` process generates a per-instance `instance_id` at startup (`lb-{ISO8601}-{6-hex}`) and writes it on every outbound message. The watcher filters peer messages by `instance_id != self.instance_id`. The `sender` field becomes purely human-readable (what shows in `letterbox tail` and message logs); `instance_id` becomes the mechanical filter.
**Rationale:** Decouples human identity from mechanical identity. Enables any topology (cross-harness, same-harness, three-way) without configuration gymnastics. The cost is one JSON field per message and a startup UUID generation — negligible. Catches what would otherwise be a Day-1 footgun for anyone trying to run two tabs of the same harness.
**Supersedes:** Implicit "filter by sender label" assumption in earlier drafts. Later refined by ADR-022 into a combined filter.

## ADR-008 — Notification template language: stdlib `str.format`

## ADR-002 — Files are the source of truth (no daemon, no network)
